# Chapter 32 — Profile / Employee Self-Service Portal

> *In one paragraph.* The Profile Portal is the employee-facing window onto the very same `employees` row that HR edits in Chapter 3. Every signed-in user lands on `/profile` and sees a tabbed self-service hub — Profile, My Leave, My Pay, Benefits, Documents, Tickets, Security — that lets them keep their personal record current, see how much annual leave they have left, check their last six payslips, confirm which benefit plans they are enrolled in, pull HR-uploaded documents, glance at their open service tickets, and change their password. It is also where the password-must-change wall lands new hires on their very first login, where the data subject reaches the Act 843 self-service queue from Chapter 26, and where notification-channel preferences from Chapter 16 are surfaced. The portal writes to the same `PATCH /employees/{id}` endpoints behind the scenes only via dedicated `/profile/*` routes that always operate on `$request->user()->employee`, so an employee can never escape their own row — and it 404s cleanly when an admin account without an attached Employee record opens it.

## Where to find it

- **Sidebar location:** The portal is not pinned to the sidebar group list — it is reached from the top-bar user-account dropdown (top-right avatar) under **My Profile**, and from the `/profile` route directly. The bell icon's "Channels" link in the Notifications module (Chapter 16) and the **My DPA Requests** entry in the user-account dropdown (Chapter 26) live alongside it as related self-service shortcuts. New hires never have to find it — the password-must-change middleware redirects them to `/profile#security` on first login.
- **Roles that see it:**
    - **Every authenticated user**, regardless of role, sees their own Profile portal — there is no permission gate on `/profile`. Super_admin, ceo, hr_admin, dept_head, manager, finance_officer, auditor, it_support, employee — they all land on the same page and see the same six tabs. The page renders for users without an attached `Employee` row too (legacy admin-only accounts created before the employee-portal expansion) — they get the Account and Security sections, plus an empty state on every other tab.
    - **No role sees anyone else's profile** through this URL. There is no `/profile/{id}` form. To see another person's record, HR uses the directory in Chapter 3, the data subject uses the Act 843 portal in Chapter 26, or the auditor uses the masked register in Chapter 24.
- **Related modules:** Employees (Chapter 3) — the data source; every write on this page lands on the same Employee row HR edits from the directory, and the salary visibility, soft-delete, and visibility-scope rules all carry through unchanged. Leave (Chapter 4) — the My Leave tab pulls balances and the latest five requests via `LeaveService::balances()` so the employee sees what HR sees, scoped to themselves. Payroll Engine (Chapter 19) — the My Pay tab lists the latest six `payments` rows, with the gross monthly salary card shown only when `employees.view_salary` is held (so admins and finance see their own number; an ordinary employee sees the payslip list without the salary card unless they hold the permission). Benefits (Chapter 23) — the Benefits tab renders the active enrolment list from the same `benefit_enrolments` relation the HR-side profile uses. Notifications (Chapter 16) — the channel-preferences screen at `/notifications/channels` is a sibling self-service surface; the Profile page links across to it from the user-account dropdown rather than embedding it inline. Identity (Chapter 25) — the National ID field on the Profile tab is the same `national_id` column that the Ghana Card verifier reads; the green "verified" tick is roadmapped to land on the hero card once the NIA adapter ships. DPA & Privacy (Chapter 26) — the in-app subject self-service queue at `/privacy/my` is reached from this page (the cross-link is a known nav gap recorded in Ch 26 — "employees reach `/privacy` from the Profile portal"). Standards (Chapter 44) — every Act 843 §38 data-subject right relies on the read surface this page provides.

## The password-must-change wall

A new hire created by HR on the Add Employee panel from Chapter 3 inherits a temporary password set by the HR officer. The user row is also stamped with `password_must_change = true` (the migration `2026_06_01_000001_add_password_must_change_to_users` added the column with a default of `false`; the `IssuePasswordResets` console command and the HR seed flow both flip it on for fresh users). On the next login, the `ForcePasswordChange` middleware intercepts every request and redirects the user to `/profile#security` with a flash error: *"Please set a new password before continuing."*

The middleware allows a tiny set of routes through so the user is not trapped in a redirect loop:

- `profile.edit`, `profile.update`, `profile.personal`, `profile.password` — so the user can land on the page and submit the change.
- `logout`, `password.confirm` — escape hatches for re-auth and exit.

Every other route (Dashboard, Employees, Leave, anything) is blocked with a redirect for HTML requests and a `403 Password change required before continuing.` for JSON / API requests. The flag clears as soon as the user submits a successful `PATCH /profile/password` — the `updatePassword` controller method writes `password_must_change => false` alongside the new hash in the same update, so the user's very next click goes through unimpeded.

> This is the only modal-feeling guardrail in the entire product. It is deliberately not a JavaScript modal — a middleware redirect is harder to bypass with browser tooling and works even on the JSON API surface.

## Channel preferences and DPA requests

Two adjacent self-service surfaces live alongside the Profile portal and are reached from the same user-account dropdown rather than being embedded inline:

- **Notification Channels (`/notifications/channels`, Chapter 16)** — toggle email, in-app, WhatsApp, Slack, and Teams individually; provide a WhatsApp phone and explicit consent (Meta requires the pairing of consent + phone before template messages are allowed); provide a Slack user id for direct messaging. The `NotificationChannelController::update` endpoint stamps `whatsapp_consent_at` only when both consent and phone are present, and clears the timestamp the moment either is revoked — a deliberate posture so the institute always has provable consent on file for every WhatsApp outbound. The channel preferences are stored on `users.notification_channels` (JSON) and read by `MessagingDispatcher` to decide which adapters fire for any given event.
- **My DPA Requests (`/privacy/my`, Chapter 26)** — file an Access / Rectification / Erasure / Portability / Objection / Information request under Act 843 §38, see every request you have ever filed, download the fulfilled ZIP export when ready, and withdraw an open request. The page is reachable from the user-account dropdown, surfaces a six-card "education panel" restating every right in plain English, and shows a register of the subject's own requests with the statutory deadline countdown. The MVP does **not** have a sidebar entry for this surface — that is a recorded nav gap in Chapter 26 — and the Profile portal is the documented in-app entry point.

> Both surfaces are sibling self-service screens to the Profile portal rather than tabs inside it. The split keeps each page focused on a single concern and lets each carry its own RBAC / consent posture without polluting the others.

## The data behind it

Every write on this page goes to one of three tables — `users`, `employees`, or `employee_skills` — through dedicated `/profile/*` routes that always resolve the target row from `$request->user()->employee` rather than from a URL parameter. There is no `/profile/{employee}` route; an employee can never edit somebody else's record from this surface.

What every reader of the screen needs to keep in mind:

- **The Profile portal writes to the same row HR edits.** The personal, emergency, and bank fields on the Profile tab are the very same columns the HR-side Edit Employee panel in Chapter 3 writes to. The two surfaces are layered on the same table, gated by different controllers — `ProfileController` for the employee, `EmployeeController` for HR — but the data lands in the same place. A change saved from one surface is visible from the other on the very next render.
- **Avatar uploads land on the public disk.** Files write to `storage/app/public/avatars/` and are served from `/storage/avatars/{filename}`. The `php artisan storage:link` step must have been run once at deployment; if it hasn't been, the photo upload succeeds but the URL 404s — a known gotcha called out in the deployment doc.
- **Salary visibility is enforced at the API layer, not the UI.** The My Pay tab's Gross Monthly Salary card renders only when the `salary` key is present on the resource — and that key is omitted entirely for viewers who fail `EmployeePolicy::viewSalary`. The same gate is in force on the HR-side directory and detail page, so the posture is consistent across every salary surface in the product.
- **The portal degrades gracefully for legacy admin accounts.** The Profile portal renders for users who don't have a linked Employee row — the seed data's `super_admin` account is one such — but the Personal, Emergency, Bank, and Skills cards all hide (`v-if="hasEmployee"`), and the tabs that need an employee (My Leave, My Pay, Benefits, Documents, Tickets) render with empty states. The Account and Security tabs always work.
- **Password change clears the must-change flag in the same update.** The `updatePassword` controller method writes `password_must_change => false` alongside the new hash, so a user who lands on the security tab through the password-must-change wall is freed the moment they submit a successful change.

## How it talks to other modules

- **Employees (Chapter 3)** — the data source. `ProfileController::edit` calls `EmployeeService::find($user->employee->id)` and passes the resulting `EmployeeResource` to the Vue page. Every write from this surface lands on the same `employees` row that HR edits from the directory in Chapter 3 — the `UpdateEmployeeRequest::authorize()` three-tier rule (HR > department head > self) is the same one this surface implicitly relies on; the `/profile/*` routes are a thin convenience that always resolves to the self-edit branch.
- **Leave (Chapter 4)** — the My Leave tab calls `LeaveService::balances($employee->id, now()->year)` for the ring tiles and reads `$employee->leaveRequests()->latest()->limit(5)` for the recent-requests list. The "All requests →" link routes to `/leave-requests`, the same page HR uses to manage the institute's leave queue.
- **Payroll Engine (Chapter 19)** — the My Pay tab reads `$employee->payments()->latest()->limit(6)`. Every payment is a row produced by `PayrollService::calculate()` and approved through the 2FA-gated approval lane; the portal is the read-only consumer surface.
- **Benefits (Chapter 23)** — the Benefits tab renders `emp.benefit_enrolments` from the EmployeeResource. The enrolment list is the same one HR sees on the detail page in Chapter 3; the portal does not surface dependants or claims (those live in the full Benefits module).
- **Notifications (Chapter 16)** — the user-account dropdown links across to `/notifications/channels` for per-user channel preferences (email / in-app / WhatsApp / Slack / Teams). Channel preferences are a sibling self-service surface to the Profile portal rather than a tab inside it.
- **Identity (Chapter 25)** — the National ID field on the Profile tab writes to the same `employees.national_id` column the Ghana Card verifier reads. The MVP accepts any 64-character string; Phase 1's NIA adapter will validate the format synchronously on save and write an `IdentityVerification` row asynchronously via `VerifyEmployeeIdentity`. The green "verified" tick on the hero card is roadmapped to land alongside the adapter.
- **DPA & Privacy (Chapter 26)** — the in-app data-subject self-service queue at `/privacy/my` is reached from this page (the cross-link is the only in-app entry point — see the recorded nav gap in Chapter 26). The page is the Act 843 §38 data subject's right-to-read surface for their own data; the DPO acts on it from `/admin/privacy`.
- **Audit Logs (Chapter 24)** — every write on this page is event-sourced through the same Employee event listeners HR's writes are. A bank-detail change saved from this portal is indistinguishable in the audit chain from one saved from the HR-side edit panel, except for the `actor_id` (the employee themselves, not an HR officer).
- **Off-boarding (Chapter 10)** — the Danger Zone account-deletion path soft-deletes the user and cascades to the Employee. The official institute exit path is the Off-boarding module, which clears settlements, ends benefit enrolments, runs the final payroll, and only then archives the employee. The Danger Zone is a Breeze-template inheritance, not the recommended path.

## Standards touchpoints

- **Data Protection Act, 2012 (Act 843) §38 — rights of the data subject** — the entire Profile portal is the Act 843 right-to-read surface for the employee's own personal data. Every field on the Profile tab is data the subject is entitled to see and (subject to the institute's contractual position) correct. The "My DPA Requests" page reachable from the user-account dropdown is the formal lifecycle endpoint for the same rights at scale (Access, Rectification, Erasure, Portability, Objection, Information) — see Chapter 26 for the full Act 843 mechanism. See Chapter 44.
- **Data Protection Act §17 — lawful basis for processing personal data** — the employee's voluntary self-edits of phone, address, gender, DOB, national ID, emergency contact, and bank details are processed under the contractual basis (their employment contract). The "Prefer not to say" option on the Gender select honours data-minimisation expectations under the same statute.
- **Data Protection Act §18 — data quality** — the self-service portal is the operational mechanism for satisfying §18; the data subject is the most reliable source of their own information and giving them a permanent self-service surface to update it is the cleanest way to keep the record current.
- **Ghana Labour Act, 2003 (Act 651)** — the My Leave tab surfaces the entitlement and balance figures (Annual, Sick, Maternity) computed against Act 651's statutory floors; the Leave module in Chapter 4 owns the calculus.
- **WCAG 2.1 AA (form accessibility)** — every form input on the page carries an explicit `aria-label`, every button has a visible focus state, the colour contrast across the cobalt-on-white system clears the 4.5:1 floor, and the tab order follows the visual reading order. The avatar upload button is keyboard-reachable (the underlying `<input type="file">` is hidden but focusable), the file input carries its own `aria-label`. The form-validation errors render as red text directly under the offending field rather than as a top-of-page summary — a "near the error" pattern WCAG 3.3.1 recommends. See Chapter 44.
- **ISO 30414 §4.1 (workforce composition)** — the self-edited Gender and Date of Birth fields are inputs into the workforce composition disclosures Chapter 13 produces. Honest self-disclosure here improves the quality of the analytics output downstream.
