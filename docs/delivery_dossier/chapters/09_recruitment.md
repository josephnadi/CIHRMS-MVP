# Chapter 9 — Recruitment + Public Careers

> *In one paragraph.* Recruitment is the two-faced module that takes the institute's hiring story from "we have a vacancy" all the way to "we sent the offer letter for signature" — without forcing anyone to leave the system at any point. The HR-facing side, under **Recruitment** in the sidebar, is where postings are drafted, applicants are dragged across a six-column kanban, CVs are downloaded, and offer letters are dispatched to Zoho Sign or DocuSign as a rendered PDF envelope. The public-facing side, served unauthenticated under `/careers/{id}`, is a clean job-detail page with an apply form and a single CV upload — no account, no login, no friction. Both sides write to the same `applicants` row, the same `job_postings` row, and the same `applicants.esign_envelope_id` field that the inbound webhook will later update when the candidate signs. The flow stops short of one thing that the architecture is otherwise set up for — the actual *hire* (turning a signed offer into an Employee) is still a manual step today.

## Where to find it

- **Sidebar location:** **Workforce** group → **Recruitment** (top-level item, briefcase icon). The page lives at `/jobs` with three Inertia routes underneath: `/jobs/{job}` for the posting + applicant board, `/jobs/{job}/applicants` for the dedicated full-screen pipeline view, and the public twin at `/careers/{job}` (no auth, no sidebar). The "Recruitment" tile is also surfaced in the app switcher (the 3×3 grid in the top bar) so any role with `recruitment.apply` or `recruitment.manage` lands one click away.
- **Roles that see it:**
    - **super_admin** and **ceo** — full access through the Gate::before wildcard; every posting, every applicant, every offer.
    - **hr_admin** — holds both `recruitment.manage` and `recruitment.apply`. They are the operational owner of the module: create postings, run the pipeline, send offers.
    - **manager / dept_head / employee / finance_officer / it_support / auditor** — `recruitment.apply` only. They can see the job list and the public posting view, but the "Post New Job" CTA, the kanban drag handles, and the "Send offer for e-sign" button are all hidden. The applicant pipeline at `/jobs/{job}/applicants` is gated by `permission:recruitment.manage` middleware so non-HR users get a 403 if they try to reach it directly.
    - **anonymous public** — the `/careers/{job}` page and the `POST /careers/{job}/apply` endpoint are explicitly outside the `auth` middleware group. The apply endpoint is `throttle:5,1` (five submissions per minute per IP) to discourage flooding.
- **Related modules:** Employees (Ch 3) — the destination of every hire; today's promotion from Applicant → Employee is still a manual HR step in the Employees module, not an automated event handler. Documents (Ch 13) — offer letters are rendered server-side as PDF and dispatched through the `ESignProvider` contract; the contract was built for Documents but is currently wired only to Recruitment's offer flow. Establishment (Ch 8) — job postings reference the role textually rather than against the `positions` table in the MVP; Phase 1 promotes the link to a foreign key. Notifications (Ch 16) — the candidate receives the eSign provider's email (Zoho Sign / DocuSign), not a CIHRMS email; in-portal stage-change notifications to HR are roadmapped. DPA / Privacy (Ch 26) — applicant personal data is processed under the consent + pre-contract basis of Act 843 §20.

## The data behind it

CIHRMS stores a recruitment process in three logical layers, all anchored on two simple tables:

- **The vacancy** — `job_postings` holds title, description (text up to 10 000 chars), optional `closes_at` date, a `status` string backed by the `JobPostingStatus` enum (`draft / open / closed / filled`), standard `timestamps`, and a `SoftDeletes` flag. Soft delete is in place but not exposed in the UI — the only operational way to take a posting off the public page today is to flip `status` away from `open`, which is itself not yet wired into the UI either. (See "Honest gaps" below.)
- **The applicant** — `applicants` holds the `job_posting_id` foreign key (cascade-on-delete), `name`, `email`, optional `cv_path`, a `status` string backed by the `ApplicantStatus` enum (`applied / shortlisted / interviewed / offered / hired / rejected`), and five eSign columns added in Wave 11: `esign_provider`, `esign_envelope_id`, `esign_status`, `esign_sent_at`, `esign_completed_at`. Same `SoftDeletes` flag. The applicant row is the single record that survives the whole journey from public application to (intended) hire.
- **The envelope** — there is no separate `offer_envelopes` table; the five `esign_*` columns on `applicants` *are* the envelope record. The rendered PDF itself is not persisted on the CIHRMS side — it lives only at the eSign provider after dispatch. This is a deliberate MVP simplification — the institute's compliance position is that the provider's signed-document URL (returned by `ESignProvider::signedDocumentUrl($envelopeId)`) is the authoritative artefact, and the institute downloads it on demand when needed.

What every reader of the screen needs to keep in mind:

- **The applicant is not an employee.** There is no foreign key, no automatic promotion, no shared identity. An `applicants.email` of `j.doe@example.com` and a `users.email` of `j.doe@cihrm.gh` are entirely separate rows. Hiring an applicant requires HR to create the Employee record manually in the Employees module (Ch 3) using whichever data they want to carry over from the applicant row. The `ApplicantHired` event class is defined and a listener (`CreateZohoContactOnHire`) is wired, but **no code path dispatches `ApplicantHired` today** — moving an applicant to the `hired` column is purely a status update.
- **eSign is the only externally-integrated step.** Everything else — posting creation, applicant capture, stage transitions, CV download — is entirely internal. The eSign provider is selected via `INT_ESIGN_DRIVER` (`zoho_sign` by default; `docusign` is the alternative). Webhooks land at `/webhooks/esign` and are dispatched against the applicants table by `esign_envelope_id`. There is no provider-side de-duplication — re-sending an offer to the same applicant after a previous attempt was voided creates a new envelope and overwrites the columns.
- **The public path is genuinely public.** `/careers/{job}` and `/careers/{job}/apply` are outside the `auth` middleware group entirely. The apply endpoint does not require a CSRF token (it does — but the public Inertia page hydrates with one) and the rate limit is per-IP at five-per-minute. Applicant data lands directly in `applicants` with no holding pen, no review step, no email confirmation back to the candidate from CIHRMS (only the eventual eSign provider email).
- **Status moves are not audited.** Unlike the Documents module's `document_events` table, there is no `applicant_events` log. The transition from `applied → shortlisted → interviewed → offered → hired` overwrites the `status` column in place. Phase 1's audit-pack work folds applicant transitions into the tamper-evident audit log that ships across the rest of the system.

## How it talks to other modules

- **Employees (Ch 3) — hire flow.** This is the gap. The intent is `ApplicantHired` → listener creates a `User` + `Employee` row from the applicant's `name` + `email` + the offer's salary/start-date → fires `EmployeeCreated` → benefits enrolment + welcome notification + identity-verification queue → done. In code, only step 1's event class and step 2's listener (`CreateZohoContactOnHire`) exist; the actual `Employee` creation is still a manual HR step from the Employees module. This is the single most-impactful Phase 1 deliverable on this module.
- **Documents (Ch 13) — offer letter pipeline.** The `ESignProvider` contract was built to be reusable but in the MVP it is wired only to Recruitment's offer letter. The Documents v2 "Phase 1: eSign integration for the documents pipeline" item extends the contract so any routed document can optionally be co-signed via Zoho Sign or DocuSign. The PDF is rendered with DomPDF (not TCPDF, which Documents uses) — convergence on TCPDF is in scope for the same Phase 1 work. Note also that CVs are **not** routed through the Documents pipeline; they live in the simpler `storage/app/public/applicant-cvs/` directory, with no annotation, no routing, no version history. Phase 1 promotes CVs to Documents-class objects so that `confidential` classification + per-viewer watermarks apply.
- **Establishment (Ch 8) — position references.** Job postings carry a free-text `title` rather than a foreign key to the `positions` table. Phase 1 promotes the field to a `position_id` so postings inherit the grade, step, salary band, and required qualifications from the establishment record. Until then a posting can be created for a role that does not exist in the establishment, which is a small workflow risk.
- **Notifications (Ch 16).** Recruitment does not send any in-portal notifications in the MVP. The candidate receives the eSign provider's own email when the envelope is dispatched (rendered and branded by Zoho Sign or DocuSign, not by CIHRMS). HR receives nothing — there is no "new applicant submitted to your posting" mail, no "envelope was signed" mail, no daily digest. This is Phase 1 backlog: a `RecruitmentNotificationMailer` mirroring the Documents pattern.
- **Audit Logs (Ch 24).** Recruitment does not write to the audit log today. Postings created, applicants moved, offers sent, envelopes signed — none of these surface in the audit trail. The Phase 1 audit-pack work folds them in via a new `recruitment_events` table mirroring `document_events`.
- **Analytics events.** The base `analytics_events` table exists and is written to from Employees and Leave, but Recruitment does not publish events to it. The `applicants_count` chip on each posting card, the hiring funnel, and the KPI tiles are all derived client-side from the same `jobs` payload — there is no aggregate query and no per-stage history.

## Standards touchpoints

- **ISO 30414 §4.2.3 — recruitment (turnaround time, quality of hire, time-to-fill).** The data model captures `applicants.created_at` and the timestamp of every status transition (via `updated_at`, which is overwritten on every PATCH) but there is no per-stage history. Turnaround-time can be approximated as `hired_applicant.updated_at − applicant.created_at` on the rare hire that does land, which is a usable proxy but loses fidelity once an applicant cycles back from rejected. Quality-of-hire is unreachable without a post-hire performance link. Phase 1 lands the `applicant_events` log that makes both metrics first-class; the Analytics export (Ch 24) consumes them. See Chapter 44.
- **Data Protection Act, 2012 (Act 843) §20 — basis for processing personal data (consent + pre-contractual).** Applicant data is collected from a public, unauthenticated form — the lawful basis is `consent` (the applicant has actively submitted) augmented by `pre-contractual necessity` (processing is necessary to consider entering an employment contract). The public Careers page does **not** display a privacy notice or a consent checkbox in the MVP — this is a real DPA gap. The institute's compliance position is that the act of clicking "Submit Application" constitutes informed consent under §17(1)(b), but a visible notice + tick-box pair is the auditor-defensible pattern and is Phase 1 work. See Chapter 26.
- **Data Protection Act §40 — right to erasure.** Soft delete is in place on the `applicants` table but is not surfaced in the UI; an applicant cannot self-request erasure through the public Careers page today. The cross-cutting Data Subject Request flow (Ch 26) accepts public submissions from ex-applicants and routes them to HR, where the deletion has to be done from the database console. Phase 2 wires this up as a real workflow.
- **Labour Act, 2003 (Act 651) §43 — equal opportunity / non-discrimination.** The system itself does not enforce anti-discrimination on shortlist decisions — that lives in HR policy, not in the code. What the system *can* show on audit is the full stage-by-stage transition history of every applicant, so an after-the-fact equal-opportunity review can be conducted. Once the `applicant_events` log lands in Phase 1, the institute can report on demographic-stage-transition rates if applicant demographics (optional, with consent) are captured. The MVP does not collect demographic fields.
- **Electronic Transactions Act, 2008 (Act 772) §10 — recognition of electronic signatures.** The offer letter signing path is the highest-stakes electronic-signature workflow in the institute — it is the legal acceptance of an employment contract. Wiring it through Zoho Sign or DocuSign (rather than the in-portal `signature_pad` canvas used by Documents) is a deliberate choice: §10's authentication and integrity requirements are met by the provider's own audit trail, certificate-of-completion PDF, and tamper-evident signature placement. The provider returns a `signedDocumentUrl` that the institute can archive on demand. See Chapter 26.

## Honest gaps

This module ships the public-facing path end-to-end (post → list → apply → kanban → offer envelope) and the eSign integration is real and exercised in tests. What it does not yet do:

- **The offer-letter signed callback does not complete the hire.** When the candidate signs the envelope, `ESignWebhookController::handle` writes `esign_status = 'completed'` and `esign_completed_at = now()` on the applicant row, then dispatches `EnvelopeStatusChanged`. *Nothing listens to `EnvelopeStatusChanged` to advance the applicant's `status` to `hired`, dispatch `ApplicantHired`, or create the Employee + User rows.* The pipeline pill changes colour to green ("Offer signed") and the HR team has to manually move the applicant to the `hired` column and then go to the Employees module and create the record by hand. The pieces are all built — the `ApplicantHired` event class, the `CreateZohoContactOnHire` listener that consumes it, the `EmployeeCreated` event that the Employees module fires — but the chain is not connected. This is the single most-impactful Phase 1 deliverable on this module.
- **No privacy notice on the public Careers page.** The form collects name, email, and CV with no visible privacy notice, no consent tick-box, and no retention statement. The lawful basis is defensible under DPA §20, but the UI presentation is not auditor-friendly.
- **No de-duplication on applicant email.** The same email can apply to the same posting twice and create two rows. There is no unique constraint on `(job_posting_id, email)`. Phase 1 backlog.
- **No applicant communication from CIHRMS.** Applicants receive only the eSign provider's own email when an offer is dispatched. There is no acknowledgement email when they apply, no "we have shortlisted you" mail when they move stage, and no rejection email. The public confirmation card is the only feedback they get from CIHRMS itself.
- **Reject is a one-click action with no reason capture.** Rejecting an applicant via the inline "Reject" button or the detail panel's "Move to stage" grid just flips the status — there is no required (or even optional) reason field. The Documents module's "Reject route" pattern (mandatory comment) is the target.
- **Postings are immutable after creation.** There is no "Edit posting" UI. Typos in title or description require either an admin to update directly in the database or a soft-delete + recreate. Phase 1 adds an edit slide panel.
- **The "Draft" status on `JobPostingStatus` is unreachable.** The enum case exists; the creation form has no "save as draft" option. The status chip on the library does filter for it, so a posting flipped to draft via the API would render correctly, but the UI cannot produce one.
- **No bulk actions on the applicant pipeline.** Moving five applicants from `applied` to `shortlisted` requires five separate drag operations or five separate clicks in the detail panel.
- **CVs are not in the Documents pipeline.** They live in the simpler `applicant-cvs/` directory with no `confidential` classification, no per-viewer watermarking, and no audit trail on download. Phase 1 promotes them.
- **Closing dates are not enforced server-side.** A posting whose `closes_at` is in the past will display the "Closed" tone on the public page (`v-if="j.status === 'open'"` is the only gate on the apply form), but the apply endpoint itself does not check the date — a client that hand-crafts the POST can still submit against an expired posting. The `JobPosting::isExpired()` helper exists but is not called by `RecruitmentController::apply`.

## What's planned next

Phase 1 of the government-grade roadmap lands six things on this module: (1) **the hire chain** — `EnvelopeStatusChanged` listener that, on `completed`, advances the applicant to `hired`, dispatches `ApplicantHired`, and creates the User + Employee rows from the applicant's name/email and the offer's salary/start-date, firing the existing `EmployeeCreated` cascade; (2) **the audit log** — `applicant_events` table mirroring `document_events`, with every stage transition, offer send, envelope status change, and rejection writing an immutable row that the audit pack (Ch 24) consumes; (3) **the consent UI** — visible privacy notice + tick-box on the public Careers form, retention statement, and the public Data Subject Request link for applicant erasure; (4) **the establishment link** — `position_id` foreign key on `job_postings` so postings inherit grade, step, salary band, and required qualifications from the establishment record (Ch 8), with the free-text `title` field demoted to a display override; (5) **the bulk + edit affordances** — multi-select on the pipeline kanban + list view, edit slide panel on the posting library, and a required-reason text field on Reject; (6) **CVs into the Documents pipeline** — every uploaded CV becomes a `Document` with confidentiality `confidential`, so the institute's standard share-guard, per-viewer watermark, and download audit trail apply. Phase 2 adds hCaptcha on the public apply endpoint, applicant communication (apply ack, shortlist mail, rejection mail, offer-sent confirmation), and the post-hire performance link that makes ISO 30414 quality-of-hire computable. See the government gap analysis (Chapter 46) for the full sequence.
