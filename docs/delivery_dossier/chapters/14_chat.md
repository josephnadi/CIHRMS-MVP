# Chapter 14 — Chat

> *In one paragraph.* Chat is the institute's internal one-to-one messenger — the place where two members of staff can have a private, threaded conversation without leaving CIHRMS. It is not the SMS console (that is Messaging in Chapter 15), and it is not the Notifications stream (Chapter 16). It is people-to-people, inside the building, on the same screen as their work. The May 2026 redesign reduced the directory to a single calm column with avatar, name, last-message preview, and a cobalt indicator bar that lights up when there is something unread. Behind the screen are three tables — `conversations`, `conversation_user`, and `chat_messages` — and a 4-second polling loop that keeps each open thread fresh until the institute moves to real-time websockets in Phase 2.

## Where to find it

- **Sidebar location:** **Organization** group → **Chat** (top-level item, `forum` icon). The same entry-point is wired to the global unread badge in the top bar — the number next to the icon is `unreadTotal`, recomputed on every page load and on every poll tick.
- **Roles that see it:**
    - **Every authenticated user** — Chat is visibility-flat. There is no `chat.view` permission and no `chat.manage` permission; every employee can open the directory, every employee can start a 1:1 with any other employee, every employee can send into a thread they are a participant of.
    - **super_admin, ceo, hr_admin, dept_head, manager, employee, finance_officer, auditor, it_support** — identical UI, identical capabilities. The sidebar item is marked `visible: true` unconditionally.
    - **Conversation visibility itself** is gated row-by-row by `ConversationPolicy::view`: you can only open a thread if you are on its participant pivot. A non-participant who guesses an id gets a 403 from the policy before the controller even loads the messages.
- **Related modules:** Employees (Ch 3) — every participant is an Employee + User pair; Messaging admin (Ch 15) — the sibling, separate, SMS-and-USSD outbound channel run by HR; Notifications (Ch 16) — the in-app notification stream is a *separate* surface, and at MVP there is no listener that converts a new chat message into a notification (see "Honest gaps" below); Audit Logs (Ch 24) — message create/delete events are not yet wired into the tamper-evident audit trail (also a gap); DPA & Privacy (Ch 26) — message retention and consent are addressed at policy level under DPA Act 843; Standards (Ch 44) — ISO 27001 A.13 communications-security mapping lives there; Roadmap (Ch 46) — the websocket cut-over (Laravel Reverb), file attachments, group chats, and audit hookup are all on the Phase 2 backlog.

## The screens

There are exactly two screens in this module: the **directory** at `/chat` and the **thread view** at `/chat/{conversation}`. There is no separate "compose" screen — composing is the act of clicking a person on the directory, which either opens the existing 1:1 thread with them or creates a brand-new one and drops you straight into it. That is intentional: a private message between two named people does not need a "to" field.

## Every button, every action

### Chat directory (`/chat`)

| Button / field | What it does | Who can use it | Why it exists |
|---|---|---|---|
| **Search box** ("Search name, staff ID, or email…") | Free-text filter against `users.name`, `users.email`, `users.staff_id` with a `LIKE` scan; submits on Enter (or via the "Search" button). Uses Inertia `router.get(... preserveState, replace)` so it doesn't pile up history entries. | Anyone logged in | Most lookups in an institute of even a few hundred staff are by name; the search is unindexed `LIKE` on purpose because the user table is small and we wanted the staff_id and email matches in the same query. |
| **Search** button | Same as pressing Enter — calls `router.get` with the current `q`. | Same | Mouse-and-trackpad users; a visible button doubles as a "did it run?" affordance. |
| **Clear** | Empties the query and re-fires the request without `q`. Only appears when a query is active. | Same | One-click undo. |
| **Continue chatting row** (avatar + name + last-message preview + time) | Clicks through to `/chat/{conversation}` — the existing thread. The cobalt left bar lights when `unread_count > 0`; a small red badge on the avatar shows the unread integer (capped to a small badge but never truncated in number). | Anyone logged in (any participant of the thread) | Surfacing your six most recently active threads at the top is how the screen pays for itself — it's where 80% of the clicks land. The bar + badge combination gives the eye two signals for the same fact. |
| **All people row** (avatar + name + position + department + "Start chat" CTA) | Clicks through to `/chat/with/{user}` which **opens-or-creates** the 1:1 conversation in a single round-trip and redirects to its show page. Idempotent — clicking the same person twice lands on the same conversation id. | Anyone logged in | A directory entry should not require thinking about whether a thread exists yet; the controller's `findOrCreateOneOnOne` (DB-locked) handles both paths transparently. |
| **Pagination** (link strip) | Standard Inertia paginator — 18 people per page, current page highlighted in cobalt, disabled chips greyed. | Same | Keeps the page bounded on institutes with hundreds of users. |
| **Unread pill** (page header, only shown when unread > 0) | Read-only — shows the institute-wide unread total for *you* across every thread you're a participant of. Computed server-side by `ChatController::unreadTotalFor()` (joins `chat_messages` with the participant pivot and counts messages newer than your `last_read_at`, excluding your own). | Anyone logged in | One number, top-right, that answers "do I have anything waiting?" — the same number is the global badge in the sidebar. |

> *Notes:* The directory is intentionally *one* clean list, not a card grid. The pre-redesign version was a 2-column zig-zag of coloured cards with no last-message preview; that gave no information value above what the sidebar already shows. The current layout uses a `rounded-2xl` container with `divide-y` rows, a 56px avatar column on the left, the body in the middle, and the meta on the right. The cobalt accent bar lives at `position: absolute; left: 0` with `transform: scaleY` transitions so the entry and exit are physical, not just colour swaps. On screens narrower than 640 px the time and CTA hide so the row is just avatar + name + preview.

| **Message bubble** — incoming (warm white, left-aligned, small teal avatar chip) | Read-only — no delete affordance shown to non-senders. | Same | A recipient can't retract someone else's message. |
| **Date separator** (e.g. `2026-05-23`) | Read-only visual divider rendered automatically whenever the date of the next message differs from the date of the previous one. | All readers | Long threads are unreadable without day anchors. The separators are letter-spaced uppercase in JetBrains Mono between fading hairline rules — they read as headings without being noisy. |
| **Composer textarea** | Bound to `form.body`. Auto-grows up to 140 px tall. Enter sends. Shift+Enter inserts a newline. `maxlength="4000"` matches the server-side validation. | The two thread participants — `ConversationPolicy::send` mirrors `view` | Single most-used input on the page; Enter-sends is the convention everyone expects. |
| **Send button** (paper-plane icon, cobalt gradient) | Pushes an optimistic bubble (with a `tmp-…` id and `_pending: true`) into `liveMessages`, then `form.post('chat.send')`. On 2xx, `form.reset('body')` clears the textarea and the props-watcher replaces `liveMessages` with the fresh server list. On 4xx/5xx, the optimistic bubble is removed and the error is shown under the composer. > *Notes:* The thread page hydrates from one Inertia render — `ChatController::show` loads the conversation, its participants and their employees + departments, the 30 most recent threads for the sidebar rail, and the latest 200 visible messages. Past that 200 we currently truncate (there is no infinite scroll yet — see "Honest gaps"). The "Live" badge is a *poll*, not a websocket: `setInterval(pollNow, 4000)` runs while the tab is visible, calls `GET /chat/{id}/poll?since=<lastSeenId>`, and merges any new messages into `liveMessages` with a `Set` dedupe so a concurrent post-send + poll never adds the same row twice. The `visibilitychange` listener pauses the poll while the tab is in the background and fires one immediate `pollNow()` when the user returns.

### "Compose" — the implicit flow

There is no compose dialog. The user journey is:

1. Click a row in **All people** (directory).
2. The browser is sent to `GET /chat/with/{user}`.
3. The controller calls `Conversation::findOrCreateOneOnOne($me, $other)` — wrapped in a `DB::transaction` with a `sharedLock` re-check to close the race where two concurrent "open chat with B" calls from A could otherwise create two rows for the same pair.
4. The user is redirected to `/chat/{conversation}` — the brand-new thread, ready to receive its first message.

The same path is what gets called when an unrelated module deep-links into chat (for example "Message this employee" buttons in roadmapped surfaces): construct the URL with the *other* user's id, send the browser there, the controller does the rest.

## The data behind it

Three tables sit behind the screens.

- **`conversations`** — one row per thread. Columns: `id`, `is_group` (boolean, defaults `false` — the schema is group-ready but the MVP creates only 1:1 rows), `title` (varchar, null for 1:1 threads since we render the participants' names client-side), `last_message_at` (timestamp, indexed, null on freshly-created threads), `created_at`, `updated_at`, `deleted_at` (soft-delete column).
- **`conversation_user`** — the participant pivot. Columns: `id`, `conversation_id`, `user_id`, `last_read_at`, `archived_at`, `muted_at`, `created_at`, `updated_at`. A unique compound index on `(conversation_id, user_id)` enforces "each user appears at most once per conversation". A secondary index on `(user_id, last_read_at)` accelerates the unread-counter query. `archived_at` and `muted_at` are present on the schema but **not yet surfaced in the UI** — they are the levers for Phase 2's "archive thread" and "mute notifications" affordances.
- **`chat_messages`** — the messages themselves. Columns: `id`, `conversation_id`, `sender_id` (FK to `users`), `body` (text, server-validated to ≤ 4000 chars), `deleted_for_everyone_at` (timestamp, null on live messages), `created_at`, `updated_at`. Two indexes: `(conversation_id, created_at)` for thread-load and `(sender_id, created_at)` for "messages I have sent".

A few invariants the rest of the system depends on:

- **One row per pair.** `findOrCreateOneOnOne` is the only way 1:1 conversations are created, and it does a sharedLock re-check inside a transaction so two parallel "open chat with B" requests from A never produce two `conversations` rows for the same pair. The corresponding test (`opens a 1-on-1 conversation and reuses it on the second open`) asserts this directly.
- **Soft-delete on conversations** (`SoftDeletes` trait) means a thread can be hidden from every list without losing the message history; the column is present but no UI yet exposes the "delete this thread" action — see "Honest gaps".
- **Per-message soft-delete via `deleted_for_everyone_at`.** A sender's "Delete" sets the timestamp; `ChatMessage::scopeVisible()` (`whereNull('deleted_for_everyone_at')`) is applied to every read path (`show`, `poll`). The retracted row stays in the DB.
- **Read receipts** are *per-user-per-conversation*, not per-message. The pivot's `last_read_at` is touched in three places: when a participant opens `/chat/{id}` (set to `now()`), when the polling endpoint returns at least one new message (set to `now()` for the active user — they're clearly reading), and when a participant sends into the thread (their own `last_read_at` is bumped to the new message's `created_at` so the unread badge doesn't tick up against themselves). The unread count derived from this is "messages in your conversations that came after your `last_read_at`, sent by someone other than you, not retracted".
- **`unreadTotal`** is computed by a single DB-builder query in `ChatController::unreadTotalFor()` — it joins `chat_messages` against `conversation_user`, filters on the current user's pivot, excludes the user's own messages and any retracted ones, and counts everything past `last_read_at` (or everything when `last_read_at` is null). It is recomputed on every Inertia render of `/chat` and `/chat/{id}` and on every poll tick, so the global badge stays in step without any client-side accounting.

What every reader needs to keep in mind:

- **Chat is not Notifications.** A new chat message does not, at MVP, create a row in the in-app notifications table, does not send an email, does not send an SMS. The only signal a recipient gets is the unread count on their sidebar — and that count is only refreshed when they navigate to a CIHRMS page or already have a chat thread open polling. A user who has Outlook open and no CIHRMS tab will simply not know. This is a real gap, scoped for Phase 2 (see the roadmap).
- **Chat is not Messaging.** Messaging in Chapter 15 is the SMS / USSD outbound channel — HR-side, one-to-many, gated behind `messaging.view` and `messaging.send`. It writes to `sms_messages` and `inbound_sms`, runs through a `SmsDispatcher` provider, and has nothing to do with this module. The two modules share zero database tables and zero controllers; the only thing they have in common is the noun "message".
- **Polling, not websockets.** The badge "Live · auto-refreshes" is a 4-second `setInterval` against `GET /chat/{id}/poll`, not a real-time channel. Two users in the same thread will see each other's messages with up to 4 seconds of latency (less if either of them happens to send something, because the responding side's optimistic UI + next poll closes the gap). Phase 2 swaps this for Laravel Reverb broadcasting with the same DB schema underneath.
- **No group chats yet.** The schema supports them (`conversations.is_group` + arbitrary `conversation_user` membership), but every code path that creates a conversation in the MVP creates a 1:1 with `is_group = false`. There is no UI for naming a thread, adding people, or removing them. `Conversation::otherParticipant($me)` returns `null` for groups by design, so any group-rendering surface needs to special-case the title — that work is on the Phase 2 list.

## How it talks to other modules

- **Employees (Ch 3)** — every participant on a conversation is a User, and almost every User in CIHRMS has exactly one Employee row attached to it. The directory query (`User::query()->with(['employee:id,user_id,position,department_id,avatar_path', 'employee.department:id,name'])`) eager-loads the Employee + department on every directory page so each row can render a position + department subtitle without an N+1 fan-out. The thread view does the same for the two participants. A user without an Employee row (rare — typically the seeded `super_admin` initial account) shows as just a name with the default "Employee" placeholder subtitle.
- **Messaging admin (Ch 15)** — *no relationship at runtime*. The two modules are deliberately separate; Chat is internal employee-to-employee, Messaging is org-wide outbound SMS. They share the `User` table only insofar as both end up addressing users by their phone-or-name. The class name `ChatMessage` (rather than just `Message`) was chosen explicitly to avoid colliding with the existing `MessagingDispatcher` / `MessagingController` / `SmsMessage` layer.
- **Notifications (Ch 16)** — **gap.** There is no `NewChatMessage` notification class, no listener that fires on `ChatMessage::created`, no entry in the institute's in-app notification stream when a message arrives. The unread badge is the only signal. Phase 2 wires this: a database-channel notification per recipient, with the same per-message debouncing the SMS layer uses, plus an opt-out toggle stored on `conversation_user.muted_at` (the column is already on the schema).
- **Audit Logs (Ch 24)** — **gap.** Message create/update/delete events do not write to the tamper-evident audit log at MVP. Soft-deletes on messages (`deleted_for_everyone_at`) leave a DB-level breadcrumb but not a hashed audit row. Phase 2 emits `ChatMessageCreated`, `ChatMessageRetracted`, and `ConversationCreated` events and routes them through the same audit listener used by Employees and Finance.
- **Top-bar unread badge** — the layout reads `unreadTotal` from every Inertia render in the auth zone (the shared prop) and renders the badge against the sidebar's `forum` icon. That number is recomputed by `unreadTotalFor()` on every page load and every chat poll, so the sidebar badge is at most one navigation or four seconds stale.

## Standards touchpoints

- **Data Protection Act, 2012 (Act 843) §17 — lawful basis for processing personal data.** Chat messages are personal data the moment they identify the sender. The institute relies on the *employment-contract* basis for processing — staff are notified at onboarding (via the acceptable-use policy) that internal communications carried over CIHRMS are subject to retention and lawful disclosure. The notification is the consent gate; absent it, this module cannot lawfully store conversation history. The full text and signing flow live in Chapter 26.
- **Data Protection Act §40 — right to erasure / §41 right to rectification.** Per-message retraction (`deleted_for_everyone_at`) is the operational answer to §41 — a participant can have a message corrected by retracting it and re-sending. True erasure of historical messages under §40 is handled by a dedicated redaction job (Phase 2) that nulls out `body` while keeping the audit metadata, not by the per-message Delete affordance.
- **Data Protection Act §27 — security of personal data.** Chat data lives behind the same Laravel session-auth + CSRF + HTTPS-in-production stack as the rest of the application; the database column for `body` is TEXT (not encrypted at rest at MVP — encryption at rest is a Phase 2 deliverable for sensitive columns across the application, see Chapter 26). The conversation policy enforces zero-trust access at the row level: even a privileged admin cannot read into a conversation they are not a participant of without first joining its participant pivot.
- **ISO 27001 Annex A.13 — communications security.** The 1:1 conversation policy (`participants()->where('users.id', $user->id)->exists()`) is the practical implementation of A.13.2.1 (information transfer policies and procedures) — message access is strictly bound to participant-pivot membership. A.13.2.3 (electronic messaging) maps to the 4-second poll plus the optimistic-UI feedback loop; the websocket cut-over in Phase 2 brings the same control under a real-time channel without changing the policy boundary.
- **ISO/IEC 27001 Annex A.8 — asset management.** Chat messages are an information asset in the institute's information-asset register; their classification is "Internal" by default with the right to be re-classified to "Confidential" by HR on a per-thread basis (the UI to do that is on the Phase 2 list — schema-side `conversations.title` and a new `sensitivity` column will carry the label).

## What's planned next

Phase 2 of the roadmap (see Chapter 46) lands six things on this screen, in the following order:

1. **In-app notifications + email digest** on new chat messages — fires `NewChatMessage` notification per recipient, gated by `conversation_user.muted_at`. This is the single highest-impact gap on the module today.
2. **Realtime via Laravel Reverb** — the same controller plus a `MessageSent` broadcast event on the `private-conversation.{id}` channel; the Vue page swaps the `setInterval` for `Echo.private(...).listen('MessageSent', ...)`. The polling endpoint stays as a fallback for connectivity-degraded modes.
3. **Audit log wiring** — `ChatMessageCreated` / `ChatMessageRetracted` / `ConversationCreated` events into the tamper-evident audit table (Chapter 24), with full hash-chained provenance for compliance read-outs.
4. **File attachments + image previews** — `chat_message_attachments` table, S3-or-local storage, MIME-typed inline previews for common image formats and PDFs (≤ 10 MB per file, matching the Documents module).
5. **Group chats** — UI to name a thread, invite participants, remove them, and rename; the schema already supports it. Includes a "leave thread" affordance on the participant pivot.
6. **Encryption at rest for `chat_messages.body`** — application-level envelope encryption with per-tenant KEK, addressing the §27 DPA gap noted above. Decryption gated by the same policy as read access today.

Beyond Phase 2: pinned messages, message search across all your threads (Meilisearch-backed), typing indicators, presence (online/last-seen), read receipts at message granularity (not just last_read_at), and emoji reactions. None of these block the MVP shipping; all of them are on the post-Phase-2 backlog.
