# Chapter 24 — Audit Logs

> *In one paragraph.* Audit Logs is the cross-module memory of every administrative act in CIHRMS. Each state-changing HTTP request — every POST, PUT, PATCH, and DELETE made by a signed-in user — fans through one piece of middleware that captures who-did-what-to-which-resource and dispatches it to an asynchronous queue. A worker writes the row, and as it writes it links the row to the previous one with a SHA-256 hash, forming an append-only chain that is verified end-to-end every night at 03:00 by a scheduled command and a notification that wakes super-admins on the first tamper. Auditors browse the same data through a single sealed-table page on the sidebar; the Auditor-General downloads it as part of the annual report pack from Chapter 24's sibling export. This is the chapter that backs every other chapter's claim of "auditable" — payroll runs, finance journals, identity verifications, whistleblower triage, DPA fulfilments — and is itself the answer to Data Protection Act §28's "appropriate technical measures", ISO 27001 Annex A.12.4, and the Cybersecurity Act 2020 (Act 1038) §41 record-keeping duty.

## Where to find it

- **Sidebar location:** **System** group → **Audit** (top-level item, `history` icon, oxblood `#7c2d12` accent). The page lives at `/audit-logs` and is rendered by `AuditLogController::index` into `resources/js/Pages/AuditLogs/Index.vue`. The dashboard's "Audit Logs" tile in the app switcher (the 3×3 grid in the top bar) points at the same route via the `modules.audit-logs` redirect. The masthead carries an editorial-sovereign "AUDIT TRAIL · TAMPER-EVIDENT" eyebrow with the subhead "Every administrative act sealed into a SHA-256 hash chain · chain-of-custody for the Republic", so an auditor opening the page is left in no doubt about the posture.
- **Roles that see it:**
    - **super_admin** and **ceo** — see every row, every payload, every IP. CEO mirrors super-admin via the wildcard permission grant in `RolePermissionSeeder`; super-admin reaches it through the `before()` hook on every policy and the explicit `audit.view` grant.
    - **auditor** — by seed the auditor role holds `audit.view` as its primary read lane (alongside `reports.view` and `statutory.export`). This is the segregation-of-duties posture intended by the gap analysis: the auditor role exists outside the HR chain and is the default reader of this page, the AG pack, and the verify-chain output.
    - **hr_admin** — **not granted `audit.view` by the role seeder.** HR can be the *subject* of an audit log entry but cannot read the register. This is deliberate — a payroll officer who can erase the trace of their own action is not the control posture the Auditor-General will sign off on. Promoting HR's read lane requires an explicit grant on the per-user `permissions` JSON column, which is itself written through the User policy and therefore itself audit-logged.
    - **finance_officer**, **dept_head / manager**, **it_support**, **employee** — no register access. They appear as *actors* on rows, and a finance officer can see the audit timeline of a payroll run from the run page itself (Chapter 19), but `/audit-logs` is closed.
- **Related modules:** This is the cross-module hub. Every state-changing route inside the `Route::middleware(['auth', 'audit'])` group writes here. The eight modules that depend on this chapter most directly: **Payroll Engine (Ch 19)** — every Draft → Calculated → Approved → Paid → Reversed transition is reconstructable here, and the Auditor-General Report Pack bundles a chain-verification report alongside the payroll CSVs; **Disbursements (Ch 22)** — GhIPSS dispatches and MoMo callbacks pin to an audit row each; **Loans (Ch 21)** — every approve / disburse / waive action; **Off-boarding (Ch 10)** — settlement approvals; **Identity / Ghana Card (Ch 25)** — successful verifications and provider failures; **DPA & Privacy (Ch 26)** — every Acknowledge / Fulfil / Reject decision on a `data_subject_requests` row, both inside that row's own `audit_trail` JSON and here in the sealed chain; **Whistleblower (Ch 27)** — investigator actions (the case content itself is segregated, but the *fact* of an investigator action is audit-logged); **Identity & Access (Ch 25)** — SSO admin actions, two-factor enrolment, password changes. The webhook fan-out in Chapter 27's sibling section also writes here on every external delivery attempt.

## The data behind it

There is exactly one storage table — `audit_logs` — and one model — `AuditLog` — and one job — `WriteAuditLog` — and one middleware — `AuditTrail` — and one command — `audit:verify-chain` (with `audit:backfill-chain` as its idempotent sibling). The whole chapter is the story of those five files.

### Schema

The base table was created by `2026_05_12_083700_create_audit_logs_table` and extended by `2026_05_25_000007_add_tamper_evident_audit_columns`.

| Column | Type | Notes |
|---|---|---|
| `id` | bigint | PK. |
| `chain_position` | unsignedBigInteger, indexed, nullable | Monotonic 1-based counter assigned by `WriteAuditLog` inside a transaction. Nullable because legacy rows from before the May-25 migration begin life un-chained and are filled in by `audit:backfill-chain`. |
| `previous_hash` | char(64), nullable | The previous row's `row_hash` — i.e. the link in the chain. Null on the genesis row. |
| `row_hash` | char(64), nullable | SHA-256 of this row's canonical JSON (including `previous_hash`). Filled in by `WriteAuditLog` after the row's id and timestamps are settled. |
| `user_id` | foreignId, nullable | `nullOnDelete` — a deleted user does **not** erase their audit trail. The pair `(user_id, action)` lets you reconstruct an actor's history even after their account is gone. |
| `action` | varchar | Currently the route name; falls back to `unnamed-action`. A future refactor will let services pass an explicit domain event name (`payroll.calculated`, `loan.disbursed`) but the existing rows are stable. |
| `route_name` | varchar, nullable | The same route name, kept separately for query convenience and so a future widening of `action` (e.g. to include sub-events on a route) doesn't lose the original endpoint identity. |
| `method` | varchar(10) | HTTP verb. Only POST/PUT/PATCH/DELETE are written; the middleware skips GET. |
| `path` | varchar | The request path with leading slash. |
| `ip_address` | varchar(45) | IPv4 or IPv6, captured via `$request->ip()`. |
| `user_agent` | text | The raw UA string, no normalisation. |
| `payload` | json, nullable | The request body with sensitive fields stripped and upload-file instances reduced to `{name, mime, size}` descriptors. Strings longer than 500 characters are truncated with `...`. |
| `created_at` / `updated_at` | timestamps | Standard. `updated_at` is essentially read-only — anything touching it after the write is, by definition, tampering. |

Indices: `chain_position` on its own (the verifier walks the chain in order); the implicit primary-key index on `id`. There is no compound `(user_id, created_at)` index yet — the gap analysis flagged it as a Phase-2 add once row counts exceed about ten million per installation.

### Tamper-evidence: how the chain actually works

The hash chain is **real** and **enforced at every write**, not a Phase-2 plan. The mechanism, in five steps:

1. **The middleware (`AuditTrail`) captures the request** after the response has been generated — so an exception that aborts the request also aborts the log, which is correct because a 500-response is not a state change. The middleware skips unauthenticated requests and skips non-mutating verbs (GET, HEAD, OPTIONS).
2. **The middleware sanitises the payload** before dispatching: strips `password`, `password_confirmation`, `current_password`, `token`, `_token` (so the captured row never carries credentials), reduces any `UploadedFile` to `{__file: true, name, mime, size}` (so the queued job is serialisable), and truncates any string over 500 characters with `...` (so a malicious user can't inflate the table by pasting megabyte payloads).
3. **The middleware dispatches `WriteAuditLog` to the `audit` queue.** The job is a `ShouldQueue` implementation with `$tries = 3`. Putting the write behind a queue is what keeps the request-response cycle from paying a database transaction cost on every mutation; the trade-off is that a queue worker outage produces a backlog, not a data loss, but Wave 1 (task W1.24.b) adds a Horizon supervisor alert if the `audit` queue depth exceeds 1,000 pending jobs.
4. **The job opens a database transaction, takes a `lockForUpdate()` on the latest row, computes `chain_position = latest.chain_position + 1` and `previous_hash = latest.row_hash`, creates the row with a null `row_hash`, and then computes the SHA-256 of the row's canonical JSON (including the `id`, `chain_position`, `created_at` and the just-computed `previous_hash`) and saves the row with `saveQuietly()` so the hash assignment itself does not produce a recursive audit entry.** The `lockForUpdate` is what gives the chain a total order even under concurrent writes — a parallel `WriteAuditLog` from another request will block on the row-level lock until the first transaction commits, then read the new tail and link off it.
5. **The verifier (`audit:verify-chain`) walks every row in `chain_position` order**, asserts that the positions are sequential, that each row's `previous_hash` equals the prior row's `row_hash`, and re-computes the SHA-256 of the canonical JSON and compares it constant-time with `hash_equals`. Any mismatch breaks the loop, the command prints `Audit chain BROKEN at position X (audit_logs.id=Y): {reason}` to stderr, exits non-zero, and — if invoked with `--notify` — sends `AuditChainBroken` to every `super_admin` via mail + database.

The canonical JSON used by the hash is fixed in `AuditLog::canonicalJson()`. The field order, the timestamp format (ISO-8601 with `toIso8601String()`), and the `JSON_UNESCAPED_SLASHES | JSON_UNESCAPED_UNICODE` flags are load-bearing: any change to that method is a chain-breaking change and must be paired with a re-hash migration. The model carries a comment to that effect.

### What the chain protects against

- **Direct database UPDATE on an `audit_logs` row** — say a malicious DBA changes the `path` on the row that records a fraudulent payroll approval. The next `audit:verify-chain` run re-hashes the row, computes a SHA-256 that no longer matches the stored `row_hash`, and breaks the chain at that position. The Pest test `verify-chain command detects a tampered row` proves this end-to-end.
- **Insertion of a new row in the middle of the chain.** The `chain_position` column is a monotonic counter; a row inserted at position 7.5 would have to lie about its position, and either it claims `chain_position = 7` (breaks uniqueness in the index and shows up as a duplicate) or it claims `chain_position = 8` (and now the row that used to be position 8 doesn't link back to it via `previous_hash`). Either way the verifier breaks.
- **Deletion of a row.** The verifier asserts `chain_position` is sequential starting at 1 (or at the `--from` value); a gap is detected on the next position-check.
- **Re-writing the chain end-to-end with new hashes** (the only mathematically valid tampering path). This requires the attacker to re-hash every row from the tampered position to the present, and — crucially — to do so without the chain being verified between the tamper and their re-hash. The nightly schedule narrows that window to a single calendar day; pairing the verification with off-site replication (Phase 3) collapses it to seconds.

### What the chain does **not** protect against

- **A bug in `AuditLog::canonicalJson()` that drops a field silently.** The canonical JSON is the contract; if a developer adds a column to `audit_logs` and forgets to add it to the canonical JSON, the column will be invisible to the hash and therefore freely mutable post-write. This is why the model comment is explicit and why a Phase-2 test will assert that every fillable column is named in the canonical JSON.
- **Compromise of the queue worker.** A worker writing fake rows can compose a valid chain — the chain proves no rows have been *altered* after the fact, not that the rows were *honestly written* in the first place. The next layer of defence is the route middleware: every audit row's `action` corresponds to a route name that exists in the codebase, and a fake row claiming `action = payroll.approve` is detectable by cross-referencing the `payroll_runs.audit_log_id` column when it is present.
- **Clock skew.** `created_at` is part of the hash; if the application server's clock jumps, two adjacent rows can carry timestamps that look out of order to a human reader without breaking the hash. Wave 1 (task W1.24.c) adds an NTP-drift assertion to the verifier so a sustained skew surfaces as a chain warning rather than a silent timeline corruption.

### Backfill of legacy rows

Installations that upgraded across the May-25 migration have `audit_logs` rows that pre-date the chain columns — `chain_position`, `previous_hash`, and `row_hash` are all null on those rows. The `audit:backfill-chain` command walks them in `(created_at, id)` order, assigns sequential positions, and computes hashes off the previous row's hash, exactly as a live write would. The command is idempotent (running it on an intact chain reports "no gaps") and supports `--dry-run` for change-control review. It also gracefully extends a partial chain — if some rows are already hashed and a legacy row is found after them, the legacy row is chained off the tail rather than restarting the chain from genesis. The four Pest tests in `AuditChainBackfillTest` lock these behaviours in.

## How it talks to other modules

The audit log is a passive sink: it does not push to anywhere; it accumulates. But it is touched by almost every other module in the system, in one of three ways:

- **Implicit (the default).** Every state-changing route inside the `Route::middleware(['auth', 'audit'])` group in `routes/web.php` writes here automatically. There is no per-controller wiring; adding `'audit'` to a route group is the entire contract. The block at line 199 covers profile, employees, departments, leave, tickets, payroll runs, payments, audit-log search itself (no, not really — it's a GET, so it skips), reports, announcements, notifications, AI assistant, integrations, webhooks, the Auditor-General Report Pack, the whistleblower investigator dashboard, the DPA fulfilment lane, and the SSO provider admin pages. A second smaller `'auth', 'audit'` group at line 1057 covers SSO provider admin specifically (kept separate so the routes can sit visually alongside the SSO controllers).
- **Explicit, via the per-row `audit_trail` JSON.** Some modules keep their own per-row timeline in a JSON column on the same row — `data_subject_requests.audit_trail` (Chapter 26) and `whistleblower_actions` (Chapter 27) are the two examples. These rows give a human-readable, contextual timeline scoped to a single case that the audit register cannot match (the register is system-wide and unsorted by case). The per-row trail and the sealed chain are **both** written for the same action — the per-row trail by the service, the sealed chain by the middleware — so the auditor can choose either lens.
- **Explicit, via foreign keys back into `audit_logs`.** A handful of high-stakes tables carry an `audit_log_id` pointer at the moment of a critical action — e.g. `payroll_runs.approval_audit_log_id` (Phase 2) points at the sealed row that recorded the dual-control approve. This pattern lets the run page render "Approved by Joseph Nadi · audit row #12,341" without a join through user-action heuristics, and lets a forensic audit trace from a payroll row back to its sealed proof.

The eight modules that pin themselves to the audit chain most heavily, in order of how often they appear in a typical day's rows:

1. **Payroll Engine (Chapter 19)** — every Draft / Calculate / Approve / Mark-Paid / Reverse transition; every statutory return generation; every GhIPSS bank-file regeneration.
2. **Disbursements (Chapter 22)** — every GhIPSS dispatch, every MoMo provider callback, every reconciliation entry.
3. **Loans (Chapter 21)** — every Apply / Approve / Disburse / Waive on `loan_accounts`.
4. **Finance Hub (Chapter 20)** — every journal-entry post (the JV refusing to balance is an audit row in its own right), every Chart-of-Accounts mutation, every bank-reconciliation match.
5. **Identity & Ghana Card (Chapter 25)** — every verification submission and outcome; every re-verification reminder; every provider failure with the failure reason verbatim in the payload.
6. **DPA & Privacy (Chapter 26)** — every Acknowledge / Fulfil / Reject on the DPO queue; every export-bundle generation with the SHA-256 of the bundle in the payload.
7. **Whistleblower (Chapter 27)** — every investigator triage, every internal action, every onward referral. The case *content* is segregated and does not appear in this register; only the *fact* of the action does.
8. **Identity & Access (Chapter 25's sibling)** — every SSO provider admin change; every password change; every 2FA enrolment or reset; every role grant or revocation.

### The nightly schedule

The chain is verified once a day, at 03:00, by `routes/console.php`:

```php
Schedule::command('audit:verify-chain --notify')->dailyAt('03:00')->withoutOverlapping();
```

The time is chosen on purpose — late enough that the day's writes have settled, early enough that a broken-chain alert is in the super-admin's inbox by the time HR walks in. `--notify` makes the command dispatch `AuditChainBroken` to every user with `role = super_admin` (note: the role is read from the legacy `users.role` enum, not the DB-backed `roles` pivot; tightening this to also consider DB-grants is a Phase-2 task). The notification ships via mail and database channels — deliberately not Slack or SMS, because a broken chain is a high-severity but not user-facing event and the audit trail itself is the primary record of who saw the alert and when.

The notification body names the broken position, the `audit_logs.id`, the failure reason ("row_hash mismatch (data tampered)", "previous_hash mismatch", or "expected chain_position N, got M"), and the count of rows verified before the break. It also pre-types the recovery command: `php artisan audit:verify-chain --from={broken_position - 5}` so an investigator can replay the surrounding rows without computing the offset themselves.

### The Auditor-General Report Pack

When the institute's external auditor — under the Auditor-General Act 2000 (Act 584) — asks for the annual evidence pack, the operator goes to `/reports/auditor-general`, enters the fiscal year, and clicks Generate. The pack is built by `AuditorGeneralReportPack::generate()` and bundles fourteen artefacts into a single ZIP:

1. `MANIFEST.md` — an index with the SHA-256 of every other file in the pack (so any tampering between generation and submission is detectable).
2. `payroll/runs.csv` and `payroll/run_lines.csv` — every approved/paid run with full per-employee granularity.
3. `statutory/PAYE-{year}.csv`, `SSNIT-{year}.csv`, `TIER2-{year}.csv`, `NHIA-{year}.csv` — concatenated statutory returns.
4. `statutory/bank_files/` — verbatim copies of the GhIPSS disbursement files.
5. `identity/verifications.csv` — Ghana Card register with the PIN masked to `GHA-•••••••••-N`.
6. `loans/accounts.csv` and `loans/outstanding.csv` — full loan ledger plus year-end open balances.
7. `offboarding/cases.csv` — exits and final settlements.
8. `whistleblower/summary.csv` — **counts only**, segregated by Whistleblower Act §10.
9. `audit/chain_verification.txt` — the literal stdout of `php artisan audit:verify-chain`, with the total row count, the first and last chain position, the last row's hash, and the exit code.

Generation requires `statutory.export` and a fresh 2FA challenge (`2fa:fresh` middleware on the generate route). Download requires `statutory.export` and is restricted to filenames matching `/^AG-\d{4}-\d{8}-\d{6}\.zip$/` to prevent path traversal. The pack itself is sealed by the manifest's SHA-256 of every file, so the auditor can verify on their machine that nothing changed between download and presentation. See the route block in `routes/web.php` lines 644–652.

## Standards touchpoints

- **Data Protection Act, 2012 (Act 843) §28 — security of personal data.** §28(1) requires the data controller to take "appropriate technical and organisational measures" to prevent unauthorised processing, loss, or destruction. The tamper-evident chain is the system's principal technical measure for §28(1)(c) "ability to detect intrusion or interference"; the daily verification with super-admin notification is the §28(1)(d) "process of regularly testing" requirement; the masking of payload fields (`password`, `_token`) is the §28(1)(b) minimum-necessary principle as applied to logs. See Chapter 26 for the full DPA mapping and Chapter 44 for the standards register.
- **Data Protection Act §29 — security breach notification.** A broken chain is a "security breach" under §29(2) and the `AuditChainBroken` notification is the first link in the breach-notification chain — the DPO (the auditor role) is in the recipient set because the role seeder grants `audit.view` to auditor and the auditor role doubles as the DPO under Chapter 26.
- **Cybersecurity Act 2020 (Act 1038) §41 — record-keeping by critical information infrastructure.** §41 obliges designated CII operators to keep records of all significant security events for a period prescribed by the Cyber Security Authority. CIHRMS does not yet self-designate as a CII operator, but the audit register is built to that standard — sealed, append-only, queryable, and exportable.
- **ISO/IEC 27001:2022 Annex A.12.4 — logging and monitoring.** A.12.4.1 (event logging), A.12.4.2 (protection of log information), A.12.4.3 (administrator and operator logs) all map directly to this chapter. The middleware-captured trail is A.12.4.1; the SHA-256 chain is the protection of A.12.4.2; the explicit super-admin and CEO grants are A.12.4.3.
- **ISO/IEC 27001:2022 Annex A.5.7 — threat intelligence** *(indirect)*. A broken-chain notification is a threat intelligence signal in its own right; routing it to super-admins via mail and database channels is the minimum response posture.
- **NIST SP 800-92 (Guide to Computer Security Log Management) §4 — log integrity protection.** The hash-chain pattern is named in §4.4 as the standard mechanism; this chapter implements it directly.
- **Auditor-General Act 2000 (Act 584) §13 — access to records.** The Auditor-General Report Pack is the operational answer to §13's "all books, records, returns, reports". The pack is sealed by SHA-256 manifest so a presented copy can be cryptographically tied back to the source.
- **Whistleblower Act 2006 (Act 720) §10 — confidentiality.** The audit register records the *fact* that an investigator triaged a case but never the content; the AG pack ships counts only. This segregation is built into the AG pack's `writeWhistleblowerSummary()` method, which selects `category, severity, status, count(*)` and nothing else.
- **Public Financial Management Act 2016 (Act 921) §52 — record-keeping by accounting officers.** Payroll, disbursements, and loans all settle into journal vouchers that are themselves audit-logged on post; the chain is what gives §52's "complete and accurate records" its technical teeth.

## What's planned next

Phase 2 lands four things directly on this module:

1. **Action vocabulary widening.** Today the `action` column is the route name. Phase 2 introduces an explicit `audit_event` parameter on a small service-layer helper (`AuditLogger::record('payroll.calculated', $payload)`) so service-level events that are not 1:1 with a route (a scheduled retry, an internal queue handover, a soft-delete cascade) can be captured with a stable name. The existing route-name path stays as the default.
2. **Auditor-friendly detail page.** The current page is a register only — there is no per-row detail. Phase 2 adds `/audit-logs/{id}` with a single-row view that pretty-prints the JSON payload, decodes the route name into the file:line of the handler (via a route-name → controller-action map), shows the row's hash and the previous and next row in the chain, and offers a one-click "Verify this row" button that re-hashes the row and the surrounding two and reports the result. The detail page is what makes this module usable to non-developer auditors during a live audit visit.
3. **Per-row foreign keys.** A small number of high-stakes tables (`payroll_runs`, `loan_accounts`, `data_subject_requests`, `offboarding_settlements`) get an `audit_log_id` pointer to the sealed row that recorded their last state-changing action. This collapses "what audit row recorded this approval?" from a free-text search to a single join, and is the foundation of the forensic-audit feature described in Chapter 47.
4. **Off-site sealed replication.** The daily verification narrows the tamper-detection window to 24 hours. Phase 3 ships the chain to an off-site append-only object store (S3 Object Lock or equivalent) every fifteen minutes so the window collapses to that interval, and any divergence between the local chain and the replicated chain is itself a sealed event.

Phase 3 then extends the model into two areas that the gap analysis flagged as government-grade requirements: (a) **integration with the Cyber Security Authority's incident-reporting workflow** so a broken-chain incident automatically opens a CSA case under Act 1038 §59; and (b) **a write-only WORM mode** in which a deployment can be configured to refuse `UPDATE` or `DELETE` on `audit_logs` at the database privilege level, leaving the verifier as a belt-and-braces second line of defence rather than the primary one.
