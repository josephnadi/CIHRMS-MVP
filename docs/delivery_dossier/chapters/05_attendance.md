# Chapter 5 — Attendance

> *In one paragraph.* Attendance is the institute's time-and-attendance ledger — every clock-in, clock-out, late arrival, and overtime hour for every employee, computed from raw punches into a day-by-day summary that payroll trusts as its source of truth. The same ledger is fed from four channels (biometric devices, the public web kiosk, the employee's own browser with GPS, and HR-typed manual entries), reconciled against shifts and approved leave, and exposed in three views — an org-wide register for HR, a personal monthly card for each employee, and a kanban for managers who decide on correction requests. It's the module the gap analysis (Chapter 44) flagged as Phase-2 work for Ghana public-sector compliance with the Labour Act §35; it shipped early and is in use today.

## Where to find it

- **Sidebar location:** **Workforce** group → **Attendance** (clock-face icon). The module tile is also pinned in the app switcher (the 3×3 grid in the top bar) so any role with `attendance.view` lands on the org-wide register in one click. Employees without `attendance.view` see a "My Attendance" entry instead, which goes to `/attendance/me`.
- **Roles that see it:**
    - **super_admin** and **ceo** — every page, every employee, every device.
    - **hr_admin** — every page, plus the only role that can record manual entries (`attendance.manage`) and define new shifts (`attendance.shift_manage`).
    - **dept_head / manager** — the org-wide register, plus the Corrections kanban for requests from their reports. They can approve corrections (`attendance.approve`) but not change shift definitions.
    - **employee** — only the personal "My Attendance" page at `/attendance/me`. They can clock themselves in/out (`attendance.clock_self`) and raise correction requests (`attendance.correct`); they never reach the org-wide register.
    - **finance_officer, it_support** — `attendance.correct` only (they can raise their own corrections); they don't see the register.
    - **auditor** — read-only on the register and the ability to raise their own corrections; no approvals, no shift edits.
- **Related modules:** Employees (Ch 3) — every attendance row belongs to one; Leave (Ch 4) — approved leave days show up as "On Leave" status in the daily summary and are excluded from the absent denominator; Payroll (Ch 19) — `PayrollService` reads `attendance_summaries.overtime_hours` straight off this module to compute overtime pay; Kiosk (Ch 29) — the unauthenticated `/kiosk` device hands events off to the same `AttendanceService::record()`; Identity Verification (Ch 25) — biometric devices and the kiosk's future face-recognition path tie into the same Ghana Card verification stack; Notifications (Ch 14) — correction-request events fire analytics rows that drive the manager-side review badges; Standards Benchmark (Ch 44) — this module is the Labour Act §35 and ISO 30414 §4.5 absenteeism touchpoint.

## The data behind it

CIHRMS stores attendance across four tables, with two more supporting catalogue entities:

- **`attendance_records`** — the raw clock event ledger. One row per punch, never edited, only inserted. Carries `employee_id`, `device_id` (nullable — manual entries don't have one), `source` (`biometric` / `gps_mobile` / `web_kiosk` / `manual` / `webhook`), `direction` (`in` / `out`), the device-side `event_at` timestamp, the server-side `recorded_at` timestamp (so an offline device that batches up a day and replays at 6 pm doesn't lie about when the punch happened), optional `geo_lat`/`geo_lng`, the device's confidence score (string, raw from the vendor), the `recorded_by` user id for manual entries, the required reason for manual entries, and the full vendor payload as JSON in `raw_payload`. Indexed on `(employee_id, event_at)` and `(device_id, event_at)` so the daily recompute is O(records-for-this-day) and not a table scan.
- **`attendance_summaries`** — the derived per-day record that everything else reads. One row per (employee, day) pair, enforced by a unique index. Carries the computed `status` (`present` / `late` / `half_day` / `absent` / `on_leave` / `holiday` / `weekend`), `first_in` / `last_out` clock times, `hours_worked` and `overtime_hours` (both decimal(5,2)), boolean `is_weekend` / `is_holiday` flags so dashboards can colour cells without re-deriving the calendar, and the "dominant" `source` for the day. This table is the payroll contract — `PayrollService` reads `overtime_hours` straight from here.
- **`attendance_corrections`** — the request/decision lifecycle. Carries the optional FK to the `attendance_record` being corrected, the requester user id, the requested timestamp + direction, the reason text, the `status` (`pending` / `approved` / `rejected`), the reviewer's user id, the decision timestamp, and the decision notes. Soft-deleted (so dismissed requests stay in the audit log). Indexed on `(employee_id, status)` and on `status` alone — the manager queue sorts by status and the employee history filters on employee.
- **`shifts`** — the catalogue of named working-time patterns. `code` (unique short identifier), `name` (display label), `start_time` / `end_time`, `grace_period_minutes`, `full_day_hours` / `half_day_hours`, the array of allowed `working_days`, an optional `department_id`, an `is_active` flag, and soft-delete.
- **`shift_assignments`** — the bridge between an employee and a shift, with an effective-from date and an optional effective-to. Soft-deleted. The "current" assignment is found by `effective_from ≤ today` and (`effective_to` is null OR `effective_to ≥ today`), ordered by latest effective-from.
- **`biometric_devices`** — registered hardware. `code` (unique, e.g. `DEV-ACCRA-MAIN-01`), `name`, `vendor` (`zkteco` / `hikvision` / `suprema` / `other`), `location`, optional `department_id`, the per-device HMAC `shared_secret` (encrypted via Laravel's `encrypted` cast and hidden from JSON serialization), optional `geo_lat` / `geo_lng` / `geo_radius_m` for geofencing, an `is_active` flag, and a `last_seen_at` timestamp that the ingestion service updates every time it accepts a payload from this device.

What every reader of the screen needs to keep in mind:

- **Summaries are derived, not authoritative.** Every time a new `AttendanceRecord` lands (from any source — biometric, kiosk, self-service, manual, or a webhook), `AttendanceService::recomputeDailySummary()` runs synchronously for the affected day. If you ever need to rebuild the summary you can just call the service again; the source of truth is always the raw records.
- **The OvertimeCalculator implements Ghana Labour Act §35.** Beyond 8 hours on a weekday accrues at 1.5× (`weekday_15x`); any hour worked on a weekend or public holiday accrues at 2× (`premium_2x`); beyond 12 hours on any day flips into the 2× bucket regardless of weekday/weekend. The `overtime_hours` column stores **premium-weighted equivalents** — payroll multiplies them by the basic hourly rate and adds the result to gross pay, no further multiplier needed.
- **Public holidays come from the `PublicHoliday` model.** Any day matching a `PublicHoliday` row is automatically flagged `is_holiday = true` in the summary and statused `holiday`, regardless of whether anyone clocked in or out. This honours statutory holidays (Independence Day, AU Day, Eid, Christmas) without HR having to remember.
- **Approved leave wins over absence.** `AttendanceService::isOnApprovedLeave()` checks for any `LeaveRequest` with `status = approved` covering the day; if it matches, the day is statused `on_leave` and excluded from the working-days denominator in `aggregatePeriod()`. That's why the absenteeism rate stays honest — leave isn't punished as absence.
- **Geofence is enforced server-side.** When a clock event has both a `device_id` and a GPS pair, the service computes the haversine distance from the device's stored coordinates; anything outside `geo_radius_m` throws a `DomainException` ("Clock event 412m outside 50m geofence for device DEV-ACCRA-MAIN-01") and the punch is rejected before insert. For self-service clocking through `/attendance/clock`, the service auto-pairs the punch with the nearest registered device within a half-degree latitude/longitude box (~55 km) before applying the same geofence.
- **Manual entries must carry a reason.** The `Manual` value of the `AttendanceSource` enum has `requiresReason() = true`. The service throws `DomainException` if the reason is missing, the FormRequest validates 5-500 chars, and the UI marks the field required. Three layers of the same guard.
- **Biometric secrets never leave the database in cleartext.** The `shared_secret` column is cast `encrypted` and listed under `$hidden` on the model. Even a `dd($device)` in tinker won't print it.

## How it talks to other modules

- **Leave (Ch 4)** is read by `AttendanceService::isOnApprovedLeave()` on every summary recompute. A day covered by an `approved` leave request is automatically statused `on_leave` and excluded from the working-days denominator. There is no double-counting — a day can be either `on_leave` or `absent`, never both.
- **Payroll (Ch 19)** reads `attendance_summaries.overtime_hours` directly inside `PayrollService::calculateLine()`. The sum across the pay-run period multiplied by the basic hourly rate (`basic / 173.33` — the standard 52w × 40h / 12 monthly hours) becomes the `overtime_pay` line on the payslip. There is no separate "overtime approval" step in the MVP — once a punch lands and the summary recomputes, those overtime hours are billable.
- **Kiosk (Ch 29)** is a thin shell over the same `AttendanceService::record()`. `KioskController` matches the employee by `employee_no` + name, then calls the same service with `source = WebKiosk`. The kiosk has no auth; the matching pair is the "proof you're you" until face recognition lands.
- **Public biometric webhook** is exposed at `POST /webhooks/biometric`, signed with HMAC-SHA256 over `{X-Biometric-Timestamp}.{body}` keyed by the device's `shared_secret`. The `VerifyWebhookSignature` middleware (`webhook.signature:biometric`) verifies the signature, the device code, and a 5-minute timestamp window to prevent replay attacks. The `BiometricIngestionService` then translates the vendor-neutral payload (`device_code` + `events[]` with `employee_no`, `direction`, `event_at`, optional `score`/`geo_lat`/`geo_lng`) into one `AttendanceRecord` per event, returning `{accepted, skipped, errors[]}` to the device for its own retry logic. The device's `last_seen_at` is bumped on every accepted batch.
- **Public v1 API** (`GET /api/v1/attendance/summaries`) gives integrators a read-only paginated view, scoped by `api.scope:attendance:read`. Filters: `employee_id`, `from`, `to`, `status`, `per_page` (max 500). This is what the IPPD/GIFMIS integration bus reads from, and what an external BI tool would hit.
- **Analytics module** picks up `AttendanceCorrectionRequested` and `AttendanceCorrectionDecided` events via `RecordAnalyticsEvent`, writing `attendance.correction.requested` and `attendance.correction.decided` rows so the manager-side activity stream and the executive dashboards can count them.
- **Identity Verification (Ch 25)** is the future tie-in: biometric devices already carry an HMAC secret and a confidence score on every event; the next phase plugs that into the Ghana Card biometric match so a clock-in can be one-and-the-same with an identity check.
- **Notifications (Ch 14)** is wired for correction lifecycle events through the same dispatchable but does not currently mail anyone on a correction request — the kanban itself is the inbox. Mail is roadmapped for the SLA-breach case (a request older than 3 days).

## Standards touchpoints

- **Ghana Labour Act, 2003 (Act 651) §33-39 — Hours of work, overtime, and rest periods.** The default schedule (Mon-Fri, 08:00-17:00, 8h full day, 1h unpaid lunch, ≥ 15 min grace) honours the §33 maximum 40-hour week. The `OvertimeCalculator` implements §35's three-tier rule (weekday OT at 1.5×, weekend/holiday at 2×, deep-OT beyond 12 hours at 2×). The calculator file (`app/Services/Attendance/OvertimeCalculator.php`) explicitly cites §35 in its docblock. The institute's public-holiday register feeds `PublicHoliday::isHoliday()` so every gazetted holiday automatically pays at the §36 rate. See Chapter 44.
- **ISO 30414:2018 §4.5 (workforce productivity / absenteeism).** `AttendanceService::aggregatePeriod()` returns an `attendance_ratio` (`days_worked / working_days` excluding excused days) on every render of the My Attendance page; the org-wide register aggregates the same number across the workforce. The Standards Benchmark in Chapter 44 lists this as the institute's ISO 30414 absenteeism touchpoint.
- **ISO 30414:2018 §4.7 (workforce availability)** — the live "on shift now" coverage count on the Shift Schedules page surfaces real-time availability against the rota, the metric the standard recommends for operational-readiness disclosures.
- **Data Protection Act, 2012 (Act 843) §17 — Lawful basis for processing personal data.** Biometric punches are processed on the employment-contract basis; the device-side `shared_secret` is encrypted at rest and never exposed via the API. The geofence pre-check rejects out-of-area punches before they enter the audit trail, which constrains the personal-data footprint to what the employer needs (the §18 data-minimisation principle). DPA §40 (right to erasure) requests are handled by the redaction job described in Chapter 27 — soft-delete is the operational default for attendance corrections, not for raw records.
- **Data Protection Act §20 — Special personal data (biometric).** The biometric ingestion path stores only the `biometric_score` (a confidence string from the device) and the `raw_payload` JSON; the underlying fingerprint / face / iris template never leaves the device. This is the consent-required category under §20; the Phase 1 work (Chapter 25) wires the consent flow per employee. See Chapter 27 for the full Special-Category mapping.
- **National Identification Authority Act, 2006 (Act 707) — Ghana Card integration** is the planned downstream join: once a device's confidence score can be matched against a NIA biometric record, a single clock-in becomes both an attendance event and an identity verification.

## What's planned next

Phase 2 of the government-grade roadmap (Chapter 46) lands four things directly on this module: (1) **Mobile app with offline buffer** — a thin React Native client around `/attendance/clock` that caches events in IndexedDB when offline and replays them on reconnection, so field staff on patchy networks don't lose punches; (2) **Per-shift overtime policies** — the `OvertimeCalculator` is currently hard-coded to the Labour Act §35 default rates, but for shifts (e.g. NIGHT_WATCH) that need a richer rule (night-premium percentages, double-time after midnight, etc.) the policy will move onto the `shifts` table as a configurable JSON column; (3) **Roster-style shift planning** — today's `shift_assignments` table is one-shift-per-employee-per-window; the planned roster module sits above it as a weekly view that bulk-assigns shifts in a single drag-and-drop pass, with conflict detection against approved leave; (4) **Manager-side bulk approve** for the corrections kanban so end-of-pay-period clean-up doesn't require clicking each card individually. The screen-level gaps — UI for editing and archiving shifts, UI for editing assignments, and a face-recognition clock path on the kiosk — are tracked in the same wave; the routes and services are already in place.
