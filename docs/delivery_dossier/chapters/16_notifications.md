# Chapter 16 — Notifications

> *In one paragraph.* Notifications is the in-app nervous system of CIHRMS — every other module fires events here when something happens that a person needs to know about. A leave request approved (Ch 4), a ticket assigned (Ch 11), a complaint closed (Ch 12), a document awaiting your signature (Ch 13), a payroll run marked paid (Ch 19), a Ghana Card expiring (Ch 25), a tamper-evident audit chain broken (Ch 26) — they all funnel through the same `notifications` table, surface through the same bell in the top bar, and write through the same channel-preference layer so each person decides where they want to be reached. The chapter that follows covers the bell dropdown, the full notifications page, the per-user channel preferences screen, the three-pack sound subsystem that gives the bell its voice, and every event source that feeds the queue.

## Where to find it

- **Sidebar location:** Notifications is **not** a sidebar item. It lives in the top bar of the authenticated layout, between the **Sound effects toggle** and the **theme toggle** — a single bell icon with a red counter badge. Clicking the bell opens a 360-pixel dropdown of the ten most recent unread notifications; the dropdown footer links to the full page at `/notifications`. The channel-preferences screen is reached from the user-account dropdown (top-right avatar) under **Notification Channels**, and from the **Channels** button on the full page header.
- **Roles that see it:**
    - **Every authenticated user**, regardless of role, sees their own bell, dropdown, and full-page index. The notifications shown are scoped through Laravel's `Notifiable` trait to the signed-in `User` row — you only ever see notifications addressed to you.
    - **super_admin** and **ceo** see no special view of other people's notifications; the audit log (Ch 26) is the surface for cross-user activity, not this one.
    - **dept_head / manager / hr_admin** receive role-specific notifications (a leave-approval reminder, an incident assigned, a document awaiting their action), but the screens themselves are the same as every other user's.
- **Related modules:** every other workflow chapter is a *producer* of notifications. The top five sources by volume in the MVP are **Leave (Ch 4)** — request submitted, approved, rejected, withdrawn, plus the 3-day approval reminder; **Tickets (Ch 11)** — assigned, resolved, closed; **Documents (Ch 13)** — awaiting your action, completed, rejected; **Employees (Ch 3)** — new-hire welcome mail; **Identity (Ch 25)** — Ghana Card re-verification reminders. **Incidents (Ch 27)**, **Audit (Ch 26)**, and **Public DPA (Ch 26)** also write here, with more producers landing in later phases (Payroll, Complaints, Performance, Loans). The full inventory is in *How it talks to other modules* below.

> *Notes:* The bell's data is pumped in by `HandleInertiaRequests` as two **deferred** Inertia props — `notifications` (the last ten unread, mapped to `{ id, type, message, kind, time }`) and `notificationCount` (an integer). Deferred means the layout paints first; the bell hydrates a moment later with a follow-up Inertia request, so a slow notifications query never blocks navigation on any other page. The cost is one extra round-trip per page load — a fair trade for sub-second TTI everywhere else.

## The data behind it

CIHRMS stores notifications in a single canonical Laravel table:

- **`notifications`** — a UUID primary key; `type` (the fully-qualified notification class name, e.g. `App\Notifications\LeaveStatusChanged`); a polymorphic owner pair (`notifiable_type`, `notifiable_id`) that points back to the `User` who should see this row; `data` (a JSON column — `text` in the migration, cast to array on read); a nullable `read_at` timestamp; and the standard `created_at` / `updated_at` pair. The schema is the stock Laravel `notifications` migration (`2026_05_14_082201_create_notifications_table.php`) — no per-row delivery channel column, no priority, no expiry, no soft delete. A row exists if and only if Laravel's `database` channel was in the notification's `via()` array.

Per-user channel preferences and consent are stored on the `users` table itself, added in `2026_05_18_000001_add_messaging_consent_columns_to_users.php`:

- **`notification_channels`** — a nullable JSON column whose key set is `{email, in_app, whatsapp, slack, teams}` mapped to booleans. When the column is null the resolver falls back to the per-channel defaults (`email: true, in_app: true, whatsapp: false, slack: false, teams: false`).
- **`whatsapp_phone`** — a `string(20)` indexed for lookup. Nullable. Stored in international format with optional leading `+`.
- **`whatsapp_consent_at`** — a nullable timestamp stamped only when the user explicitly ticks the consent box *and* a phone is present. Cleared when either condition fails. This is the legal record of opt-in; the toggle alone is not.
- **`slack_user_id`** — a nullable indexed `string(30)`. Without it, Slack notifications fall back to the configured HR channel.

There is no separate **`notification_channel_preferences`** table, no per-event opt-out (every event uses the channel matrix as a whole), and no per-channel TTL/quiet hours setting. Those are all roadmap items.

What every reader of the screen needs to keep in mind:

- **In-app delivery is the `database` channel.** Laravel writes a `notifications` row whenever the notification's `via()` array contains `'database'`. The `in_app` toggle on the channels screen is **not yet** consulted by every producer — the dispatcher honours it for the WhatsApp/Slack/Teams paths today, but the database write happens unconditionally for every notification that lists `database` in `via()`. Phase 1 closes this gap by routing every producer through a single preference-aware dispatcher.
- **Mail delivery is the `mail` channel** and uses the SMTP/SES config in `mail.php`. Notifications with `mail` in `via()` build a `MailMessage` from the notification's `toMail()` method.
- **Broadcast / realtime push is not wired.** No notification declares the `broadcast` channel in `via()`, no Laravel Echo client is registered in `resources/js/app.js`, and no Pusher/Reverb dependency is in `composer.json` or `package.json`. The bell is **polled** through the deferred Inertia prop — every page navigation re-fetches the last ten unread plus the count. Phase 1 will add Reverb/Echo for sub-second delivery.
- **Per-row state is two-valued.** A row is unread (`read_at` is null) or read (`read_at` is a timestamp). There is no "archived" or "dismissed" state and the only mutation in the controller is `unreadNotifications()->update(['read_at' => now()])` — *all at once*. Per-row mark-read and per-row delete will land with the row-action UI in Phase 1.
- **Data payloads are conventions, not contracts.** Each notification's `toArray()` returns a free-shape associative array. A few keys are honoured by the UI (`message` is the primary line shown in the bell and on the index; `kind` is read by the bell to pick a sound; `route` is read in some places — `incident.assigned` sets `route: 'incidents.show'` — but the index page does not currently navigate on row click). Every producer should include at least `message` and a `*_id` pointer back to the source entity; most include both.
- **Soft delete?** Not on this table. A notification row sticks around forever (subject to the operator's pruning policy in Phase 1). Right-to-erasure (Ch 26) handles deletion as part of a DPA workflow, not as a user-facing action.

## How it talks to other modules

Every other CIHRMS module is a *producer* of notifications — the table below is the live inventory in the MVP plus the integration channel each producer uses today.

- **Leave (Ch 4)** — `LeaveStatusChanged` fires from the `SendNotifications` listener on every `LeaveStatusUpdated` event (approved, rejected, withdrawn). `via()` is `['database', 'mail']`. The companion `LeaveApprovalReminder` is fired by the 3-day-stale cron over `['mail', 'database']`. The separate **`NotifyManagerOfLeaveRequest`** listener runs against `LeaveRequested` and DMs the manager through whatever Slack/Teams/WhatsApp channels they've configured via the dispatcher — that path does *not* write a `notifications` row (it's a one-shot external message).
- **Employees (Ch 3)** — `NewEmployeeWelcome` fires on `EmployeeCreated`. `via()` is `['mail']` only — the welcome email goes out but no in-app row is written, because the new hire's bell will already be empty on first sign-in.
- **Tickets (Ch 11)** — `TicketAssigned` (`via: ['database']`) on `TicketService::assign`, and `TicketResolved` (`via: ['database']`, with `kind: 'ticket.completed'`) when the service moves a ticket to Resolved or Closed. The `kind` field is what the bell reads to pick the celebratory `task.completed` sound.
- **Documents (Ch 13)** — `DocumentAwaitingAction` (`via: ['database', 'mail']`) fires from `DocumentRoutingService::route` when a route step is created and again when a step advances; `DocumentCompletedNotice` (`via: ['database', 'mail']`) fires to the document owner on rejection or completion.
- **Identity / Ghana Card (Ch 25)** — `IdentityExpiringReminder` (`via: ['mail', 'database']`) fires from the `identity:expiring` artisan command — Ghana Card re-verification 30/15/7 days before the 12-month window closes.
- **Audit (Ch 26)** — `AuditChainBroken` (`via: ['mail', 'database']`) fires from the `audit:verify-chain` cron when the SHA-256 hash chain over `audit_logs` fails end-to-end verification. Routed to super_admins; Slack and SMS are deliberately *not* used (the audit log itself is the primary record of who saw the alert).
- **DPA / Public Data Subject (Ch 26)** — `DpaVerificationLink` (`via: ['mail']`) fires from `PublicDpaController::store` to the public submitter's email via `Notification::route('mail', $email)` — an on-demand notifiable because public subjects don't have a User row.
- **Incidents (Ch 27)** — five listeners in `app/Listeners/Incident/` write directly into the `notifications` table with `kind: 'incident.assigned'`, etc. They construct `Notification::create(...)` directly rather than going through `notify()` — a shortcut that bypasses the `via()` array entirely.
- **Payroll (Ch 19)** *(roadmap)* — `PaymentMarkedPaid` exists in the UI type palette but the producer hasn't shipped; the bell already knows how to render it.
- **Complaints (Ch 12), Performance (Ch 6), Learning (Ch 7), Loans (Ch 21), Disbursements (Ch 22)** *(roadmap)* — none of these write notifications today; Phase 1 ports their event hooks onto the same `SendNotifications` listener pattern.

The fan-out today is mostly direct from each service: a service constructs the notification, calls `$user->notify(...)` (or `Notification::send(...)` for fan-out to multiple recipients), and Laravel handles the rest. The `SendNotifications` listener registered in `AppServiceProvider::boot()` is the canonical event-driven path — `EmployeeCreated` and `LeaveStatusUpdated` route through it; the others call `notify()` from inside the originating service. Phase 1 unifies this by moving every producer onto the listener pattern with a single `NotifyService` that consults the user's channel preferences before each `via()` is constructed.

## Standards touchpoints

- **WCAG 2.1 AA §1.4.3 (Contrast)** — every notification colour pair (red badge on cobalt bell, type-tag pill on tinted background, unread dot on the row stripe) is tested against the 4.5:1 ratio for normal text and 3:1 for the icon target. The amber driver-not-connected pill is paired against a near-white surface for 4.5:1 contrast.
- **WCAG 2.1 AA §2.2.2 (Pause, Stop, Hide)** — the persistent bell pulse loops indefinitely while unread > 0; users can stop it by clearing notifications. The badge heartbeat is a 2.2 s loop. Reduced-motion preference disables both animations and the ring entirely (`@media (prefers-reduced-motion: reduce)` in `NotificationBell.vue`'s `<style>` block).
- **WCAG 2.1 AA §2.3.1 (Three Flashes or Below Threshold)** — no part of the bell or list flashes at or above the seizure-inducing threshold; the brightest pulse step is a single colour ramp at sub-second intervals (3 s for pulse, 2.2 s for heartbeat).
- **WCAG 2.1 AA §4.1.3 (Status Messages)** — the bell's red counter badge is a status message (it changes without focus). It is paired with the `notifications` glyph's filled state so the cue is not colour-only.
- **WCAG 2.1 AA §1.4.13 (Content on Hover or Focus)** — the bell dropdown is opened by click (not hover) and stays open until the user clicks outside or selects "Mark all read"; no transient hover panels.
- **Data Protection Act, 2012 (Act 843) §17 (lawful basis)** — in-app and email notifications about the user's own employment data run on the employment-contract basis. WhatsApp, Slack, and Teams are *separate* processing operations under §18 (consent) — each has its own toggle, and WhatsApp additionally records an explicit consent timestamp (`whatsapp_consent_at`) because Meta requires it. Revoking consent clears the timestamp and force-disables the channel.
- **Data Protection Act §40 (right to erasure)** — notification rows are pruned by the DPA redaction job (Ch 26) when the underlying subject is erased; the rows are not user-deletable from the UI yet.
- **Mental Health Act, 2012 (Act 846) / minor-PII flag** — none of the notification payloads in the MVP carry health information; the type palette deliberately has no "medical" or "clinical" category, so a future leave-with-medical-cert notification will need a sensitive-payload channel rule before it ships (tracked in Phase 1).

## What's planned next

Phase 1 of the government-grade roadmap lands four things directly on this screen: (1) **Realtime push** — Laravel Reverb is wired as the WebSocket broker, every notification class adds `broadcast` to its `via()` array, and `NotificationBell.vue` subscribes to a private `App.Models.User.{id}` channel via Echo so the bell rings *sub-second* instead of polling on every navigation; (2) **Row-level actions** — `PATCH /notifications/{id}/read`, `DELETE /notifications/{id}`, and a per-row click handler that uses the `route` and entity-id fields in the JSON payload to navigate to the source module (the data is already there — only the UI is missing); (3) **Preference-aware dispatcher** — a `NotifyService` that consults `notification_channels` *before* building the `via()` array, so the `in_app` toggle finally hard-mutes the database write; per-event-type opt-outs land in the same release; (4) **Sound preferences persisted on the server** — the per-device localStorage choice is mirrored on the `users` row so a user who picks "gamified" on their laptop hears the same pack on their phone. The roadmap also adds quiet hours, a digest-mode option ("daily 5pm summary instead of per-event"), and SMS as a first-class channel through MTN/Vodafone Ghana for sites that don't run WhatsApp.
