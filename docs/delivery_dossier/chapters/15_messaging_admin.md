# Chapter 15 — Messaging Administration

> *In one paragraph.* Messaging Administration is CIHRMS's out-of-app reach into staff phones — SMS for everyone and USSD for the ones whose phones can't run a browser. It is a deliberately narrow module: a one-off SMS sender for HR, a 4-digit-PIN issuer that gates the `*920*HR#` self-service dial code, an outbound log of every message the system has ever sent, and an inbound stream of every short-code reply. Chat (Chapter 14) is staff talking *inside* CIHRMS; Messaging is CIHRMS talking *outside* it — to a handset on the MTN, Vodafone, or AirtelTigo network. There is no campaign builder, no audience picker, no scheduler, and no template designer in the MVP. The provider stack is pluggable (Hubtel by default in Ghana, Twilio as the international fallback, a no-op log driver for staging), the consent model is opt-in for WhatsApp and assumed-implied for SMS to your own employees, and every outbound message is persisted with provider, segment count, cost, and a delivery receipt for the auditor.

## Where to find it

- **Sidebar location:** **Administration** group → **Messaging** (top-level item, `sms` icon). The module is visible only to roles holding `messaging.view`; the link uses the `sms` Material Symbol so it sits visually next to "Chat" in the sidebar even though it is a different sibling concept. There is no entry in the app-switcher grid in the MVP — Messaging is staff-comms plumbing, not a daily-use module for line staff.
- **Roles that see it:**
    - **super_admin** and **ceo** — every screen, every action. Both pass the `messaging.view`, `messaging.send`, and `messaging.manage` checks via the wildcard / mirrored-permission paths in `User::hasPermission`.
    - **hr_admin** — the primary operator. Holds all three Messaging permissions in the seed; runs the outbound log, sends one-off SMS, issues USSD PINs.
    - **manager**, **dept_head**, **employee**, **finance_officer**, **it_support**, **auditor** — *not* seeded with any `messaging.*` permission in `RolePermissionSeeder`. They never see the sidebar item and a direct URL hit on `/admin/messaging` returns 403 from the `permission:messaging.view` middleware.
- **Related modules:** Chat (Ch 14) — the in-app sibling; the two share neither table nor model so an HR broadcast is not a chat thread; Employees (Ch 3) — the audience: every outbound SMS resolves a phone via the `employees.phone` or `users.whatsapp_phone` column; Notifications (Ch 16) — the *channel router* that decides email vs WhatsApp vs Slack/Teams for system events, layered on top of the SMS dispatcher described here; Bank Changes (Ch 22, Disbursements) — the busiest non-PIN caller of the SMS dispatcher, sending the 6-digit confirmation code that the subject types back into USSD option 5; Whistleblower (Ch 27) — the public anonymous channel also accepts a USSD path through the same shortcode; DPA (Ch 26) — consent storage and erasure for the messaging audit trail; Standards (Ch 44) — DPA Act 843 §20 direct-marketing language, NCA SMS-sender registration, and ISO 27001 A.13 communications-security coverage.

| Field | Validation | Who can set it | Notes |
|---|---|---|---|
| **Recipient phone** | required, string, max 32 chars | `messaging.send` | No format check on the controller — the provider normaliser handles `0XX`, `233XX`, `+233XX`, or any other variant. | **Cancel** | Closes the panel without firing | Anyone | Standard escape. |

> *Notes:* The send path is synchronous — `SmsDispatcher::send()` POSTs to Hubtel (or Twilio) right inside the controller before redirecting. A slow gateway will block the operator's request; a dead gateway times out at 10 seconds and the row lands as `failed` with the timeout reason captured. Phase 1's roadmap moves dispatch into a queue worker so the UI returns instantly and the gateway is allowed to be slow. The "Reply STOP to opt out" string is **not** appended automatically by the dispatcher — it is the operator's responsibility to include the opt-out phrasing for any message that could be classed as marketing under DPA Act 843 §20. The dedicated USSD PIN sender (next section) prints it explicitly; the operator-driven Send SMS does not. This is a documented gap that needs to be closed in Phase 1 (see "What's planned next").

> *Notes:* The PIN-issue flow is the only place in the MVP that uses the `2fa:fresh` middleware outside of disbursements and DPA fulfilment — issuing a PIN is materially equivalent to issuing a credential, and it gets the same treatment. The bcrypt hash means that even a database leak does not expose the four digits. Failed USSD attempts increment `failed_attempts`; five misses (configurable via `messaging.ussd.pin_max_attempts`) lock the account for 15 minutes (`messaging.ussd.pin_lock_minutes`) and a sixth miss rotates the lock — there is no permanent lockout. The PIN SMS includes `"Do not share. Reply STOP to opt out."` — the only outbound point in the system that prints the DPA opt-out language by default.

## The data behind it

Messaging stores four tables, all created by the WS18 migration:

- **`sms_messages`** — every outbound message, ever. Columns include the `to_phone`, the optional `from_sender` (an alphanumeric sender ID like `CIHRMS`, registered with the NCA), the message `body`, the `provider` slug (`hubtel` / `twilio` / `log`), the `status` (`queued` → `sent` → `delivered`, or `failed`, or `expired`), the `provider_message_id` returned by the carrier (used to match delivery receipts back), the `segments` count, the `cost` to four decimal places, an optional `failure_reason`, a polymorphic `context_type` + `context_id` pair (so a bank-change SMS knows it belongs to a `PendingBankChange`, a payroll notification knows it belongs to a `PayrollRun`), the `triggered_by` user id, and `sent_at` + `delivered_at` timestamps. Indexes on `(to_phone, status)` and `(context_type, context_id)` keep the log fast.
- **`inbound_sms`** — every message that arrived at the short code from a staff handset. Columns include the `from_phone`, the `to_shortcode` it came in on, the `body`, the carrier `provider`, the carrier's `provider_message_id`, a `parsed_intent` slug (`PAYSLIP` · `LEAVE` · `CLOCK_IN` · `CLOCK_OUT` · `TRACK` · `OPTOUT`, or null if the parser didn't match), an optional `parsed_args` JSON blob for future structured replies, a `received_at` timestamp, an optional `processed_at` (when a future worker has acted on it), and a `reply_sent` body field. The intent parser is deliberately simple — first-token, case-insensitive keyword match — and it does *not* dispatch downstream actions in the MVP. It tags the row and waits for an HR operator (or, in Phase 1, an automated worker) to act on it.
- **`ussd_sessions`** — the running state of every active dial of `*920*HR#`. One row per provider `sessionId`, with `phone`, `shortcode`, `state` (a 13-value enum from `welcome` to `whistleblower_status`), an optional `employee_id` once the caller has authenticated, a free-form `context` JSON for breadcrumbs, the `last_input` and `last_response` for support diagnostics, and an `expires_at` set 180 seconds in the future on every step (USSD sessions are short-lived by definition). The state machine is provider-agnostic; the controller normalises Hubtel, mNotify, and AfricasTalking payloads into the same `{sessionId, msisdn, shortcode, text}` tuple before handing them off.
- **`staff_phone_pins`** — one row per employee that has been onboarded to USSD. Columns: `employee_id` (unique — only one PIN per employee), `phone`, `pin_hash` (bcrypt), optional `pin_expires_at`, `last_used_at` for "when did this employee last self-serve over USSD", a `failed_attempts` counter, and a `locked_until` timestamp. The hash is `$hidden` from JSON serialisation so it never leaks into an Inertia payload.

In addition, two columns added to **`users`** by the Wave 12 migration carry the per-user messaging preferences:

- **`notification_channels`** (JSON) — a map of channel slug (`email` · `in_app` · `whatsapp` · `slack` · `teams`) to a boolean. Email and in-app default to true; the three out-of-app channels default to false (opt-in only). Read by `User::notificationPreferences()`.
- **`whatsapp_phone`** (string, indexed) and **`whatsapp_consent_at`** (timestamp) — the WhatsApp business-channel address and the consent capture timestamp. `User::hasWhatsappConsent()` is the canonical gate used by `MessagingDispatcher::sendWhatsApp` — a user with consent revoked never has a template message dispatched to them.

What every reader of the screen needs to keep in mind:

- **SMS, not in-app chat.** A message in the outbound log is bytes on a cellular network, not a row in the chat thread table. The two systems do not share a primary key, a service, or a Vue page. A reply to an SMS lands in `inbound_sms` and is invisible to Chat; a message in Chat never produces a row in `sms_messages`.
- **`log` is not "fake-send".** The `log` driver writes the message to the configured log channel and returns a synthesised message id — a row still lands in `sms_messages` with `provider=log`, status `sent`, cost 0. This is the staging-environment default; switching to Hubtel in production is a single `SMS_DRIVER` env var change. The log driver is also what keeps the test suite from hitting a real gateway.
- **WhatsApp is not on this page.** The WhatsApp Cloud driver exists (see `MessagingDispatcher` and `WhatsAppWebhookController`) but it is wired into the *notifications* layer — system events fan out via the user's preference map — not into the operator-facing "Send SMS" flow. There is no UI on the Messaging dashboard that sends a WhatsApp template from a slide-over. That is on the Phase 1 list.
- **There are no templates and no campaigns in the MVP.** Every outbound is either a one-off typed body or a system-generated string (`"CIHRMS PIN for {$employee->employee_no}: {$pinPlain}…"`, `"Your bank-change code is {$code}…"`). There is no `sms_templates` table, no audience picker, no scheduled send, no A/B variant, no segmentation. This is a documented gap, not an oversight — the MVP scope was deliberately narrow.
- **The "STOP" opt-out is parsed but not enforced.** The webhook controller recognises `STOP` and `UNSUB` as the `OPTOUT` intent and writes them to `inbound_sms.parsed_intent`. Nothing in the outbound path then checks that intent before sending the next message. An operator with `messaging.send` can send an SMS to any number, opt-out or not. This is a real DPA gap — see "Honest gaps" and Phase 1.

## How it talks to other modules

- **System triggers that send SMS via `SmsDispatcher`** in the MVP: (1) the one-off "Send SMS" panel on this page; (2) the "Issue USSD PIN" panel on this page; (3) `BankChangeRequestService::initiate()` — sends the 6-digit code to the subject's registered employee phone when HR (or the subject themselves) initiates a bank-account change. That is the complete trigger surface. There are no payroll-payslip SMS, no leave-approved SMS, no shift-reminder SMS in the MVP — those are notifications routed through email and in-app, not the SMS gateway.
- **USSD self-service paths** that the issued PIN unlocks: latest payslip readout, current-year leave balance, clock-in / clock-out, and bank-change confirmation. The whistleblower case-tracking path is reachable from the welcome screen without a PIN (option 0). See the `UssdSessionHandler` state machine in `app/Services/Messaging/Ussd/`.
- **Inbound delivery receipts** flow through `webhooks/sms` (signature-verified with `webhook.signature:hubtel_sms`). The provider POSTs `{type: "delivery_receipt", messageId}` and the `SmsDispatcher::markDelivered()` method flips the matching `sms_messages` row from Sent to Delivered with a `delivered_at` timestamp.
- **Inbound short-code messages** flow through the same `webhooks/sms` endpoint with `type: "inbound_message"`. The body is persisted, the first token is keyword-parsed into an intent, and the row sits in `inbound_sms` waiting to be acted on. Phase 1 wires a worker that auto-handles `PAYSLIP`, `LEAVE`, and `OPTOUT` without HR intervention.
- **USSD callbacks** flow through `webhooks/ussd` (signature-verified with `webhook.signature:hubtel_ussd`). Every key press is one round trip — the controller reads the cumulative `text`, slices off the last segment, dispatches it to the state machine, and returns either `CON <body>` (keep the session open) or `END <body>` (terminate). USSD sessions live in `ussd_sessions` for 180 seconds.
- **WhatsApp Cloud webhooks** at `webhooks/whatsapp` (GET for Meta's `hub.challenge` handshake, POST for messages). Inbound WhatsApp keywords (`LEAVE 3 SICK`, `URGENT printer broken`) are parsed into draft `LeaveRequest` and `Ticket` rows directly — this is the live cross-channel ingestion path, and it is documented here for symmetry with the SMS short code even though the operator never sees it on the Messaging dashboard.
- **Notification channel preferences** (Chapter 16) live on the User row (`notification_channels` JSON, `whatsapp_phone`, `whatsapp_consent_at`, `slack_user_id`). When a domain event needs to reach an employee, the `MessagingDispatcher` (note: a *different* class to the `SmsDispatcher` documented above; they share a name family but not a contract) fans the body out to the channels the user has opted into, respecting the WhatsApp consent gate and the Slack/Teams driver availability.
- **Permission seeding** is in `RolePermissionSeeder`. Only super_admin, ceo, and hr_admin hold `messaging.view`, `messaging.send`, and `messaging.manage` in the MVP. Any other role hitting the URL gets a 403 from the middleware before the controller runs.

## Standards touchpoints

- **Data Protection Act, 2012 (Act 843) §20 — direct marketing.** Section 20 requires a clear opt-out mechanism on any direct-marketing communication. The PIN-issue SMS does it correctly (`"Reply STOP to opt out."`); the operator-driven Send SMS does not, today, append the phrase automatically. The keyword `STOP` is parsed into the `OPTOUT` intent but not enforced on subsequent sends. See Chapter 26.
- **Data Protection Act §17 — lawful basis.** Employment-contract is the lawful basis for SMS / USSD reach to staff; consent (§17(1)(a)) is the lawful basis for WhatsApp template messages (Meta's commercial policy is stricter than the DPA's floor). The `whatsapp_consent_at` timestamp is the audit-grade evidence of that consent.
- **Data Protection Act §40 — right to erasure.** When a data subject requests erasure, the existing redaction job covers user PII; the `sms_messages` and `inbound_sms` audit rows are retained for the regulatory window and then redacted in bulk. Detail in Chapter 26.
- **National Communications Authority (NCA) — Alphanumeric sender ID registration.** Ghana's NCA requires the alphanumeric sender ID (`CIHRMS`) to be pre-registered with the carrier. The system carries no validation that the ID is registered — it is an operational checklist item before going live with Hubtel. See Chapter 44.
- **NCA — Inbound short-code provisioning.** The `*920*HR#` USSD code is shared / leased per institute and is provisioned out-of-band by the carrier. The `messaging.ussd.shortcode` env var simply mirrors whatever the institute has registered.
- **WhatsApp Business Policy — 24-hour customer-care window.** The `MessagingDispatcher::sendWhatsApp` path enforces Meta's rule that free-form text is allowed only inside the 24-hour reply window; outside it the caller **must** pass a pre-approved template name (the HSM template id). The interface signature on `MessagingProvider::sendTemplate` makes this explicit. See Chapter 44.
- **ISO/IEC 27001 Annex A.13 — Communications security.** Webhook signature verification (`webhook.signature:hubtel_sms`, `webhook.signature:hubtel_ussd`, `webhook.signature:whatsapp`) gates every inbound; bcrypt-hashed PINs satisfy A.9 credential storage; the per-channel kill-switch on each driver's `isConfigured()` returns false if env vars are missing, which means a half-provisioned environment cannot send mistaken messages. See Chapter 44.
- **PCI / payment-channel adjacency.** The bank-change confirmation flow described above means the SMS gateway sits in the trust path between HR and an employee's bank account. The system mitigates this by sending only a one-time code (no account numbers), masking the new account to its last four digits on the USSD confirmation screen, and requiring the subject to explicitly approve via option 5. The code itself is single-use and short-lived. See Chapter 22 and Chapter 44.

## What's planned next

Phase 1 of the government-grade roadmap (8–10 weeks, see the gap analysis) lands six things directly on this screen:

1. **Templates.** A first-class `sms_templates` table with named templates (`PAYSLIP_READY`, `LEAVE_APPROVED`, `SHIFT_REMINDER`, `BANK_CHANGE_CODE`), Mustache-style variable substitution (`{{employee_name}}`, `{{period_label}}`, `{{net_amount}}`), and a template-picker on the Send SMS panel. WhatsApp templates piggyback on this — same row, additional `whatsapp_template_name` and `whatsapp_language` columns.
2. **Campaigns and audience selection.** A "New campaign" flow lets HR pick a template, an audience (all employees · by department · by role · by an arbitrary employee-list CSV upload · by a leave-status filter · by a payroll-payment-status filter), and a send-time (now / scheduled). A `messaging_campaigns` table tracks the run; per-recipient rows in `sms_messages` link back via `context_type = 'campaign'`.
3. **Queue-backed dispatch.** `SmsDispatcher::send()` moves behind a `DispatchSms` job so the UI returns instantly and the gateway is allowed to be slow. Failed sends retry with exponential backoff up to five times; the row stays `queued` between attempts and only flips to `failed` when retries are exhausted.
4. **DPA-grade opt-out enforcement.** A new `sms_optouts` table records every `OPTOUT` intent observed on inbound. Before any outbound the dispatcher checks the recipient against this table; an opt-out blocks the send (with `failure_reason = 'opted_out'`) unless the message is a transactional class (PIN-issue, bank-change confirmation) that survives opt-out per DPA Act 843 §20(2). The operator-driven Send SMS panel surfaces an inline warning when the typed number is opted out and refuses to submit unless the operator ticks an explicit "transactional override" box.
5. **WhatsApp send from the dashboard.** The MVP's WhatsApp wiring is inbound-and-system-notifications only. Phase 1 surfaces a "Send WhatsApp" panel that picks a pre-approved template + parameters, audiences against `users.whatsapp_phone WHERE whatsapp_consent_at IS NOT NULL`, and logs each send into `sms_messages` alongside SMS for a single unified outbound log.
6. **Delivery analytics.** A dedicated Reports & Analytics tile (Chapter 31) chart-builds delivery rate by hour-of-day, by carrier (MTN / Vodafone / AirtelTigo derived from the MSISDN prefix), and by template — answering "which template is the most read", "which carrier is silently dropping us", and "when in the day is the gateway most reliable". The data already exists on `sms_messages`; only the chart wiring is outstanding.

Phase 2 adds bulk-import audience uploads (CSV/XLSX), inbound auto-handling (a worker that responds to `PAYSLIP` with a real payslip summary instead of waiting for HR), and a per-message bilingual rendering (English + Twi) for any template that has a translation row in the new `sms_templates` table. The MVP's narrow scope was deliberate; the foundations described in this chapter — provider abstraction, `sms_messages` as the canonical log, consent columns on the user — are the pieces every Phase 1 / Phase 2 item builds on top of.
