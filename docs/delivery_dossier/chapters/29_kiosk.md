# Chapter 29 — Kiosk (Shared-Device Attendance Terminal)

> *In one paragraph.* The Kiosk is the single-purpose, no-login attendance terminal that turns any browser — a wall-mounted tablet at reception, a low-spec laptop at a guard post, a refurbished desktop in the factory loading bay — into a clock-in / clock-out station that any employee can walk up to and use without having to remember a password. There is no session, no sidebar, no app shell; the page identifies the employee with two pieces of public information (their employee number and a fragment of their name), confirms the match face-to-screen, records the punch through the same `AttendanceService::record()` that every other attendance source uses (Chapter 5), and reverts to "Welcome" in six seconds so the next person can step up. The whole module is one Vue page, one controller, two FormRequests, and five routes — it is deliberately tiny because all the heavy lifting (lateness, overtime, daily summary, payroll integration, audit log) is done by Attendance. The Kiosk is the front door; the ledger lives next door.

## Where to find it

- **Public URL:** **`/kiosk`** — the only screen in the module. There is no sidebar link from the rest of CIHRMS to this page on purpose: the kiosk URL is opened once, by IT, on the dedicated device, and then the device stays on that page in fullscreen for its working life. An admin browsing the main app does not "navigate to" the kiosk; they walk over to the device.
- **Roles that see it:** **none — and everyone.** The kiosk routes are explicitly outside the auth middleware. Anyone who can reach the URL — typically only the device itself, on the office LAN — can use it. The "permission" check is the identity step described below (employee_no + a fragment of the user's name); there is no role gate because the device has no logged-in user to gate against.
- **Related modules:** **Attendance (Ch 5)** — every kiosk punch lands in `attendance_records` with `source = web_kiosk` and is reconciled into `attendance_summaries` by the same recompute that runs for biometric devices, self-service GPS clocks, and HR manual entries; **Employees (Ch 3)** — the identity match runs against `employees.employee_no` joined to `users.name`, no extra table; **Identity Verification (Ch 25)** — the face-recognition route exists as a 501 stub waiting on the same Ghana-Card biometric stack that Chapter 25 is building; **Standards Benchmark (Ch 44)** — the kiosk is the WCAG 2.1 AA touchpoint for shared-device accessibility and a Cybersecurity Act 2020 (Act 1038) touchpoint for the device-hardening section.

The kiosk is one page with three sequential stages — Identify, Confirm, Success — that flip in place rather than navigate. The browser URL never changes once `/kiosk` is open.

## Every button, every action

### Identify stage (`/kiosk`, default view)

| Button / field | What it does | Who can use it | Why it exists |
|---|---|---|---|
| **Employee ID input** | Free text, max 20 chars on the client and max 50 on the server. The on-screen keyboard types into whichever field is "active"; tapping the field with a finger sets it active, as does the Tab key. | Anyone walking up to the device | The matching primary key. Case-insensitive on the backend so `GH-HR-001` and `gh-hr-001` both match the same row. |
| **Name input** | Free text, max 60 chars on the client and max 120 on the server. Shift starts on (capitalising the first letter of the first word), then flips off after the first letter; tapping space turns it back on for the next word (so "Ama Asante" comes out right by default). | Same | The "proof you typed an ID that's yours" step. A loose substring match against the user's full name — see the Data section. |
| **On-screen QWERTY keyboard** | Four rows of keys (number row + Q-row + A-row + Z-row) plus shift, backspace, clear, Tab, space, Enter. `mousedown.prevent` on every key keeps focus on the active input so the caret stays put and a tapped key always goes into the right field. Shift is "caps-once" — typing a letter while shift is on flips it off automatically, matching the convention of every other touch keyboard the user has ever met. | Same | Wall-mounted tablets often have no hardware keyboard. An on-screen one is the difference between "usable at reception" and "needs a Bluetooth dongle". |
| **Tab ⇄ key** | Switches the active field between Employee ID and Name. | Same | Faster than reaching out and tapping the other field, especially on tablets that are mounted above arm-comfort height. |
| **Clear key** | Empties the active field (not both). | Same | Single-mistake undo without resetting the other field. |
| **Backspace key** | Removes the last character from the active field. | Same | Standard. |
| **Shift key** | Toggles caps. Highlights in gold when on. | Same | "ATM-style" name entry for compound surnames where the user wants to enter the second word capitalised differently. |
| **Enter key (cobalt)** | Submits both fields to `POST /kiosk/verify`. Identical to clicking the "Continue →" button. A hardware keyboard's Enter key fires the same handler via `@keydown.enter` on each input. | Same | Single muscle-memory submit, regardless of input method. |
| **Continue →** (cobalt CTA at the bottom) | Same as Enter — submits to `/kiosk/verify`. The button text flips to "Verifying…" while the request is in flight and the button is disabled to prevent double-submit. | Same | Big finger target for the second-most-common path (tap-keyboard, then tap-CTA rather than tap-Enter). |
| **Inline error** (magenta) | Appears in place under the keyboard when the server returns `ok: false` from `/kiosk/verify`, or when the client-side guard catches an empty field. Disappears as soon as the user touches a key. | Same | The mistake is almost always a typo; the message ("Employee ID or name did not match. Please try again.") is deliberately vague to avoid revealing whether the ID exists or only the name was wrong (anti-enumeration). |

> *Notes:* Both inputs have `autocomplete="off"` so a shared device does not autofill with the previous user's name. The page never stores typed values in `localStorage`, `sessionStorage`, or a cookie — when the auto-reset timer fires (or "Done — next person" is tapped), `resetFlow()` zeros every reactive ref and the next person walks up to a blank form. The Employee ID input has `tracking-[0.18em]` monospace so a long ID is easy to read at arm's length without leaning in.

### Confirm stage

| Button / field | What it does | Who can use it | Why it exists |
|---|---|---|---|
| **Identity panel** (avatar + name + position + ID) | Read-only confirmation block. The avatar is the same `avatar_url` shown everywhere else in the app; if the employee has no avatar, the initial of their first name shows in a gradient circle. | Anyone (whoever just identified themselves) | The visual half of the identity check — the face and name on screen versus the person standing in front of it. This is the kiosk's authentication: a wrong typed name might still pass the substring match, but a wrong face won't pass the eyes of the person standing next in line. |
| **"Last today" hint** | "Last today: IN at 08:14" or "No events recorded today yet." Computed by `presentEmployee()` from the most recent record on today's date for this employee. | Same | Direction hinting. The employee glances at it and knows whether they're meant to clock in or out — important after lunch, especially for shift workers who forget which side of break they're on. |
| **Clock In** (large cobalt button) | Posts `{ employee_no, name, direction: 'in' }` to `/kiosk/clock`. On 200 OK with `ok: true`, the page flips to the Success stage. Has a gold ring (`[data-recommended="true"]`) when the suggested direction is "in". | Same | The headline action. Big finger target, high colour contrast, gold ring when it's the expected next move. |
| **Clock Out** (large ghost button) | Posts the same payload with `direction: 'out'`. Same flow, same ring when it's the suggested direction. | Same | The other half of the pair. Both are always tappable so a self-correcting employee ("oh, I already clocked in") can do the right thing without being blocked. |
| **← Not me, start over** (link) | Calls `resetFlow()` — zeros the typed values, returns to the Identify stage. | Same | False-match escape hatch. The loose substring match described in the Data section can occasionally surface the wrong employee (two people both named "Ama" in the same office); this is the way out without waiting six seconds. |
| **Inline error** (magenta) | Same as on the Identify stage — shows the server message from `/kiosk/clock` if the request fails. Does not eat the form state; the user can retry the same button. | Same | Same as above — usually a transient network or rate-limit error; the user just taps the button again. |

> *Notes:* The Confirm stage does not auto-advance and does not auto-reset — it waits for a tap on In, Out, or "Not me". The reason: someone whose name resolved correctly might still be deciding which button to press (the canonical case is "I just got in but need to step back out before counting as on-the-clock"). The Clock In button intentionally remains tappable even when "Clock Out" is the suggested direction; the daily-summary recompute is the canonical reconciliation, and double-in or double-out punches at the kiosk are recovered there rather than blocked here.

### Success stage

| Element | What it does | Why it exists |
|---|---|---|
| **Check medallion** | A 4.5 rem cobalt circle with a tick, pop-in via a 450 ms cubic-bezier transform. | Single reassuring visual — the punch is in. |
| **"Clocked IN" / "Clocked OUT" eyebrow** | The recorded direction in lowercase except for the verb. | Names the action that just happened. |
| **Employee name** | Full name in 40-pt bold. | Confirmation. |
| **Employee number** | Monospace, muted. | The audit handle. |
| **Recorded time** | The wall-clock time of the punch in 40-pt gold, monospaced, tabular numerals. Sourced from the `record.event_at` field returned by the server, not the client clock, so it is the same time payroll will see. | The one piece of information the employee leaves with. The gold and the tabular-nums treatment make it readable from the back of a queue. |
| **"Have a productive day."** | A closing line in tiny uppercase. | Human courtesy. |
| **"Done — next person"** (link) | Cancels the auto-reset and immediately flips the page back to Identify. | Lets the next person in the queue start without waiting six seconds. |

> *Notes:* The Success stage has a `setTimeout` (six seconds) that calls `resetFlow()` when it expires. Any tap on "Done — next person" clears the same timer (`clearTimeout`) before resetting, so the page does not accidentally double-fire the reset. The page does not navigate, refresh, or change URL during any of this — Inertia is not involved; the three stages are reactive state in a single Vue component. That means the page is offline-safe between renders: a momentary network drop after `/kiosk/clock` returns 200 does not prevent the Success stage from showing.

## The data behind it

Five HTTP routes, all prefixed `/kiosk`, all unauthenticated, all rate-limited at **60 requests per minute per IP** (`throttle:60,1`):

| Route | Method | Returns | Notes |
|---|---|---|---|
| `kiosk.show` | `GET /kiosk` | Inertia page (`Kiosk/Index`) + `serverTime` | The entry point. `serverTime` seeds the live clock so the first second on screen is the server's time, not the device's. |
| `kiosk.recent` | `GET /kiosk/recent` | JSON `{ recent: [...], serverTime }` | Polled every 15 seconds by the open page. Returns up to 8 of today's kiosk punches in descending time order, projecting only **first name + direction + event_at**. Staff IDs, last names, and positions are deliberately omitted — see "Public wall privacy" below. |
| `kiosk.verify` | `POST /kiosk/verify` | JSON `{ ok, employee?, message? }` | Identity check only — no record is written. Returns the matched employee (name, position, avatar URL, last event today, suggested direction) on success; returns 422 with `ok: false` on miss. |
| `kiosk.clock` | `POST /kiosk/clock` | JSON `{ ok, employee?, record?, message? }` | The actual write. Calls `AttendanceService::record()` with `source = AttendanceSource::WebKiosk` and `eventAt = now()`. No device ID, no GPS coordinates, no reason. Returns 422 with `ok: false` on any `DomainException` from the service. |
| `kiosk.face` | `POST /kiosk/face` | JSON `{ ok: false, message }` with status **501 Not Implemented** | Future face-recognition path. The route exists so device firmware and the frontend can wire against it without a backend redeploy; the implementation lands when the Ghana-Card biometric stack from Chapter 25 is wired in. |

What every reader of the screen needs to keep in mind:

- **No authentication. None.** The kiosk routes sit outside every auth middleware in the `web` group. The "authentication" for a punch is the loose substring match between the typed name and the employee's `users.name`. This is appropriate for the threat model — the device is on the office LAN, the screen is visible to everyone, false punches are visible on the live wall, and the daily-summary recompute is the canonical reconciliation. It would not be appropriate for any high-trust action (no payroll changes, no leave approvals, no document downloads); the kiosk is intentionally limited to write-one-row-into-attendance-records.
- **The identity match is "loose substring", not exact equals.** `KioskController::matchEmployee()` lowercases and trims both sides, then runs `str_contains($fullName, $typedName)`. "Ama" matches "Ama Asante"; "Asante" matches it too; "A" does not (the controller enforces a 2-character minimum to avoid trivial matches like "a" matching every employee). This is forgiveness-by-design — staff with diacritics in their official names (Akua Adoma vs the legal "Akua Adǝma") can still log in by typing the everyday spelling.
- **The same `AttendanceService::record()` runs as for every other source.** Kiosk gets no shortcuts: the service still writes one immutable `attendance_records` row, then synchronously calls `recomputeDailySummary()` to refresh the day's `attendance_summaries` row, then calls `OvertimeCalculator` for Labour Act §35 overtime, then statuses the day Present / Late / Half-Day / Absent / On-Leave / Holiday. The kiosk gets exactly the same lateness grace period, exactly the same overtime rates, and exactly the same payroll downstream as a biometric punch from a fingerprint device.
- **Geofence is not triggered on kiosk punches.** The Attendance service's geofence rejection (Chapter 5) only fires when the call passes a `deviceId` together with `geoLat`/`geoLng`. `KioskController::clock()` passes none of those, so the haversine check is skipped. The kiosk's location guarantee is physical — the device is bolted to the wall in the room where staff are supposed to be — not GPS. Phase 2 will tie the kiosk to a registered `BiometricDevice` row (with stored coordinates + a shared secret) so the same geofence rule can apply; that work is tracked in the punchlist alongside the face-recognition path.
- **Public wall privacy.** The "Today's wall" pill list at the bottom of the kiosk shows only `first_name + direction + event_at`. There is a dedicated test (`KioskRecentTest::it exposes only first name + direction + event_at — no Staff ID or full name`) that asserts the response payload does not carry `employee_no`, `name`, or `position`. This is deliberate: anyone within sight of the wall-mounted device — visitors, contractors, the person standing in the queue — must not be able to harvest staff IDs or surnames off the screen.
- **The wall is also a liveness indicator.** Because the recent-punches endpoint is polled every 15 seconds and renders pills as they arrive, the absence of new pills on a busy morning is a visible signal that something has gone wrong with the device or the network — IT does not need a separate uptime monitor for "is the kiosk working" beyond walking past the device.
- **Auto-reset and no persistence.** No typed value survives beyond the active flow. `resetFlow()` is called automatically six seconds after a successful punch, manually by "← Not me, start over", and manually by "Done — next person"; every call zeros `employeeNo`, `fullName`, `errorMessage`, `matched`, `lastRecord`, sets `activeField` back to the ID field, and turns shift back on. No `localStorage`, no `sessionStorage`, no cookies. The kiosk's working memory is its current Vue state and nothing else.
- **Rate limit is per-IP, not per-employee.** `throttle:60,1` means each IP can hit the kiosk routes 60 times a minute. On a single shared device this is generous — a punch takes at least a few seconds of typing — but it does mean that an attacker who can reach the URL on the LAN cannot brute-force the substring match faster than once per second on average. Combined with the 2-character minimum on the name field and the anti-enumeration "Employee ID or name did not match" generic error, the kiosk is not a usable channel for harvesting employee names.

## How it talks to other modules

- **Attendance (Ch 5)** is the only downstream consumer. Every kiosk clock hits `AttendanceService::record()` with `source = AttendanceSource::WebKiosk` and is reconciled into the same `attendance_summaries` row that the org-wide register, the personal "My Attendance" card, and the payroll engine all read from. There is no separate "kiosk records" table; the kiosk is a thin shell over Attendance.
- **Employees (Ch 3)** is the lookup target. The match runs against `employees.employee_no` joined to `users.name`, the same two columns every other identity lookup in the system uses; there is no kiosk-specific identifier or alias table.
- **Identity Verification (Ch 25)** is the planned upgrade path. The `/kiosk/face` route already exists as a 501 stub so the device firmware and the frontend can wire against it without a backend redeploy; when the Ghana-Card biometric SDK from Chapter 25 is integrated, the route turns into a real handler that takes a face image, matches it against the NIA template, and calls the same `AttendanceService::record()` with `source = WebKiosk` and a confidence score on the audit trail. The contract is already designed; only the SDK is missing.
- **Payroll (Ch 19)** never reads from the kiosk directly. It reads `attendance_summaries.overtime_hours`, which is recomputed every time a kiosk punch lands. The "punch on kiosk → flows into payroll" link is end-to-end through Attendance.
- **Biometric webhook** (Chapter 5) is the sibling channel for hardware fingerprint / face devices. Both write to the same `attendance_records` table via the same `AttendanceService::record()`; they differ only in their `source` value and in whether they carry a `device_id` + geo coordinates (the webhook does; the kiosk does not).
- **Notifications (Ch 16)** is intentionally not wired. The kiosk does not email or message anyone on a punch — the live wall + the success screen + the next render of the My Attendance page are the only notifications the punch triggers. A future "clocked-out without clocking-in" exception alert is roadmapped through the standard analytics-event listener path, not as a kiosk-specific feature.

## Standards touchpoints

- **Ghana Labour Act, 2003 (Act 651) §33-39 — Hours of work and overtime.** The kiosk is a recording channel, not a calculation engine; the calculation it feeds into is governed by `OvertimeCalculator` in Attendance (Chapter 5), which implements the §35 three-tier overtime rule. The kiosk's contribution to the standard is that it provides a recording option for workplaces without biometric devices or per-employee browsers — making the §33 weekly-hours obligation enforceable in field offices that would otherwise have to fall back on paper sign-in sheets. See Chapter 44.
- **Cybersecurity Act, 2020 (Act 1038) §59 — Critical information infrastructure protection.** Shared-device terminals are an attractive lateral-movement target on the office LAN; the kiosk is therefore deliberately minimal in attack surface — five routes, no authenticated state, no admin actions, rate-limited at the route layer, no persistence in the browser, and a payload contract that omits Staff IDs and surnames from the public wall. The Phase 2 hardening work (device fingerprinting, per-kiosk shared secrets matched to a `BiometricDevice` row, IP allow-listing per location) is tracked in the same punchlist as the face-recognition path.
- **WCAG 2.1 AA — Web Content Accessibility Guidelines.** The kiosk's on-screen keyboard, large text, high-contrast palette, big tap targets (Clock In / Clock Out are each ~144 px tall on a tablet), and live `role="alert"` announcement of the error message are the WCAG touchpoints for a shared-device interface that must work for someone with low vision, motor-impaired hands, or arthritis. The auto-reset timer of six seconds is on the long side of the recommended 3-5 second range deliberately — the standard's "Enough Time" guideline (§2.2.1) is honoured by giving users time to read the success screen before it disappears.
- **Data Protection Act, 2012 (Act 843) §18 — Data minimisation principle.** The public wall's contract — `first_name + direction + event_at`, with `employee_no`, `name`, and `position` actively excluded by the controller and asserted-out by a dedicated test — is the §18 touchpoint: only the minimum personal data necessary for the social-proof and liveness purposes leaves the device. See Chapter 26 for the full DPA mapping.

## What's planned next

Phase 2 of the government-grade roadmap (Chapter 46) lands four things on this module: (1) **Face recognition** — the existing `/kiosk/face` 501 stub turns into a working endpoint once the Ghana-Card biometric SDK from Chapter 25 is integrated, replacing the typed-name step with a camera frame and confidence-score check; (2) **Device registration** — every kiosk gets a registered `BiometricDevice` row (with stored coordinates, a HMAC `shared_secret`, and a `last_seen_at` heartbeat field), so the same geofence rule that protects the biometric webhook applies to kiosk punches and IT can see at-a-glance which terminals are online; (3) **Offline buffer** — a service-worker-cached version of the page that queues punches in IndexedDB during a network drop and replays them on reconnection, so a temporary office outage does not lose punches; (4) **Per-employee PIN fallback** — for workplaces that prefer not to display the typed-name match on screen, an opt-in 4-digit PIN issued via the staff-phone-PIN flow in Chapter 25, swapping the name input for a PIN pad. The screen-level gap — a dedicated admin page for IT to provision and revoke registered kiosks — is tracked in the same wave; the routes and service layer are already in place.
