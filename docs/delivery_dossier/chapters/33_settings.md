# Chapter 33 — Settings

> *In one paragraph.* Settings, in CIHRMS today, is not a single "system configuration" screen — it is the umbrella under which six small admin surfaces live: three asset libraries that feed the Documents pipeline (Stamps, Letterheads, Watermarks), and three integration surfaces that govern how external systems talk to the institute (API Tokens, Webhooks, API Reference). The "Settings" link in the sidebar itself points at the My Profile page in Chapter 16 — there is no central organization-profile/branding/locale page in the MVP because those values live in environment variables and seeders, not in the database. Everything else that feels like a "system setting" lives in its purpose-built module: notification channel toggles in Notifications (Ch 16), SMS provider config in Messaging Admin (Ch 15), retention defaults in DPA (Ch 26), SSO providers in their own admin (Ch 28), and integrations such as Paystack in the Integrations admin (Ch 28). This chapter documents the six screens that genuinely sit under the `/settings/...` URL prefix and the `Settings/` Vue namespace, plus the honest gaps around the configuration that should live here in Phase 1 but does not yet.

## Where to find it

- **Sidebar location:** **System** group, towards the bottom of the rail. The umbrella exposes six entries:
    - **API Tokens** (`vpn_key` icon, route `api-tokens.index`) — gated on `api.token_manage`.
    - **Webhooks** (`webhook` icon, route `webhooks.index`) — gated on `api.webhooks_manage`.
    - **API Docs** (`menu_book` icon, route `api.docs`) — visible to every signed-in user; the page itself is the live developer reference.
    - **Settings** (`settings` icon, route `profile.edit`) — this is the catch-all label; today it deep-links to My Profile (Ch 16) because there is no system-config screen yet.
    - **Audit Logs** (full-system roles only) — covered in Chapter 24, listed here only because it lives in the same sidebar section.
    - The three asset libraries — **Stamps**, **Letterheads**, **Watermarks** — are not in the sidebar at all. They are reached from inside the Documents module (Ch 13) and the composer's asset picker, but they live at `/settings/stamps`, `/settings/letterheads`, `/settings/watermarks` and are documented here because they are administrative-grade, organisation-scoped, and they share the `Settings/` Vue folder.
- **Roles that see it:**
    - **super_admin** and **ceo** — every entry, every action. CEO mirrors super_admin through the `Gate::before` fall-through (see Ch 24).
    - **hr_admin** — everything except API Tokens and Webhooks unless explicitly granted; holds `document_assets.manage` (the flag that unlocks the **organization** scope in every asset library) by default in the seeded RBAC.
    - **dept_head / manager** — can see the asset libraries (personal + their department) but the **organization** radio is disabled unless they hold `document_assets.manage`. They do not see API Tokens or Webhooks.
    - **employee / finance_officer / it_support / auditor** — personal-scope asset libraries only; no API Tokens, no Webhooks. API Docs is visible because the page itself is internal documentation, not a privileged tool.
- **Related modules:** Documents (Ch 13) — every stamp, letterhead, and watermark is consumed there; the composer's letterhead picker, the annotation toolbar's stamp picker, and the burn-time watermark layer all read from these libraries. Notifications (Ch 16) — channel toggles for email/SMS/in-app live on the per-user notification preferences screen, not here. Messaging Admin (Ch 15) — the SMS provider config (Hubtel/Twilio/Arkesel keys) is environment-driven and surfaces in the Messaging admin's "Provider" tab. DPA (Ch 26) — retention defaults and erasure schedules sit on the privacy admin, not in Settings. SSO Providers (Ch 28) — the OIDC/SAML credential vault is at `/sso-admin`, gated on `sso.manage`. Integrations (Ch 28) — the Paystack public/secret key pair, Zoho refresh token, and other third-party credentials are in `/admin/integrations`, gated on `integrations.manage`.

## Every button, every action

### API Tokens (`/admin/api-tokens`)

| Button / field | What it does | Who can use it | Why it exists |
|---|---|---|---|
| **Issue token** (cobalt CTA, top right) | Opens the right-hand slide panel. Required fields are Name and at least one Scope; optional fields are Audit note (purpose), Rate limit per minute (1–6000, defaults to 60), Expires in days (1–3650, defaults to 365). On submit, calls `Sanctum::createToken` on the current user with the chosen abilities and writes an `api_token_metadata` row with purpose, rate limit, expiry, and issuer. | `api.token_manage` | One-step issuance for partner systems (GIFMIS, IPPD, Ghana Card). The metadata row carries the audit story the bare Sanctum table cannot — who asked for it, why, and how fast it can hit the API. |
| **Token name** | Required, max 120 chars. | Same | Free-text label that appears in the table and in the audit log; "GIFMIS production integration" is more useful than "Token #34". |
| **Audit note (purpose)** | Optional textarea, max 500 chars. | Same | Surfaces in the audit log and in a tooltip on the token row. The intent is that every revoke decision has the original "why" one click away. |
| **Scopes** (multi-select grid) | At least one required. The seeded scopes are `employees:read`, `payroll:read`, `attendance:read`, `positions:read`, `statutory:export`, `identity:read`, `identity:write`, `webhooks:manage`, and `*`. The `*` wildcard is rendered in red with a "use sparingly" footnote. | Same | Sanctum abilities are checked per-endpoint; minimum-scope tokens limit the blast radius of a leak. |
| **Rate limit / min** | Optional integer 1–6000, defaults to 60. | Same | Per-token throttling at the gateway; partners with batch jobs can be raised to 600/min, public-facing scripts kept at 60. |
| **Expires in days** | Optional integer 1–3650, defaults to 365. | Same | Every token has a default sunset; long-lived integrations must be renewed annually, which forces the audit conversation on rotation. |
| **Issue token** (footer) | Posts to `POST /admin/api-tokens`. On success the page redirects back to the index with `flash_token` set to the plaintext; the secret banner reveals it once. | Same | Sanctum stores only the hash — once the response is gone, the token cannot be retrieved. The single-reveal banner is the only place it is ever shown. |
| **Copy** (on the secret banner) | Writes the plaintext to the clipboard via `navigator.clipboard.writeText`; the button turns emerald and reads "Copied" for 2.5 s. Silent fail if clipboard access is blocked. | Anyone seeing the banner | The token is long and base64-ish; manual copying is error-prone. |
| **Revoke** (per row, rose button) | Confirmation dialog ("Revoke this token? Any system using it will be disconnected immediately.") → DELETE `/admin/api-tokens/{id}`. Sets `revoked_at` and `revoked_by` on the metadata row, then calls `delete()` on the `PersonalAccessToken` itself. | `api.token_manage` | Compromise response — instant cut-off with an audit trail of who pulled the plug. |
| **KPI tile band** | Read-only — Total issued (from `tokens.meta.total`), Active (`is_usable` true), Used 24h (last-used within 86 400 000 ms), Revoked (`revoked_at` set), Expired (not usable and not revoked). | Anyone with view | The five questions an integrations lead asks on landing: "how many do we have, how many are live, how many actually get used, how many have we killed, how many have aged out?". |

> *Notes:* The plaintext is never persisted — Sanctum hashes it before storing. The `flash_token` survives exactly one Inertia redirect cycle thanks to Laravel's session flash mechanism; refresh the page and the banner is gone. The metadata table carries `allowed_ip_cidrs` (validated on issue) and `issued_to_user_id` (so an admin can issue tokens *to* another service account) but the MVP UI does not surface either yet — they are reachable via the API. `is_usable()` is the canonical "can this token still authenticate a request?" check; it folds expiry and revocation into a single predicate.

The MVP works because the table accepts the alternative names through the Eloquent fillable list, but the audit trail will flag this as an inconsistency to clean before market release — it is listed honestly in the "What's planned next" section below. Either the migration column names or the controller payload need to converge; the recommendation is to migrate the column names to match the API surface (`callback_url`, `subscribed_events`, `failure_count`, `secret`) because the API surface is the contract partners see.

## The data behind it

There is, today, no `organization_settings` table and no `Setting` model. The values that some HRMS products keep in such a table live in three places in CIHRMS:

- **Environment variables (`.env`)** — the organisation name (`APP_NAME`), locale (`APP_LOCALE`, defaults to `en`), timezone (`APP_TIMEZONE`, defaults to `Africa/Accra`), default currency (`CASHIER_CURRENCY`, defaults to `GHS`), mail driver and credentials, queue connection, broadcast driver, and every third-party API key (Paystack public/secret, Zoho refresh token, NIA endpoint URL, etc.). These are read by `config/*.php` files and cached at boot via `config:cache` — they are immutable from the running application and require a deploy to change.
- **Seeders (`database/seeders/*.php`)** — the seeded organisation letterhead (the default `is_default = true` row in `letterhead_templates`), the seeded role catalogue (`RolePermissionSeeder`), the seeded GL chart of accounts (`ChartOfAccountsSeeder`), the seeded tax brackets and statutory rates. These can be re-run on demand but are not edited from the UI.
- **Per-feature admin pages** documented in their own chapters — notification preferences in Ch 16, SMS provider in Ch 15, retention defaults in Ch 26, SSO providers in Ch 28, Paystack/Zoho/integration credentials in Ch 28.

What this chapter's six screens *do* persist in the database is:

- **`stamp_assets`** — id, `owner_scope` (enum `personal` / `department` / `organization` via `AssetOwnerScope`), `owner_id` (nullable — the user id, department id, or null for org-wide), `name`, `storage_path` (relative to the local disk), `mime`, `default_w_pct`, `default_h_pct`, `created_by`, timestamps. PNG bytes live on the `local` disk under `stamps/`.
- **`letterhead_templates`** — same scope/ownership columns plus `header_height_mm`, `is_default`, the PNG/JPEG path, and a `created_by`. The seeded organisation default has `is_default = true` and is policy-protected against deletion.
- **`watermark_templates`** — same scope/ownership columns plus `type` (`text` / `image`), `text`, `color`, `opacity`, `angle_deg`, and an optional `storage_path` for image watermarks. Text watermarks store no file.
- **`personal_access_tokens`** (the Sanctum table) plus **`api_token_metadata`** — the latter carries the audit context Sanctum's bare table does not: `purpose`, `rate_limit_per_minute`, `expires_at`, `revoked_at`, `revoked_by`, `issued_to_user_id`, `issued_by_user_id`, `allowed_ip_cidrs` (JSON). The token plaintext is never stored — only the hash, on the Sanctum row.
- **`webhook_subscriptions`** plus **`webhook_deliveries`** — the subscription row carries the partner name, callback URL, the encrypted signing secret (`encrypted` cast in the model), the JSON event-type array, an active flag, the creator, the last success and failure timestamps, and the consecutive-failure counter. The deliveries table logs every attempt with response code, response body excerpt, attempt counter, and final status (`pending` / `delivered` / `failed` / `retrying`); it powers the replay-during-reconciliation flow.

What every reader of these screens needs to keep in mind:

- **Plaintext is shown exactly once.** Both API tokens and webhook signing secrets are revealed only on the redirect immediately after creation. The flash store carries them for one Inertia hop; refresh the page and they are gone. The database holds only the hash (tokens) or the encrypted blob (webhook secrets), and there is no admin path to recover the original.
- **Asset deletion is hard, not soft.** Unlike most CIHRMS records, removing a stamp/letterhead/watermark unlinks the file from disk and drops the row. There is no soft-delete because there is no audit benefit to retaining a removed visual asset — what was burned into a PDF is already inside that PDF.
- **Default letterhead is protected.** The seeded `is_default = true` row cannot be deleted; the policy refuses and the UI hides the Remove button. If the row is missing on boot the `LetterheadSeeder` reinstates it.
- **Scope is enforced server-side.** The asset library index controllers compose a `where` over the three scope variants, parameterised by the current user's id and department id. The Personal/Department/Organization tabs in the UI are a client-side filter over the result set, not a re-query — the user only ever sees rows the controller deemed visible.
- **Webhook subscribers see only what they asked for.** Even when an event matches `subscribesTo()`, delivery is filtered by the per-subscription `event_types` array (with `*` as the catch-all). A partner that subscribes only to `payroll.run.approved` will not see `loan.disbursed` even if the dispatcher publishes both.
- **The auto-pause threshold is 10 consecutive failures.** This lives in the delivery worker, not in the Settings UI, but the index header advertises it so admins are not surprised when a long-broken endpoint stops being retried.
- **No system-config screen exists in the MVP.** The "Settings" sidebar link points at My Profile because that is where personal preferences live (theme, language, password, avatar, sound pack — see Ch 16). The organisation-level configuration that would normally sit behind a settings page is currently environment-driven and requires a deploy to change. This is on the Phase 1 list (see "What's planned next").

## How it talks to other modules

- **Documents (Ch 13)** is the sole consumer of the three asset libraries. The composer reads `LetterheadTemplate::visibleTo($user)` to populate its letterhead picker. The annotation toolbar reads `StampAsset::visibleTo($user)` for the stamp picker. The PDF burn path reads `WatermarkTemplate::find($document->watermark_id)` to composite the watermark layer onto the flattened output. Removing an asset that has already been burned into a PDF does not affect the existing PDF — the burn was at render time, not by reference.
- **Documents (Ch 13)** also reuses the same `Settings/Stamps.vue`, `Settings/Letterheads.vue`, and `Settings/Watermarks.vue` pages — the screens live in the `Settings/` Vue namespace but the documentation cross-link in Ch 13 points at the same files. Either chapter can be the entry point.
- **Notifications (Ch 16)** dispatches webhook events through the same queue worker that posts to partner endpoints. When a `payroll.run.approved` notification fires, the worker fans out to every `WebhookSubscription::active()->subscribesTo('payroll.run.approved')` row. Failures bump `consecutive_failures` on the subscription and are logged to `webhook_deliveries` for replay.
- **Audit Logs (Ch 24)** records every API-token issue, revoke, webhook-subscription create, update, and delete via the `Auditable` trait on each model. The `purpose` field on the token metadata and the `partner_name` on the subscription provide the human-readable context the audit log displays.
- **Integrations admin (Ch 28)** is the other "credential vault" surface in the system — Paystack keys, Zoho refresh tokens, NIA endpoint URLs, biometric device PSKs. It is gated on `integrations.manage`, not on `api.token_manage`, because those are inbound third-party credentials, not tokens *issued by* CIHRMS. The two should not be confused: API Tokens are what *partners* use to call *us*; Integrations are what *we* use to call *them*.
- **SSO Providers (Ch 28)** holds the OIDC and SAML credential pairs for Google, Microsoft, and any government IdP. Like Integrations, it is gated on `sso.manage` and lives outside the Settings umbrella, even though conceptually it is a system setting.
- **Messaging admin (Ch 15)** holds the SMS provider toggle (Hubtel / Twilio / Arkesel) — the keys themselves live in `.env`, the selector lives on the Messaging admin's "Provider" tab.
- **DPA (Ch 26)** owns the retention defaults, the data-subject-request workflow, and the consent registry. Retention windows are not exposed as a system setting because they are policy decisions tied to the Data Protection Act, not preferences.
- **My Profile (Ch 16)** is what the "Settings" sidebar link points at today. Personal preferences (theme, language, password, avatar, sound pack) live there; system-wide preferences do not yet have a home.

## Standards touchpoints

- **ISO/IEC 27001:2022 Annex A.5.16 (Identity management) and A.5.17 (Authentication information)** — every API token is scoped, time-bounded, revocable, and audited; the secret is hashed at rest, shown once, and can be cut off instantly. The webhook signing secret follows the same pattern (encrypted at rest, shown once, rotatable by re-registering). See Chapter 44.
- **ISO/IEC 27001:2022 Annex A.8.5 (Secure authentication)** — the wildcard `*` scope on a token is visually called out in red with a "use sparingly" footnote; the default token expiry is 365 days; partner endpoints must be HTTPS. The webhook signing recipe uses HMAC-SHA256 over `timestamp + '.' + body`, mitigating both transport tampering and replay (the timestamp window is enforced at the partner end). See Chapter 44.
- **Data Protection Act, 2012 (Act 843) §28 — security safeguards** — the encrypted signing secret column on `webhook_subscriptions`, the hashed token storage on `personal_access_tokens`, and the policy-enforced scope checks on the asset libraries are the operational expression of the Act's "appropriate technical and organisational measures" obligation. The annual audit pack collects evidence from these tables. See Chapter 27.
- **OpenAPI 3.1** — the API Docs page serves both YAML and JSON encodings of a single hand-maintained spec; partners can generate clients from either. The Stoplight Elements explorer is embedded in-portal so first-time integrators can probe endpoints without leaving the application.
- **National Identification Authority Act, 2006 (Act 707) — Ghana Card integration** — the webhook event catalogue includes `identity.verified` and `identity.duplicate_detected` so downstream systems (GIFMIS, IPPD, banks) can react to verification outcomes without polling. See Chapter 25.

## What's planned next

Phase 1 of the government-grade roadmap lands four things on this surface: (1) **An actual `OrganizationSetting` model and `/settings/organization` page** — institute name, address, contact, primary logo, favicon, hero colour, default currency (overriding the env default), default locale, default timezone, and a feature-flag panel for toggling whole modules on or off without a deploy. This closes the gap that today forces "set the institute name" into a deploy-time `.env` edit. (2) **A `/settings/branding` sub-page** for the visual layer — uploadable logo, favicon, login-page background, accent colour, email template header — so a hosted multi-institute deployment can re-skin without code changes. (3) **The webhook schema/controller drift fixed** — either the migration columns rename to `callback_url`, `subscribed_events`, `failure_count`, `secret` (the recommended path, because that is the contract partners see) or the controller payload renames to the migration's `target_url`, `event_types`, `consecutive_failures`, `signing_secret`. The audit pack in Ch 24 will flag this until it converges. (4) **A CI check that diffs `routes/api.php` against the curated endpoint catalogue** in `Settings/ApiDocs/Index.vue` so a new endpoint cannot land without a documentation update. Beyond Phase 1, the asset libraries will gain an "Insert into composer" action that opens the document composer with the chosen letterhead pre-selected, and the API Tokens screen will gain a per-token IP allow-list UI to surface the `allowed_ip_cidrs` column that exists today but is API-only.
