# Chapter 18 — Payments

> *In one paragraph.* Payments is the **one-off payment ledger** — every cedi that leaves CIHRMS *outside* the monthly Payroll Engine (Chapter 19) gets written here. A reimbursement for a HR officer who paid for the team's lunch out of pocket; a single ex-gratia bonus for a security guard who stopped a break-in; a Christmas hamper top-up; a one-time advance against an unbanked driver's salary; a one-shot consultancy fee for an external trainer who is not on payroll — all of these are Payment rows. The screen is deliberately humble: pick an employee, type a description, type an amount, save. It lives next to the per-employee Ghana payslip generator that turns a single basic-salary number into a fully-statutory payslip with PAYE and SSNIT bands shown. What the Payments module is **not** is the monthly statutory salary machine — Payroll Engine (Chapter 19) owns dual-control approval, batch PayrollLine generation, GRA/SSNIT/NHIA/Tier-2 schedule exports, the IPPD2/IPPD3 file, and the GIFMIS journal. This chapter is the chequebook; Chapter 19 is the salary register.

## Where to find it

- **Sidebar location:** **Workforce & Finance** group → **Payroll** (top-level item, `payments` icon). The label still reads "Payroll" in the MVP UI for historical reasons — this is the screen long-tenured users will type-ahead to when they look for "make a payment". Routes are mounted under `/payments`, the controller is `PaymentController`, the model is `Payment`, the URL slug stays as `payments` so the API and saved bookmarks survive. A separate **Payroll Runs** entry on the same menu group leads to the batch engine in Chapter 19.
- **Roles that see it:**
    - **super_admin** and **ceo** — every payment, every field. `PaymentPolicy::before()` grants `super_admin` a blanket pass.
    - **hr_admin** — creates, marks paid, generates per-payment payslips. Holds `payroll.manage`.
    - **finance_officer** — same operational permissions in the MVP (`payroll.manage` is the single gate on the route group); the splitting between "HR creates" and "Finance disburses" that exists for AP Payments (Chapter 20) is *not* enforced here yet.
    - **employee** — does not reach `/payments` directly; their own payments surface as the Payroll tab on their employee detail page (Chapter 3) and via the Profile portal (Chapter 32). They never see another employee's payments.
    - **auditor, dept_head, manager, it_support** — no access in the MVP. The whole route group sits behind `middleware('permission:payroll.manage')`, so anyone without that gate is bounced before the controller is even constructed.
- **Related modules:** Employees (Ch 3) — every payment points at one `Employee` (`employee_id`); Departments (Ch 15) — analytics groups top earners by department via the joined Employee row; Payroll Engine (Ch 19) — the monthly statutory salary machine that this module *is not* (see "Distinction from Payroll Engine" below); Loans (Ch 21) — loan disbursements are issued through this module today, pending the dedicated loan-disbursement flow noted in Chapter 22's roadmap; Disbursements (Ch 22) — off-cycle one-offs run through `PaymentService` rather than `BatchDisbursementService`, with no GhIPSS batch file; Audit Logs (Ch 24) — every create and every mark-paid would land here once the dispatch is wired (see "Honest gaps" below); Notifications (Ch 16) — `PaymentMarkedPaid` is in the bell's icon palette today, ready for the producer to fire; Profile Portal (Ch 32) — the employee-facing read view; Reports (Ch 33) — `PayrollExport` reads from this table for the month-scoped Excel pull.

## The data behind it

CIHRMS stores Payments in a single table with one child table for line items:

- **`payments`** — one row per off-cycle payment. Carries `employee_id` (nullable, with `nullOnDelete` — a terminated employee's payments survive their deletion), `processed_by` (the User id that last touched the row), `description` (free text, max 255), `amount` (decimal(12,2)), `currency` (string, default `GHS`), `status` (string, defaults to `pending`, cast to `PaymentStatus` enum on read), `paid_at` (nullable timestamp), and the standard Eloquent timestamps plus `SoftDeletes`. There is **no `reference` column, no `period` column, no `category` column**, and no `payroll_run_id` foreign key in the MVP — the description carries the period as English ("Monthly Salary - May 2026"), and a one-off Payment is structurally independent of any Payroll Run.
- **`payroll_items`** — one row per line item on a generated payslip. Carries `payment_id`, `label`, `type` (`earning` or `deduction`), and `amount` (signed — deductions are stored as negative numbers so the sum-over-amount of every line item equals the payment's net pay).

The `Payment` model casts `status` to the `PaymentStatus` enum, which has four cases:

| Case | Value | Label | When it appears |
|---|---|---|---|
| `Pending` | `pending` | Pending | Default on create. Awaiting "Mark as Paid". |
| `Paid` | `paid` | Paid | Set by `markPaid()` or by `generatePayslip(mark_paid: true)`. Stamps `paid_at`. |
| `Failed` | `failed` | Failed | Reserved — no UI path writes this in the MVP. The status exists on the enum so a future payment-gateway integration can flip the row when a Paystack/MoMo callback returns a hard failure. |
| `Cancelled` | `cancelled` | Cancelled | Reserved — no UI path writes this either. The intent is "soft-delete with a reason"; today the only way out is `SoftDeletes` (via the model trait), and the slide-panel doesn't expose a Cancel button. See "Honest gaps" below. |

What every reader of the screen needs to keep in mind:

- **Currency is recorded but not converted.** A USD payment for an external trainer sits in the same table as a GHS payment for a salary advance. Analytics splits them per currency (the "Currency Mix" card) rather than translating to a single base. Reporting that needs a base-currency view today reads each currency separately.
- **`processed_by` is overwritten on every touch.** When a row is created it captures the creator; when it is marked paid it captures the *marker*. The two are often the same person but not always — there is no separate `created_by` column. An audit team that wants to know "who created this and who marked it paid" must reach for the audit log (Chapter 24) once the dispatch is wired.
- **No `payroll_run_id` foreign key.** Payments and Payroll Runs (Chapter 19) are two completely separate tables. A per-payment payslip generated from this module does *not* appear on any Payroll Run, does *not* contribute to the cached totals on `payroll_runs`, and does *not* contribute to the GIFMIS journal voucher. It is a peer of the batch system, not a child of it.
- **Statutory deductions on a generated payslip are stored as PayrollItem rows, not as separate `paye_amount` / `ssnit_amount` columns.** The split between "what is statutory" and "what is voluntary" is purely a render-time decision on the Show page — it regexes the PayrollItem labels. Any change to the label format will require a coordinated change to the render-time matcher.

## How it talks to other modules

- **Employees (Ch 3)** — every Payment row points at one `Employee`, and `Employee::payments()` is the `hasMany` relation the Employee detail page (Ch 3, Payroll tab) reads to show the most recent eight payments. The Employee row's `user.name`, `employee_no`, and `department.name` are joined into the analytics row's "Top Earners" card.
- **Departments (Ch 15)** — the analytics "Top Earners" card joins through the employee row to surface the department name. There is no per-department aggregation today — the analytics are global.
- **Payroll Engine (Ch 19)** — see "Distinction from Payroll Engine" below. The two modules share the `PayrollCalculator` (statutory bracket logic), the per-employee payslip lineage, and the legacy `Payment` model name, but they are structurally independent: a one-off Payment has no `payroll_run_id`, does not appear on the Payroll Run detail page, and does not flow through the dual-control approval workflow.
- **Loans (Ch 21)** — loan disbursements are issued through this module today (per Ch 22's note: "Payments (Ch 18) — off-cycle one-offs … run through `PaymentService` and reuse the same provider clients indirectly. They are not part of a batch and do not appear on the ledger; they get their own Payment record."). The dedicated loan-disbursement journal voucher and the loan-side ledger entries are out of scope until the loan-disbursement workflow lands in Phase 2.
- **Disbursements (Ch 22)** — Payments are **off-cycle** by definition. They do not enter `BatchDisbursementService`, do not get a `disbursements` row, and do not contribute to the GhIPSS bank-credit file. Ch 22 explicitly carves them out: "they are not part of a batch and do not appear on the ledger".
- **Audit Logs (Ch 24)** — the audit log is the same `audit_logs` infrastructure as every other module. Today, no event fires on Payment create or markPaid (see "Honest gaps"); the audit footprint of a one-off payment is the row itself plus the `processed_by` stamp.
- **Notifications (Ch 16)** — `PaymentMarkedPaid` is in the bell's icon palette already (Ch 16 confirms: "PaymentMarkedPaid exists in the UI type palette but the producer hasn't shipped; the bell already knows how to render it"). When the producer is wired, the notification template, icon, and colour tag are ready.
- **Profile Portal (Ch 32)** — employees see their own one-off Payments on the Payroll tab of their profile. The portal honours the same `PaymentPolicy::view` gate as the HR-facing detail page.
- **Reports (Ch 33)** — `App\Exports\PayrollExport` reads from the `payments` table filtered by `YearMonth(created_at) = ?`, with headings `[Employee, Description, Amount, Currency, Status, Paid At]`. The export is reached from the Reports module's monthly-payroll card, not from the Payments index — the Payments index has no explicit Export button in the MVP UI (see "Honest gaps").
- **Per-payment payslip mirror to OneDrive/Drive** — `App\Events\PayslipGenerated` exists with the `UploadPayslipToCloud` listener (gated by the `mirror_documents_to_cloud` feature flag). The listener uploads to `CIHRMS/Payslips/{year}/{employee_no}/{filename}` when an integration is connected. **The event is never dispatched anywhere in the codebase today** — Ch 19 explicitly calls this out: "the per-payment payslip lineage is the only path that has the cloud-mirror listener wired, but the producer is missing." See "Honest gaps" below.

## Distinction from Payroll Engine (Chapter 19)

This is the single most important section of the chapter. The two modules sit next to each other in the sidebar; they share the `PayrollCalculator`; they share the legacy `Payment` model name on certain join paths. They are not the same thing.

| Aspect | Payments (Ch 18) | Payroll Engine (Ch 19) |
|---|---|---|
| **Cadence** | One-off, on-demand. A row is created when an operator clicks "Quick Payment" or "Generate Payslip". | Monthly, batch. A `PayrollRun` is created for a (year, month, scope) and runs over every active employee at once. |
| **Trigger** | Manual — operator opens the slide panel, fills the form, clicks Save. | Manual create → calculate → approve → paid lifecycle, with dual-control on approve (the creator can't approve their own run) and fresh-2FA on approve and reverse. |
| **Statutory deductions** | **Optional**. Quick Payment writes zero deductions; Generate Payslip writes the full Ghana statutory set (PAYE + SSNIT Tier-1 5.5% + Tier-3 voluntary if entered + every cash allowance as fully taxable income). | **Mandatory**. Every PayrollLine has the four statutory deductions (PAYE, SSNIT Tier-1 employee 5.5%, SSNIT Tier-1 employer 13%, NHIA 2.5% split, Tier-2 employer 5%) plus voluntary deductions with the 1/3 take-home floor. |
| **Approval workflow** | None. The single `markPaid` action flips status — the same user who created it can mark it paid. | Dual control. `PayrollRunPolicy::approve` refuses if `created_by === user_id`; the route sits behind `2fa:fresh` middleware. |
| **Output files** | None today. The per-payment payslip can in principle be mirrored to OneDrive / Drive via `PayslipGenerated` + `UploadPayslipToCloud`, but the producer hasn't shipped. | Full statutory export — GRA PAYE return CSV, SSNIT Tier-1 schedule, NHIA allocation, one Tier-2 schedule per active NPRA trustee, GhIPSS bank-credit file, plus the IPPD2/IPPD3 CAGD file and the GIFMIS sub-ledger journal voucher. |
| **Ghost-worker gates** | None. The operator selects an employee from the dropdown — no identity-verification check, no zero-attendance gate. | Two hard gates before producing a line: `Employee::hasUsableIdentity()` (Ghana Card verified against NIA) and `AttendanceService::aggregatePeriod()` (days_worked + days_on_leave > 0). |
| **GIFMIS journal voucher** | Never generated. A one-off Payment does not contribute to any sub-ledger JV. | Minted on the Paid transition (when `payroll.gifmis.auto_mint_on_paid` is on) with the balanced double-entry that the GIFMIS bulk loader accepts. |
| **Disbursement file** | Never generated. The Finance team pays the employee through whatever rail they already use; the Payment row records the fact. | `MaterialiseDisbursements` listener creates one Disbursement row per PayrollLine on approval, ready for GhIPSS / MoMo push. |
| **Reversal** | None. The only way out is `SoftDeletes` from the console. | Full `reverse` lifecycle with a required reason ≥ 10 chars, fresh-2FA, audit-log signature, and `PayrollRunReversed` event that fans out to the loan-repayment ledger. |
| **Use case** | Reimbursements, ex-gratia bonuses, salary advances, ad-hoc consultancy fees, one-time top-ups, hire-month proration, mid-period back-pay, loan disbursements (today). | The monthly salary register — payroll for the institute, end to end, every month. |

The simplest rule for an operator: **if it happens every month for every employee, it's Payroll Engine. If it happens once, for one employee, with no statutory schedule waiting at the GRA portal, it's Payments.**

## Standards touchpoints

- **Income Tax Act, 2015 (Act 896), §116 — Withholding tax on employment income (PAYE)** — the per-payment payslip path runs the same `PayrollCalculator` as the batch engine, which transcribes the seven-band monthly PAYE schedule from Act 896 Sixth Schedule as amended for FY 2026 (0% to GHS 490, 5% to 600, 10% to 730, 17.5% to 3,896.67, 25% to 19,896.67, 30% to 50,416.67, 35% above). The compliance strip on the index page reminds the operator of the 15th-of-following-month deadline for filing and remittance. The actual filing happens through Chapter 19's GRA PAYE return generator — this module does not file. See Chapter 44.
- **National Pensions Act, 2008 (Act 766), §3 — Mandatory three-tier pension scheme** — Tier-1 (5.5% employee + 13% employer to SSNIT), Tier-2 (5% employer to NPRA-licensed trustee), and Tier-3 (voluntary, up to 16.5% combined, tax-relieved) are all computed by the calculator on Generate Payslip. The SSNIT Maximum Insurable Earnings cap (GHS 61,000/month effective 1 Jan 2025) is honoured. The Tier-3 cap (16.5% of basic) is displayed live under the field. See Chapter 44.
- **Social Security and National Insurance Trust Act, 1991 (PNDCL 247), §64 — Contribution remittance deadlines** — the index compliance strip and the per-payment Remittance Schedule card both name the 14th-of-following-month SSNIT deadline. The actual SSNIT Tier-1 schedule generator that the contribution remittance is filed against sits in Chapter 19. See Chapter 44.
- **National Health Insurance Act, 2012 (Act 852), §29 — NHIA contribution** — the 2.5% portion of SSNIT Tier-1 employer that routes to NHIA is in the `PayrollCalculator` and is part of the Generate Payslip workings, but a one-off Payment does not by itself produce a separate NHIA allocation file — that's the Chapter 19 statutory return generator's job. See Chapter 44.
- **Data Protection Act, 2012 (Act 843) §§17, 18 — Lawful processing of personal financial data** — salary and bank-account information surfaced on the per-payment payslip is gated at `PaymentPolicy::view`. An employee can view their own Payments; another employee cannot. Bulk export through the Reports module (Ch 33) is behind a separate permission. See Chapter 26 and Chapter 44.
- **IFRS — Expense recognition (matching principle)** — every Payment row is an expense at the point it is created, not at the point cash leaves the bank account. The `status = pending` state is the accruals view; `status = paid` is the cash view. Until the GIFMIS journal hookup lands (Phase 2), the bookkeeping is done by the Finance team off-system. The Lifetime paid analytics card and the monthly volume area chart are the two operational views that approximate the accrual schedule for the period. See Chapter 44.
- **Electronic Transfer Levy Act, 2022 (Act 1075) — E-Levy** — out of scope for this module. E-Levy is applied at the disbursement layer (`BatchDisbursementService::E_LEVY_FALLBACK_RATE = 0.015`), and one-off Payments do not enter the batch disbursement service. If a one-off Payment is settled by mobile-money out-of-system, the levy is borne by the rail, not by CIHRMS. See Chapter 22 and Chapter 44.

## On the SequenceService reference convention

Every reference in the Finance build-out (F2 onwards) is minted by `App\Services\SequenceService::next($prefix, $padding, $year)`, which `lockForUpdate`s a counter row in the `sequences` table and atomically increments it. This was put in place after PR #21 closed the C3 race-condition gap that came from generating references with `Model::count() + 1` (per the `project_finance_sequences.md` memory entry). The AP Payment module (Chapter 20) mints `APP-2026-NNNN`, the AR Receipt module `ARR-2026-NNNN`, the Payment Intent module `PI-NNNNNN`, the Journal Engine `JE-NNNNNN` — all through the same service.

**The Payments module does not use SequenceService today.** The `payments` table has no `reference` column at all. The user-facing identifier on the Show page is the integer primary key (`Payment #123`); the masthead breadcrumb on Show reads `Payroll / Payslip #{id}`; the row in the table has no reference column. This is fine for the MVP because the row's natural key is the (employee, description, amount, created_at) tuple — the description carries the period as English text. But adding a `PAY-YYYY-NNNN` reference is the correct path forward, and when it lands it must go through `SequenceService::next('PAY', 4, now()->year)` rather than `Payment::count() + 1`. The convention is enforced by the audit pass that landed with PR #44 ("V2 audit complete" in the memory entries).

## Honest gaps

The Payments module is one of the oldest in the codebase and shows it. The gaps are:

- **No domain events fired on create or markPaid.** `PaymentService::create()` and `PaymentService::markPaid()` neither dispatch nor `event(...)`. The `PaymentMarkedPaid` icon and colour token already sit in the Notifications palette (Ch 16) waiting for a producer; the `audit_logs` table never sees a Payments-module row today. Wiring `event(new PaymentCreated(...))` and `event(new PaymentPaid(...))` is a one-day change.
- **`PayslipGenerated` event is defined but never dispatched.** The `App\Events\PayslipGenerated` class with the `UploadPayslipToCloud` listener exists, the integration manager is plumbed, and the `mirror_documents_to_cloud` feature flag is in `config/integrations.php` — but no line of `app/` code calls `PayslipGenerated::dispatch(...)`. Until the dispatch is added at the end of `PaymentService::generatePayslip()`, no payslip PDF ever reaches OneDrive / Drive — and there isn't a PDF emitter wired either. Ch 19 lists "bulk payslip-PDF emission" as Phase 1 work; the per-payment producer is part of the same arc.
- **No "Cancel" or "Mark failed" action in the UI.** The `PaymentStatus::Cancelled` and `PaymentStatus::Failed` cases exist on the enum but no route, controller method, or button writes them. The only way to retire a mistaken Payment today is `SoftDeletes` from a console command. A "Cancel with reason" affordance with a required ≥ 10-char justification, modelled on Chapter 19's reverse flow, is the natural next step.
- **No `reference` column.** Auditors who ask "show me Payment PAY-2026-0042" are answered with "show me Payment #42" instead. See the SequenceService section above.
- **No per-employee filter in the index UI.** `PaymentService::list()` honours an `employee_id` filter (the controller passes it through), but the index page does not render an employee dropdown — only status tabs and the month picker. The route accepts it via query string, so deep-linking from the Employee detail page would already work; the UI control just hasn't been added.
- **No Export button on the Payments index.** `PayrollExport` exists and is reachable from Reports (Ch 33), but the Payments page itself has no inline "Export this month" CSV button. The data is there; the affordance is missing.
- **Single permission gate.** Every action on the module — viewing, creating, marking paid, generating payslips — is behind the single `payroll.manage` permission. There is no separate `payroll.view` for read-only access in the UI today (the policy supports it, the route doesn't). HR and Finance both fall under the same gate; the duties-separation that Chapter 19 enforces between hr_admin (creates) and finance_officer (approves) is not enforced here.
- **No reversal trail.** If a Paid Payment turns out to be wrong, the operator soft-deletes it. There is no Reversed status, no required reason, and no reversal event — the audit trail for a corrected payment is just two rows (the original, soft-deleted, and the new one). This is the single largest gap relative to Chapter 19's approach.
- **`processed_by` is overwritten, not appended.** The same column captures both the creator and the marker, so the row only ever remembers the last person to touch it. A separate `created_by` column (and an immutable one at that) is the right fix.
- **No link back to a Payroll Run.** If a per-payment payslip was generated for the same employee in the same period that the batch ran, nothing in the data model knows. A nullable `payroll_run_id` would let the Profile portal show "this is in addition to your May payroll" rather than rendering two unrelated rows.

These are all known and all small. The honest framing is: the module works for what HR uses it for today (reimbursements, ex-gratia bonuses, salary advances, the occasional ad-hoc consultancy fee), but the audit posture, reversal posture, and event-driven cross-module hookups are weaker than every other Wave-1 module.

## What's planned next

Phase 1 of the government-grade roadmap (see Chapter 46) lands four things directly on this module: **(1)** Wire `PaymentCreated` and `PaymentPaid` events (and `PaymentCancelled` when the Cancel action ships) so the audit log captures every state change; the listeners follow the same shape as `PayrollRunApproved`. **(2)** Wire `PayslipGenerated::dispatch(...)` at the tail of `PaymentService::generatePayslip()` so the existing `UploadPayslipToCloud` listener can mirror per-payment PDFs to OneDrive / Drive; the PDF renderer reuses the same composer that Chapter 19's bulk payslip work needs. **(3)** Add a `PAY-YYYY-NNNN` reference column on `payments` minted through `SequenceService::next('PAY', 4, now()->year)`, and surface it on the index table, the show masthead, and the Reports export. **(4)** Add a "Cancel with reason" action that flips status to `Cancelled`, records the reason on a new `cancelled_reason` column, stamps `cancelled_by` / `cancelled_at`, and refuses to fire on a `Paid` row (reversal of a Paid one-off goes through the eventual unified reversal workflow). Phase 2 adds the duty-separation between `payroll.create` and `payroll.approve` permissions so HR and Finance can be split, brings a nullable `payroll_run_id` foreign key onto the table so a per-payment payslip can be associated with the matching batch run, and adds the link in the Profile portal that says "this top-up is in addition to your May 2026 payroll line". Phase 3 promotes one-off Payments into the GIFMIS journal as a separate sub-ledger feed alongside the batch-run feed, so the Consolidated Fund picture stays balanced across both flows.
