# Chapter 8 — Establishment (Positions, Grades, Steps, Ceilings)

> *In one paragraph.* Establishment is the approved skeleton of the institute — every post the institute is funded to hold, the grade band that post sits on, the salary step inside that band, and the headcount ceiling that says "no more than this many of this grade in this department this fiscal year". The Employees table in Chapter 3 stores **people**; the Establishment tables here store **posts**. A person becomes "established" the moment they are assigned to a post — the assignment row is the link, and it carries effective dates so the audit trail can answer "who held this seat on this date" five years later. In Ghana public-sector terms this is the half of the HRMS that exists to satisfy the Public Services Commission, the Controller and Accountant-General's Department (CAGD), and the Auditor-General — not the half that pays a salary on the 25th.

## Where to find it

- **Sidebar location:** the Positions register is reachable today only by URL (`/positions`) and from the Tier-1 fix test that wires `Establishment/Positions/Show`. The dedicated sidebar group is part of Phase 1's UI follow-up — the data layer and the screens ship in this chapter; the nav entry is a separate ticket. Roles that have `positions.view` and land on `/positions` see the full register; roles that don't get a 403 from the route middleware before the Inertia render runs.
- **Roles that see it:**
    - **super_admin** and **ceo** — every position, every action, including the `establishment.exceed` override that lets them blow through an approved headcount ceiling on purpose (and audit-logged).
    - **hr_admin** — `positions.view` + `positions.manage` + `grades.manage`. The full CRUD: create posts, assign people, vacate, freeze, edit grade and step tables.
    - **dept_head** — `positions.view` only. They can see every approved post in their department, but cannot create, assign, or freeze.
    - **auditor** — `positions.view`. Read-only, for the Auditor-General reconciliation pack (Ch 24).
    - **manager / employee / finance_officer / it_support** — no establishment permission by default. Finance sees grade-step base salaries indirectly through payroll (Ch 19); everyone else has no business on this page.
- **Related modules:** Employees (Ch 3) — the four dormant columns `current_position_id`, `current_grade_id`, `current_step`, and `step_anniversary_date` are now wired live to the establishment tables; Payroll Engine (Ch 19) — pulls base salary directly from the grade × step lookup via `Grade::baseSalaryFor()`; Performance (Ch 6) — the step-increment service refuses to bump a step when the latest closed-cycle review is below 3.0/5.0; Departments (Ch 30) — every post rolls up to one department and the establishment ceiling is keyed on (department, grade, fiscal year); Recruitment (Ch 9) — job postings reference a position and inherit its grade band; Identity (Ch 25) — a Ghana Card-verified employee is a precondition for an Auditor-General-grade assignment record.

### Assign an employee to a position

| Field | Validation | Who can set it | Notes |
|---|---|---|---|
| **Employee ID** | required, must exist on `employees` | `positions.manage` (via `PositionPolicy::assign`) | The employee being assigned. The service closes any open assignment that employee already holds — a person is in exactly one substantive post at a time. |
| **Is acting** | boolean (defaults false) | Same | When true, the position's status flips to `acting`, not `filled`. The assignment row has `is_acting = true` so the Single Spine acting allowance can be calculated by payroll (Ch 19). |
| **Reason** | optional string, max 255 | Same | Free-text — `assignment`, `transfer`, `promotion`, `acting`, `demotion` are the conventional values. Saved verbatim to the assignment row; surfaces on the detail page's active and past tables. |

> *Notes:* The assign service is the single point of write for *any* establishment change — promotion, transfer, demotion, acting appointment, substantive confirmation all hit the same code path with a different `reason` value. The transaction (a) re-fetches the position under a row lock, (b) re-checks the establishment ceiling for `(department, grade, fiscal_year)` under the same lock to close the TOCTOU window two concurrent HR creates would otherwise leak, (c) closes any open assignment for the employee, (d) inserts the new `position_assignments` row with `start_date = today` and the employee's current step copied to `step_at_start`, (e) flips the position status to `filled` or `acting`, and (f) updates the employee's `current_position_id` and `current_grade_id` pointers. If anything in those six steps fails — duplicate row, exceeded ceiling, foreign-key violation — the whole assignment rolls back. The ceiling check is bypassed only when the actor holds `establishment.exceed`; the bypass itself is audit-logged so the Auditor-General sees both the override and the override-er.

## The data behind it

The establishment register spans five tables, all created in one migration on 2026-05-25:

- **`grades`** — the pay-band catalogue. Each row carries a unique `code` (`GS-08`, `GS-20`, etc.), a `name` (`Junior Officer`, `Director`), an integer `level` for ordering org charts, and the `min_step` / `max_step` range that bounds the band. Soft-deletable so a retired grade can be hidden without breaking historical payroll lookups. The demo seeder lays down seven grades — GS-08 through GS-20 — modelled on the Ghana Single Spine Salary Structure ranks; production deployments are expected to overwrite the seed with the institute's actual approved scale, but the *shape* — coded grades with min/max step ranges — is fixed by the schema.
- **`grade_steps`** — the per-grade salary table. Each row pins a `(grade_id, step, effective_from)` triple to a `base_salary` decimal and a currency (defaults to GHS). The composite unique key includes `effective_from`, which is how the system handles a Single Spine pay revision: the old row keeps its `effective_to` set, a new row carries the new `base_salary` from the revision date forward, and `Grade::baseSalaryFor(step, date)` picks the right one. The demo seeder lays down an 8-step ladder per junior grade (and 4-step or 6-step for senior grades) at a 4.5% increment per step — a stand-in for the real Single Spine table that the institute will load on deployment.
- **`positions`** — every approved post. The row carries the `code`, the `title`, foreign keys to `grade_id`, `department_id`, and `reports_to_position_id` (a self-reference for the org chart), the `cost_center`, the `funding_source` enum, the `status` enum, the `headcount_ceiling`, the `is_supervisory` flag, and the free-text `job_description`. Soft-deletable so a deprecated post can be archived without losing the assignment history that linked employees to it.
- **`position_assignments`** — the history table. One row per assignment, with `position_id`, `employee_id`, `start_date`, `end_date` (null while active), `is_acting`, `step_at_start` (frozen at the moment of assignment), and `reason`. Indexed both ways — `(position_id, employee_id)` for "who has held this post" and `(employee_id, start_date)` for "what posts has this person held". This is the table that makes the system audit-ready: every promotion, transfer, acting appointment, and substantive confirmation lands here with effective dates.
- **`establishment_ceilings`** — the approved-headcount table, keyed `(department_id, grade_id, fiscal_year)`. Each row carries the `approved_headcount` integer and a free-text `approval_reference` (typically the PSC letter or Cabinet decision number authorising the slot). The unique constraint on the triple means a department can have exactly one approved ceiling per grade per fiscal year — the establishment officer cannot accidentally double-set it.

In addition, the migration adds four columns to `employees`:

- **`current_position_id`** — foreign key to the position the employee currently holds. Set by the assign service; cleared by the vacate service.
- **`current_grade_id`** — foreign key to the grade. Set by the assign service to the position's grade.
- **`current_step`** — the step within the grade. Untouched by assignment; advanced by the nightly step-increment job.
- **`step_anniversary_date`** — the date the next step bump is due. Advanced one year forward by the increment job whether the step actually moves or not.

The four fields were already in the `employees` migration as "dormant" placeholders (Chapter 3 calls them out by name); the 2026-05-25 establishment migration is what brings them online with real foreign keys and a real increment job.

What every reader of the screen needs to keep in mind:

- **A position is not a person.** It is a funded slot. An employee fills it through an assignment row; emptying the slot does not delete the employee, and deleting the employee does not delete the position. This separation is the whole point of establishment control — it is what lets the Auditor-General reconcile "approved posts" against "warm bodies" and surface ghost workers.
- **A position can be vacant *and* still cost money.** A frozen post does not draw a salary, but its headcount line item is still on the approved establishment. The Auditor-General reads "approved minus filled minus frozen = funded vacancies" and asks the institute to justify any gap.
- **The ceiling is enforced under a row lock.** Two concurrent HR creates targeting the last slot in a (department, grade) band would, with a naive check, both see "1 slot left" and both create — overshooting the approved headcount by one. The service holds a `lockForUpdate()` on the position row, the ceiling row, *and* the count of filled positions in the same band, all in the same transaction, so the second concurrent attempt sees the first one's insert and is rejected with a `DomainException`.
- **The step is frozen at the moment of assignment.** `step_at_start` on `position_assignments` records the employee's step on the day they took the post. The employee's *current* step continues to advance through the increment job; the assignment row remembers where they started so the payroll engine can prorate back-pay calculations against the right starting point.
- **Funding source is not optional.** Every post has to declare whether it is funded by GoG, IGF, donor, IDF, or other. The Auditor-General pack pivots payroll cost on this attribute — a donor-funded post that draws salary from the consolidated fund is a finding.

## How it talks to other modules

- **Employees (Chapter 3)** — the four dormant fields are now live. `current_position_id` and `current_grade_id` are written by the assign service; `current_step` and `step_anniversary_date` are written by the `StepIncrementService`. The employee detail page reads them via the `currentPosition` and `currentGrade` belongs-to relations.
- **`Grade::baseSalaryFor($step, $date)`** — the single function the Payroll Engine (Ch 19) calls to resolve a base salary on a given pay date. The grade's steps are filtered by `effective_from <= date AND (effective_to IS NULL OR effective_to >= date)` and the most recent row wins, so a pay revision dated mid-period is handled correctly without any payroll-side logic.
- **Step increment (the `StepIncrementService::processForDate()` job)** — runs nightly. For every employee whose `step_anniversary_date` is today and whose current step is below the grade's `max_step`, the service consults the latest closed-cycle performance review (Ch 6), and: if the rating is ≥ 3.0/5.0, bumps the step and rolls the anniversary one year forward; if it is below 3.0, rolls the anniversary forward *without* the bump (the employee is re-evaluated next year); if the employee is already at top of grade, rolls the anniversary forward (top-of-grade employees never lose their anniversary, but they also can't progress without a promotion). The four outcomes — `processed`, `skipped_top`, `skipped_underperform`, `skipped_no_review` — are returned as an array so the nightly job can log them. **Idempotent** — running it twice on the same date does *not* double-increment because the anniversary moves forward in the same transaction as the step bump.
- **Performance (Chapter 6)** — the step gate above is the closed loop between performance and pay: a manager who awards a rating below 3.0 is, whether they know it or not, freezing that employee's step for a year. The system does not retro-undo the increment if the review is later revised — the increment, once granted, is final, but the *next* anniversary is judged on the *next* review.
- **Departments (Chapter 30)** — every position rolls up to one department; deleting a department is blocked while any non-soft-deleted position is still assigned to it (the same `DomainException` pattern the Employees module uses). Position counts feed the department dashboard's "approved vs filled" tile.
- **Recruitment (Chapter 9)** — a job posting references a position. The grade band on the posting is inherited from the position, and the recruitment-to-onboarding handover writes a `position_assignments` row through the same `PositionService::assign()` call the establishment UI uses, so a hire through recruitment is indistinguishable from a hire through manual HR assignment.
- **Payroll Engine (Chapter 19)** — pulls `base_salary` via the grade × step lookup for every active employee, every pay run. The `step_at_start` field on the assignment row is what the engine uses when it needs to recompute the salary from a back-dated start (most commonly for an arrears payment).
- **Auditor-General pack (Chapter 24)** — the headcount-reconciliation page is a direct join of `positions` against `position_assignments` against `employees` against `identity_verifications`: approved posts, current occupants, and whether each occupant has a verified Ghana Card. Discrepancies — a filled post with no occupant, an occupant with no Ghana Card, a position outside the ceiling — are the ghost-worker findings the pack flags.
- **Identity (Chapter 25)** — the Auditor-General-grade assignment record is one where the holder also has a passing `IdentityVerification` row. The establishment register itself does not refuse an assignment when the holder is unverified, but the audit pack does flag it; the policy decision to harden this into a *block* sits with the institute.

## Standards touchpoints

- **Ghana Single Spine Salary Structure (SSSS), 2010** — the public-service pay framework. The `grades` and `grade_steps` tables are shaped for Single Spine compliance — coded grades, integer step ladders, dated `effective_from` / `effective_to` ranges so a Cabinet-approved pay revision can be loaded as a new step row rather than overwriting the old one. The demo seeder lays down a credible Single Spine *shape* (GS-08 through GS-20, 4-to-8 step ranges, 4.5% step increments) but the actual Single Spine pay table — the one the Ministry of Finance gazettes — must be loaded by the institute on deployment. The system is Single Spine-*aware*, not Single Spine-*authoritative*; the authoritative source is the Fair Wages and Salaries Commission gazette.
- **Public Services Commission (PSC) establishment control** — every public-sector institute is required to operate within an approved establishment, with each post documented and each occupancy traceable. The `positions` table is the approved establishment; the `establishment_ceilings` table is the PSC's approved-headcount sign-off (with `approval_reference` carrying the letter or decision number); the `position_assignments` table is the occupancy history. Together they answer the three questions a PSC audit will ask first: how many posts are approved, how many are filled, and what is the audit trail for every hire, promotion, and transfer.
- **Auditor-General establishment audit** — the annual establishment audit's central exercise is reconciling approved posts against warm bodies, with ghost workers being the public-sector readout of an unreconciled count. The audit pack (Chapter 24) reads directly off the tables in this chapter; the row-locked ceiling enforcement closes the most common ghost-worker vector, which is over-establishment by accidental concurrent hire.
- **ISO 30414 §4.7 workforce planning** — the standard recommends disclosing approved headcount, filled headcount, and ratio of filled to approved as workforce-planning metrics. The four KPI tiles on the position detail page and the four-up stats band on the register expose the same numbers; the analytics chapter (Ch 13) collects them across the institute for the ISO-aligned export. See Chapter 44.
- **DPA Act 843** — the establishment register itself processes minimal personal data (employee_id pointers; no PII on the position rows). The assignment history carries an `employee_id` and dates, which counts as personal data under §17; the same lawful-basis-of-employment-contract that covers the Employees table covers it here.

## What's planned next

Three pieces of work are in flight directly against this chapter and are scheduled for Phase 1's UI follow-up (W1.21–W1.24):

1. **The sidebar entry and the create panel.** The Positions register is reachable today only by URL; the Governance sidebar group will add a "Positions" item for any role with `positions.view`, and the register will gain a cobalt "Add Position" button that opens a right-hand slide panel with the same `StorePositionRequest` fields documented above. The same panel will be reused (with the grade field swapped for a step picker) for a dedicated Promote/Transfer flow that wraps `PositionService::assign()` with a richer reason taxonomy and an effective-date picker for back-dated promotions.

2. **The Single Spine pay table loader.** The demo seeder is a placeholder; the production loader will accept the Fair Wages and Salaries Commission gazette as a CSV upload, validate it against the existing grade codes, and insert one `grade_steps` row per (grade, step) with the gazette's `effective_from` date — so a pay revision becomes a one-CSV operation rather than a hand-edited migration. The loader will refuse to import a row that overlaps an existing effective range without an explicit `--supersede` flag, so accidentally double-loading a revision can't happen.

3. **The Auditor-General reconciliation report.** Chapter 24's pack will gain a dedicated "Establishment reconciliation" sheet that joins `positions` × `position_assignments` × `employees` × `identity_verifications` and flags every row where (a) the position is filled but no open assignment exists, (b) an open assignment points at an employee with no `IdentityVerification` row in `verified` status, or (c) the count of filled positions in any (department, grade, fiscal_year) exceeds the corresponding `approved_headcount` — i.e. an `establishment.exceed` override has been used and not yet been re-approved. The report runs read-only off the same tables this chapter ships; no schema work is required to produce it.

A fourth piece — a tamper-evident audit row for every assign, vacate, freeze, and `establishment.exceed` override, written to the same audit chain the rest of Phase 1 uses — is dependency-tracked against the audit chain rollout in Chapter 44 and lands as part of that work rather than this one.
