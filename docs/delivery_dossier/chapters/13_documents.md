# Chapter 13 — Documents

> *In one paragraph.* Documents is the institute's electronic registry — every letter, memo, scanned form, or signed approval lives here as a single `Document` row that carries a year-numbered reference, a version history, a routing slip, and a layer of moveable annotations. HR composes a letter in the in-portal editor; a department head uploads a scanned invoice; a manager drops a signature on page two of an approval memo and forwards it to Finance. Throughout the journey the original file is never overwritten — annotations live in their own table and are only burned onto a flattened PDF at download time. Stamps, letterheads, and watermarks come from per-user, per-department, and organization-wide asset libraries so the same approval mark or letterhead can be reused across thousands of documents without re-uploading. Documents is the connective tissue under Tickets attachments, Recruitment CVs, Complaints evidence, Whistleblower files, and the personnel-file Documents tab.

## Where to find it

- **Sidebar location:** **Workspace** group → **Documents** (top-level item, paper-with-fold icon). The page lives at `/documents` and ships with five tabs across the top of the table: **All**, **Inbox** (badge shows the count of items waiting on your action), **Sent**, **Drafts**, **Archive**. The asset libraries are reached separately under **Settings** → **Stamps / Letterheads / Watermarks**.
- **Roles that see it:**
    - **super_admin** and **hr_admin** — every document, including org-wide override (`documents.manage`) to edit/delete any draft and to grant organization-wide shares. They also hold `document_assets.manage`, which is what unlocks the **organization** scope in every asset library.
    - **ceo** — the same view as super_admin via the Gate::before fallthrough.
    - **dept_head / manager / employee / finance_officer / it_support / auditor** — `documents.view` and `documents.create` are seeded onto every role by `DocumentPermissionsSeeder`, so any logged-in user can upload, see their own drafts, and respond to a document routed to them. What they *don't* hold by default is `documents.manage` or `documents.share_organization` — those are the two privileged extras.
    - **anyone** with a document **routed** to them (`document_routes.to_user_id`) or **shared** with them (individual / department / org-wide) is widened into `DocumentPolicy::view` automatically, even if they don't otherwise have it in scope.
- **Related modules:** Employees (Ch 3) keeps a per-employee documents drawer that runs on the legacy `employee_documents` table (HR upload + download only — no routing); Tickets (Ch 11) accepts attachments stored under the Documents pipeline; Recruitment (Ch 9) parks CVs and offer letters there; Complaints (Ch 12) and Whistleblower (Ch 27) attach evidence; Audit Logs (Ch 24) listens to the `document_events` table for the tamper-evident provenance trail; DPA / Privacy (Ch 26) enforces the confidentiality classifications and burned-PDF watermarking.

## The data behind it

The documents subsystem is built on seven tables that travel together and three new tables for asset libraries.

- **`documents`** — the master row. UUID for the route key, year-numbered `ref_no` (unique), title, description, owner, current_version_id, status (`draft / in_review / completed / rejected / withdrawn / archived`), confidentiality (`internal / confidential / restricted`), parallel_routing flag (column exists, UI on the backlog), tags JSON, **letterhead_id** + **watermark_id** + **watermark_mode** (`none / on_burn / always`), soft-delete column.
- **`document_versions`** — one row per file revision. version_no auto-increments per document; original_name and MIME preserved for display; SHA-256 stored for integrity check; storage_path is `documents/{uuid}/v{n}/{hash}.{ext}` on the `local` disk.
- **`document_routes`** — the sequential routing slip. One row per recipient with sequence number, from_user (the previous signer or the owner for the first stop), to_user, action_required (`sign / review / approve / acknowledge`), status (`pending / in_progress / completed / rejected / cancelled`), optional due_at, acted_at, and comment.
- **`document_annotations`** — the manipulable layer. Every signature, stamp, text note, initial, and highlight lives here as its own row with `type`, `page` (1-indexed), `x_pct / y_pct / w_pct / h_pct` (percentages of page dimensions so geometry survives rescaling), `rotation` in degrees, optional `route_id` linking the annotation to the signer's route step, and a JSON `data` blob carrying the PNG base64 / SVG / text / colour.
- **`document_events`** — the immutable audit trail. Every meaningful action — uploaded, version_added, routed, annotated, signed, stamped, forwarded, rejected, completed, withdrawn, downloaded, archived, updated, deleted, shared, unshared, annotation_moved, annotation_resized — writes one row with actor, occurred_at, and a JSON payload of the before/after values. This is the single source of truth for the audit pack.
- **`document_shares`** — the read-only widening of `view` permission added by Documents v2 Phase 1. (document_id, audience_type, audience_id) is uniquely constrained so the same audience can't be granted twice; the controller refreshes the existing row on re-grant.
- **`stamp_assets`**, **`letterhead_templates`**, **`watermark_templates`** — the three asset libraries. All three share the same shape: scope (`personal / department / organization`), nullable owner_id, name, storage_path + MIME (text watermarks store the text + colour on the row instead), `created_by`, indexed `(owner_scope, owner_id)`.

What every reader of the screen needs to keep in mind:

- **Annotations are not flattened.** This is the headline architectural decision. The `document_annotations` table holds the geometry; `DocumentRenderService::burn` only flattens at download time, and the burned PDF is cached per `(version_id, annotation_set_hash)` so re-downloading doesn't re-render. The on-disk original is never modified once stored. This is what makes "drag the signature" possible at all, and it's also what makes the `annotation_moved` audit-log event meaningful — there's a real before/after to record.
- **Sign-off lock is real but local to the annotation.** Once an annotation's `route_id` belongs to a `completed` route, `DocumentPolicy::moveAnnotation` refuses every further geometry change. The annotation can no longer be moved, resized, rotated, or deleted by anyone — not the creator, not the owner, not `documents.manage`. This is the integrity guarantee the routing slip leans on. (Free-floating annotations placed at Draft stage by the owner are still moveable by the owner.)
- **Restricted ⇒ always burned.** Confidentiality is enforced at the download endpoint, not at the storage layer. When `confidentiality = restricted`, `DocumentController::download` flips `$burned = true` regardless of the query string, and the burned PDF carries a per-viewer watermark with the viewer's name + timestamp + classification. A leaked PDF is traceable back to whoever pulled it.
- **Watermark mode is a per-document switch.** `documents.watermark_mode = 'always'` makes the chosen watermark apply to *every* download — including the "Original" link. `on_burn` only watermarks the Burned PDF path. `none` disables. The seeded default is `on_burn`, which means most documents look pristine when downloaded as the original and only carry a watermark when the burned-PDF path is taken. This is configurable per document on the Edit drawer.
- **Three-scope ownership.** Personal, department, organization. The same three-radio scope appears on every asset library (stamps, letterheads, watermarks) and on every share audience. The semantics are identical across all six surfaces — "owner_id is the user OR the department OR null", and the index query unions the three scopes the user can see. Policies always go: org rows visible to all → department rows visible to that department's members → personal rows visible to the uploader.
- **eSign is plumbed for Recruitment only.** There is a real `ESignWebhookController` and an `ESignProvider` interface with `createEnvelope`, `status`, `signedDocumentUrl`, and `void` methods — but in the MVP they're wired to the Recruitment module's offer-letter workflow (matching against `applicants.esign_envelope_id`) and not to the generic Documents pipeline. From Documents, eSign is a roadmap item; signing a routed document inside the portal uses the SignaturePad + annotation pair described above, which is legally valid as an electronic signature under the Electronic Transactions Act (Act 772) §10 but is not yet anchored to an external eSign provider's audit trail.

## How it talks to other modules

- **Employees (Ch 3) — Documents tab.** Uses a separate, legacy `employee_documents` table for the per-employee documents drawer in the personnel file. That drawer is HR-only and has no routing, no annotations, no asset libraries — it's the simple "attach a payslip to this employee" path. The full document register is the better home for anything that needs to be signed or routed.
- **Tickets (Ch 11) — attachments.** When a ticket has more than a single photo attached, those attachments are pushed into the Documents pipeline (with confidentiality `internal`, owner = the requester) and the ticket row holds the document id. The ticket detail page surfaces them via the share-with relation so a ticket-handler reading the issue gets `view` on the supporting documents automatically.
- **Recruitment (Ch 9) — CVs and offer letters.** Every CV uploaded by an applicant is stored as a Document with confidentiality `confidential` (so it can never be org-wide-shared, only person-to-person); every offer letter generated for the applicant is also stored here and is the surface the eSign integration writes back to when the applicant signs.
- **Complaints (Ch 12) and Whistleblower (Ch 27) — evidence.** Both modules store the supporting evidence (screenshots, letters, voice memos) as Documents with confidentiality `restricted` so the burned-PDF + viewer-watermark behaviour applies on every download. The whistleblower module additionally writes the original filename out as the document title so the evidence is searchable.
- **Audit Logs (Ch 24).** `document_events` is the single source of truth for the audit replay. Every state transition writes a row; nothing else in the system writes to that table. Audit Logs simply runs `SELECT * FROM document_events WHERE document_id = ? ORDER BY occurred_at` and renders the timeline.
- **DPA / Privacy (Ch 26).** Confidentiality classifications are the policy hook the DPA chapter relies on. The export pack uses `documents.confidentiality` + `document_shares.audience_type` to compute the data-subject access response — every document the subject can see is included; every share that names them is listed in the cover note.
- **Notifications (mail, in-app).** Two notifications: `DocumentAwaitingAction` (sent when a route turns `in_progress` — to the recipient) and `DocumentCompletedNotice` (sent when the document completes or is rejected — to the owner). Both write to the user's notification feed and to email.
- **Events (Laravel listeners).** `DocumentRouted`, `DocumentSigned`, `DocumentCompleted`, `DocumentRejected` — these are the four module-level events the Analytics module subscribes to. The Notifications module is the only other current listener.

## Standards touchpoints

- **Data Protection Act, 2012 (Act 843) §17 — lawful basis for processing personal data.** Every document the system holds is processed under the employment-contract basis when it concerns staff, and under the public-task basis when it concerns external citizens (applicants, complainants). The confidentiality field is the operational hook: documents marked `confidential` or `restricted` cannot be broadcast-shared, which is how §18 (data minimisation) is enforced in practice. See Chapter 26.
- **Data Protection Act §40 — right to erasure.** Soft delete is the default; a tamper-evident audit-log row records every delete. The DPA-grade redaction job that does true erasure (overwriting the file on disk and the row in the database) is roadmapped in Phase 2 of the government-grade plan. The seven-year retention window for HR documents (and the document-type-specific retention table) is also a Phase-2 item; today, the soft-deleted-but-not-purged row stays forever.
- **Electronic Transactions Act, 2008 (Act 772) §10 — recognition of electronic signatures.** The annotation pipeline produces a legally-recognised electronic signature: the signer authenticates with their staff_id + password, the canvas drawing is stored as a PNG owned by their user_id, the route record carries the timestamp and the routing context, and the entire act writes immutable rows to `document_events`. The annotation lock-after-sign-off is the integrity guarantee §10(d) of the Act calls out. What is **not** yet present is a qualified electronic signature with a third-party trust service — that requires an eSign provider integration, currently scoped only to the Recruitment offer-letter workflow.
- **Evidence Act, 1975 (NRCD 323) §3 — admissibility of electronic records.** The combination of (i) immutable per-version SHA-256 hash, (ii) tamper-evident `document_events` log, (iii) signer-locked annotations after completion, and (iv) per-viewer watermarked downloads gives the Documents module a reasonable evidentiary chain. The remaining gap is a court-grade export pack (a single ZIP with the original file, the burned PDF, the events log as CSV, and a manifest signed with an institute key) — that's planned in the audit-pack work referenced by Chapter 24.
- **ISO/IEC 27001 Annex A.5.12 / A.5.13 — classification and labelling of information.** The three-tier confidentiality enum (`internal / confidential / restricted`) is the classification scheme; the burned-PDF watermark is the visible labelling on every page when classification is non-default; the share-guard is the protective handling control. The classification is intentionally simple (three rungs, not five) to keep operational compliance achievable.
- **ISO/IEC 27001 Annex A.8.10 — information deletion.** Soft delete + admin restore + DPA-redaction job (Phase 2) covers the deletion lifecycle. The retention rules per document type are still policy-by-convention; turning them into a per-type retention table is Phase 2 work.

## What's planned next

Phase 1 of the government-grade roadmap lands four things on this module: (1) the **bulk routing** workflow — pick a stack of documents from the library, attach the same routing slip, fire one batch — for the Vice-Chancellor's end-of-year sign-off run; (2) **parallel routing** — the `documents.parallel_routing` column has been on the schema since v1, but the controller currently treats every route as sequential; flipping it on requires a small change in `DocumentRoutingService::route` so that every recipient starts in `in_progress` simultaneously and the document only completes when the last one acts; (3) **eSign integration for the documents pipeline** — extending the existing `ESignProvider` contract beyond Recruitment so an in-portal signature on a routed document can optionally be co-signed via Zoho Sign or DocuSign for documents that need an external trust anchor; (4) **DOCX/XLSX burn-in** — `DocumentRenderService::burn` currently passes Office documents through unmodified because LibreOffice headless conversion is not yet on the deployment image; once it is, annotations can be flattened onto DOCX downloads the same way they are on PDF. Phase 2 adds the per-type retention table (so HR letters auto-archive after seven years, payroll slips after six, ticket attachments after two), the redaction job that satisfies DPA §40 right-to-erasure, and the court-grade evidence ZIP. See the government gap analysis (Chapter 46) for the full sequence.
