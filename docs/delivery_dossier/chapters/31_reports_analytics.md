# Chapter 31 — Reports & Analytics

> *In one paragraph.* Reports & Analytics is the bird's-eye view of CIHRMS — the place where every other module's day-to-day churn shows up as numbers a director, an auditor, or a finance officer can act on. The Dashboard (cached every 60 seconds per user) frames it for whoever just logged in; the Reports Ledger turns the live database into five named XLSX exports HR teams actually file with; and the Auditor-General Report Pack bundles a fiscal year of payroll, statutory returns, identity verifications, loans, off-boarding and audit-chain output into a single SHA-256-sealed ZIP that the Auditor-General's office can verify by re-hashing. Underneath it all sits the `analytics_events` table — a queued, append-only stream that every Wave-1 module writes into and that the Dashboard's sparklines and "Live activity" feed read out of.

## Where to find it

- **Sidebar location:** **Reports** group (gold tier — the same flagship 5% accent as the Dashboard). Two children: **Overview** (`/modules/reports` → `/reports`) and **Auditor-General** (`/reports/auditor-general`). The Reports group only shows when the signed-in user has `reports.view`; the Auditor-General child further requires `statutory.export` so only finance officers and auditors see it. The **Dashboard** itself sits one rung above as the institute's landing page and is its own sidebar item (`assessment` icon, top of the rail).
- **Roles that see it:**
    - **super_admin**, **ceo**, **hr_admin** — full Reports landing, all five exports, every dashboard tile, every snapshot block (Finance + Manager + Dept-Head bundles render alongside the admin view).
    - **finance_officer** — full Reports landing + the Finance dashboard (`RoleFinance.vue`) with treasury-bulletin hero, payroll-run posture, disbursement queue and statutory-overdue posture; can generate and download Auditor-General packs (holds `statutory.export`).
    - **auditor** — full Reports landing + the Auditor-General pack download/regenerate buttons; also holds `audit.view`, so they can verify the chain output from inside the pack.
    - **dept_head** — Reports landing (everything they can see in the underlying data, scoped to their department) + their own `RoleDeptHead.vue` overview with departmental headcount, on-leave-today count and a six-row recent leave feed.
    - **manager** — Reports landing + `RoleManager.vue` overview with team size, pending-leave inbox (8 latest), and open tickets for direct reports.
    - **it_support**, **marketing** — Reports landing is invisible (no `reports.view`), but the catch-all `RoleOverview.vue` dashboard still renders with their role-tagged KPIs and quick actions.
    - **employee** — Dashboard only. No Reports landing, no Auditor-General page; their dashboard is the standard four-tile overview.
- **Related modules:** Every Wave-1 module fires the events that populate this screen. Employees (Ch 3) → `employee.created`; Leave (Ch 4) → `leave.requested`, `leave.status_updated`; Attendance (Ch 5) → `attendance.correction.*`; Tickets (Ch 11) → `ticket.created`; Payroll (Ch 19) → `payslip.generated`, `payment.paid`; Recruitment (Ch 9) → `recruitment.applicant.created`; Assets (via the asset events) → six `asset.*` types; Benefits (Ch 23) → five `benefit.*` types; Governance (Ch 28) → six `policy.*` and `certification.*` types. The Auditor-General pack is the join point with Audit Logs (Ch 24) — the pack ships `audit:verify-chain` output verbatim. Standards alignment lives in Chapter 44.

## The data behind it

CIHRMS treats the report layer as a *read* surface — almost nothing originates here. Three stores feed everything you see on these screens:

- **The live operational tables** (`employees`, `leave_requests`, `payments`, `tickets`, `departments`, `payroll_runs`, `disbursements`, `statutory_returns`, `loan_accounts`, `offboarding_cases`, `identity_verifications`, `whistleblower_reports`, `audit_logs`) — the source of truth for every count, sum and breakdown. Reports query these directly with the visibility scopes already enforced on each model (a department head sees their slice; the AG pack reads the whole table because the pack runs under a permission that already implies institute-wide visibility).
- **`analytics_events`** — an append-only, queued event log. Schema is intentionally minimal: `id`, nullable `user_id` (so system-fired events are honestly attributed), `event` (a dotted-string name like `leave.requested`), `meta` (JSON cast, free-form payload — usually two or three keys naming the resource id and a state value), `created_at`, `updated_at`. There is no migration to evolve, no schema-per-event-type — Wave 1 chose flexibility over rigidity here on purpose. The `RecordAnalyticsEvent` listener (`ShouldQueue`, queue `analytics`) is the single insert path; it pattern-matches 24+ event classes and writes one row per event.
- **`storage/app/ag-reports/`** — the on-disk archive of every Auditor-General pack ever generated, named `AG-{year}-{Ymd-His}.zip`. The listing on the Auditor-General page is a `Storage::disk('local')->files('ag-reports')` scan filtered to `.zip`. Each pack carries its own MANIFEST.md with per-file SHA-256 hashes; the pack is the audit artefact, the directory is the storage location.

What every reader of these screens needs to keep in mind:

- **The Reports landing is a *re-query* surface, not a *materialised* one.** Hitting Download regenerates the XLSX from the live database every time. There is no nightly snapshot, no warehouse, no "report run history". The 60-second preview cache is the only cache between you and the data — and it's busted by TTL alone.
- **The Dashboard is a *cached* surface.** Six different cache keys, all on the 60-second `STATS_TTL`. A change made now will appear on the dashboard within one minute, not instantly. This is a deliberate trade-off for query cost; an explicit cache flush isn't exposed in the UI.
- **`analytics_events` rows are immutable.** The listener inserts; nothing updates. Wave 1 left the meta column as schemaless JSON and the event names as free strings, which means a typo (`leave.requuested`) lives forever — the safeguard is the `match` block in the listener that only writes well-known names. Anything coming through the public `POST /analytics` endpoint (the `AnalyticsController::store` action used by client-side JS for things like "modal opened") is validated for shape but not for vocabulary.
- **The Auditor-General pack is a *sealed* surface.** Every file in the ZIP is independently SHA-256 hashed and the hash list is committed to MANIFEST.md inside the same ZIP. The pack is the audit artefact; once generated it is not edited. Re-downloading the same pack returns byte-for-byte the same bytes.
- **Whistleblower data is segregated by statute.** The pack contains *counts only* (category × severity × status) from `whistleblower_reports`. Case content is NOT in the pack — Act 720 (Whistleblower Act, 2006) puts case content under the designated investigator's separate cover, and the manifest restates that on every pack so the auditor can't accidentally request it via this channel.

## How it talks to other modules

Reports & Analytics is the *sink* for the rest of the system. The data flow is one-way: modules fire events, the analytics listener queues them, the Dashboard and Reports layer read them back. The ten biggest event producers in the codebase today, in volume order:

1. **Employees (Ch 3)** — `employee.created`. Hires per day are the spine of the Active Staff sparkline.
2. **Leave (Ch 4)** — `leave.requested`, `leave.status_updated`. Drives Pending Leave KPI + Leave Summary export sparkline + the dept-head's "on leave today" count.
3. **Tickets (Ch 11)** — `ticket.created`. Drives Open Tickets KPI + Ticket SLA Report preview + Service-load donut.
4. **Payroll (Ch 19)** — `payslip.generated`, `payment.created`, `payment.paid`. Drives the Pending Payroll KPI, the Payroll Export preview, and the Finance snapshot's paid-YTD figure.
5. **Recruitment (Ch 9)** — `recruitment.applicant.created`. Drives the catch-all `RoleOverview.vue` quick actions for Marketing.
6. **Attendance (Ch 5)** — `attendance.correction.requested`, `attendance.correction.decided`. Not surfaced on the main dashboard yet — the events are recorded so Phase 1 can pivot the dashboard's "attendance posture" tile.
7. **Assets** — six event types (`asset.assigned`, `asset.returned`, `asset.maintenance.logged`, `asset.maintenance.completed`, `asset.retired`, `asset.lost`). Feeds the Live activity stream; per-asset analytics are a Phase 1 ask.
8. **Benefits (Ch 23)** — five event types (`benefit.plan.created`, `benefit.enroled`, `benefit.dependant.added`, `benefit.claim.submitted`, `benefit.claim.decided`). Same role on the dashboard as assets.
9. **Governance (Ch 28)** — six event types (`policy.drafted`, `policy.version.added`, `policy.published`, `policy.acknowledged`, `certification.expiring`, `certification.expired`). Will drive the compliance posture sparkline once it switches off the literal-array fallback.
10. **(Anything new)** — any event class not pattern-matched in the listener falls through to the default branch and is recorded with `event = class_basename($event)` and empty `meta`. The system fails *open* on new events — it's strictly easier to add a richer mapping later than to lose the record.

Downstream:

- **Audit Logs (Ch 24)** — Reports does not write audit logs itself, but the Auditor-General pack *consumes* the audit log: `audit:verify-chain` runs against `audit_logs` and the output goes into `audit/chain_verification.txt`. Tampering anywhere on the chain shows up as a non-zero exit code in that file.
- **Payroll (Ch 19)** — the Auditor-General pack pulls `PayrollRun::whereBetween('period_start', [$start, $end])` and concatenates per-run statutory return files into yearly PAYE / SSNIT / Tier-2 / NHIA CSVs. Bank disbursement files (the GhIPSS-compatible ones) are copied verbatim into `statutory/bank_files/` so the auditor can match what was sent to the bank against what the payroll engine recorded.
- **Identity verification (Ch 25)** — the pack writes `identity/verifications.csv` with the Ghana Card number *masked* to its last digit ("GHA-•••••••••-N") so the auditor sees who is verified, not the raw PII. The full PIN stays in the encrypted database column.
- **Loans (Ch 21)** — the pack writes both `loans/accounts.csv` (every loan booked in the fiscal year, full lifecycle dates) and `loans/outstanding.csv` (open balances at year-end, sorted by `outstanding_balance DESC`).
- **Off-boarding (Ch 10)** — the pack writes `offboarding/cases.csv` with one row per case whose `last_working_day` falls in the fiscal year, joined to the settlement row for gross/deductions/net.
- **Whistleblower (Ch 27)** — counts only, by category × severity × status. The pack restates the Act 720 segregation note in `MANIFEST.md` and on the on-screen "What's bundled" card.

## Standards touchpoints

- **ISO 30414:2018 — Human capital reporting** — the international standard for what an institutional people-report should contain. The Reports landing covers a meaningful subset of its 23 disclosures *today*: workforce composition (§4.1) from the Headcount export and the Workforce donut; leadership ratios (§4.5) from the Employee directory's manager relationship surfaced by the analytics layer; recruitment metrics (§4.7) from the `recruitment.applicant.created` event stream that feeds the catch-all overview; succession planning surfaceable from the `EmployeeService::stats()` query; productivity and revenue-per-employee (§4.6) computable from Payroll + Recruitment but not yet shipped as a named report; turnover (§4.9) from the Staff Turnover export with tenure-in-days; learning & development (§4.10) from Governance/Learning chapters once the LMS lands. See Chapter 44 for the full crosswalk.
- **ISO 30414 §4.1 (workforce composition)** — directly served by the Headcount export columns (Name, Staff ID, Department, Position, Hire Date, Status) and the Dashboard's Workforce donut + department breakdown.
- **ISO 30414 §4.9 (turnover)** — directly served by the Staff Turnover export, which includes hire date, termination date, and tenure-in-days computed at row emission time.
- **Audit Service Act, 2000 (Act 584) and the Audit Service constitutional mandate (Article 187)** — the Auditor-General Report Pack is the operational answer to "what does the AG need from CIHRMS on a routine visit?" The pack version (1.0) is committed to MANIFEST.md so the AG's office can adapt their ingestion if the schema evolves.
- **Public Financial Management Act, 2016 (Act 921)** — the eyebrow above the AG page reads "AUDITOR-GENERAL PACK · ACT 921" precisely because Act 921 is the modern statutory frame for public-sector financial management; the pack's payroll + statutory + bank-file bundle is the dataset Act 921 audits hinge on.
- **Income Tax Act, 2015 (Act 896) — PAYE returns**, **Social Security and National Insurance Trust Act, 2008 (Act 766) — Tier-1 returns**, **National Pensions Act, 2008 (Act 766) — Tier-2 trustee schedules**, **National Health Insurance Act, 2012 (Act 852) — NHIA share** — all four statutory return CSVs ship inside the pack under `statutory/`, concatenated across periods, headers preserved.
- **Whistleblower Act, 2006 (Act 720)** — Section 11 of Act 720 places case content under the designated investigator. The pack respects the segregation in code (`writeWhistleblowerSummary()` selects only `category, severity, status, count(*)`) and in the on-screen card (the only amber "info" line on "What's bundled"). The manifest restates the rule for the auditor.
- **Data Protection Act, 2012 (Act 843) §17 (lawful basis) and §18 (data minimisation)** — every personally identifying number that leaves the system through a report is minimised by default. Ghana Card numbers in the AG identity register are masked to the last digit; salary is gated by `employees.view_salary` on the live exports too. The Reports layer does not introduce a new lawful-basis category — it inherits the employment-contract basis the underlying record already relies on.

## What's planned next

Phase 1 of the government-grade roadmap lands four things directly on these screens. (1) **Scheduled report delivery** — saved report definitions with parameter binding, run on cron, emailed to a list. The plumbing is already in place: the five `App\Exports\*` classes are ready to be invoked from a queued job, only the schedule + recipients UI is missing.
