# CIHRM Ghana — Authentication Credentials

Seeded login accounts for the CIHRMS development database, generated by [`database/seeders/DatabaseSeeder.php`](../database/seeders/DatabaseSeeder.php).

> **Last reviewed:** 2026-05-24

## Authentication method

Users log in with **Full Name + Staff ID + Password** (not email). The login form is wired through [`LoginRequest.php`](../app/Http/Requests/Auth/LoginRequest.php).

- The password requirement was added in PR #34 — previously the form accepted only Name + Staff ID
- All seeded accounts use the password `password`
- **First-login password change is enforced.** Every fixed account is seeded with `password_must_change = true`, which routes the user through `/password/change` immediately after authentication and blocks the dashboard until they pick a new password.

### If you can't log in with the documented password

A previous dev session may have changed the password through the must-change wall, leaving the doc out of sync. Two recovery options:

1. **Re-seed the fixed accounts only** (idempotent, won't drop any other data):
   ```bash
   php artisan db:seed --class=DatabaseSeeder
   ```
   The `updateOrCreate` calls in `seedFixedAccounts()` will reset the password to `password` for every account in the table above.

2. **Reset by staff ID** using the artisan helper (added in PR #35):
   ```bash
   php artisan users:issue-password-resets --include-default-password
   ```
   This generates one-time reset URLs for any account whose password is empty OR matches the dev default, and prints them for distribution.

## Roles and permissions

Roles are seeded via [`RolePermissionSeeder.php`](../database/seeders/RolePermissionSeeder.php) into the DB-backed `roles` and `permissions` tables, *and* mirrored into the legacy per-user `permissions` array on the `users` table.

| Slug | Display name | Permission scope |
| :--- | :--- | :--- |
| `super_admin`     | Super Administrator       | `*` (wildcard — every permission) |
| `ceo`             | Chief Executive Officer   | `*` (wildcard — mirrors super_admin per PR #38) |
| `hr_admin`        | HR Admin                  | curated HR set (~30 perms) |
| `manager`         | Line Manager              | own-team scope |
| `dept_head`       | Department Head           | own-department scope |
| `employee`        | Employee                  | self-service only |
| `finance_officer` | Finance Officer           | full Finance F1–F5 access |
| `it_support`      | IT Support                | service desk + assets |
| `marketing`       | Marketing                 | campaigns portal |
| `auditor`         | Auditor                   | read-only + DPO + whistleblower investigator |

The full permission matrix is defined in [`User::ROLE_PERMISSIONS`](../app/Models/User.php) and re-applied each time `DatabaseSeeder` runs (so new permissions land on existing roles on every reseed).

## Factory-generated demo accounts

`DatabaseSeeder` also produces a small set of factory-created users (random names + staff IDs) to populate the directory, leave queue, and tickets pipeline. After running `php artisan migrate --seed` you can find them with:

```bash
php artisan tinker
>>> \App\Models\User::orderBy('id')->get(['id','name','staff_id','role']);
```

## Resetting the database

```bash
php artisan migrate:fresh --seed
```

This drops every table, re-runs every migration, and reseeds the fixed accounts above plus the factory demo set.

## Security note

These credentials are for **local development only**. In production:

- Staff IDs must be unique and assigned by HR, not seeded.
- **First-login password change is now enforced** via the `password_must_change` flag set on every seeded account (PR #34). The dashboard is blocked until the user picks a new password.
- Never deploy with the seeded `password` value intact — run `php artisan users:issue-password-resets --include-default-password` before exposing the new login to staff (full playbook in [docs/deployment_production.md §8a](deployment_production.md)).
